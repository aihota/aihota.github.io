АНАЛИЗ ВЫЛЕТА HD_HOTA.dll+0x135AAC (сетевой старт игры)
================================================================================

Модель: Claude Opus 4.8
Дата: 27.06.2026
Заключение: HD_HOTA.dll: вызов sub_1135AA0 с нулевым this из-за
отсутствия проверки результата sub_1135BA0. Провал поиска свободного
слота (ID=0) вызван тем, что 8-й слот таблицы содержит
неинициализированный мусор кучи (пустой конструктор sub_1135B10 +
условный memset sub_1135B20), а покинувший комнату игрок не сбросил
свой слот (0x3F6 не отправлен при отключении).


КРАТКОЕ РЕЗЮМЕ
================================================================================
Исключение ACCESS_VIOLATION (запись по адресу 0x00000006). Точка сбоя:
sub_1135AA0 — инструкция mov [esi+6], eax при ESI=0 (this=NULL).
Вызов произошёл из sub_113FAD0 (обработчик сетевого сообщения 0x8A2/2210,
подтип 1) в ходе согласования старта сетевой игры.

Механизм: таблица участников (dword_2163AE4) имеет 8 слотов, но её
память НЕ ОБНУЛЕНА при создании — конструктор (sub_1135B10) пустой,
memset-сброс (sub_1135B20) вызывается условно. После добавления 7 игроков
и выхода одного без корректного закрытия (0x3F6 не отправлен), в таблице
остаётся 7 слотов с валидными ID + 1 слот с мусором кучи (не 0).
Поиск свободного слота по ID=0 не находит ни одного, возвращает NULL,
NULL без проверки передаётся как this в sub_1135AA0 — crash.

Дополнительно найден второй такой же баг в sub_1138690 (диалоговый
callback лобби, тот же шаблон: 0x8A2/1 -> sub_1135BA0 -> без проверки
-> sub_1135AA0).

4. ТОЧКА СБОЯ
================================================================================
Функция: sub_1135AA0 (строка 523805 листинга)

sub_1135AA0 proc near
  arg_0 = dword ptr 8
  arg_4 = byte ptr 0Ch

  push ebp
  mov  ebp, esp
  mov  edx, [ebp+arg_0]   ; edx = указатель на строку (имя)
  xor  eax, eax            ; eax = 0
  push esi
  mov  esi, ecx            ; esi = this (thiscall)
  push edi
  mov  [esi+6], eax        ; <--- FAULT: запись по esi+6 = 0+6 = 0x6
  mov  [esi+0Ah], eax
  mov  [esi+0Eh], eax
  mov  [esi+12h], eax
  mov  [esi+16h], al
  ...

Инструкция сбоя: .text:01135AAC  mov [esi+6], eax
Адресная проверка: ESI = ECX = 0 (из контекста), faulting = 0 + 6 = 0x6 — OK.

Функция инициализирует запись таблицы: обнуляет поля начиная со смещения 6
(имя участника длиной до 16 байт), копирует строку из arg_0, затем
сохраняет флаг из arg_4 в байт [this+5] и вызывает sub_109A600 с
lea ecx, [this+6].

Структура слота таблицы (0x17 = 23 байта):
  +0: dword — ключ ID
  +4: 1 байт (служебный)
  +5: byte — флаг (из arg_4)
  +6: 16 байт — имя
Размер 23 байта = 0x17 — совпадает с шагом поиска в sub_1135B40.


5. РЕГИСТРОВЫЕ ИНВАРИАНТЫ
================================================================================
ECX = 0: this равен нулю во всех точках вызова.
ESI = 0: this, переданный через ECX, скопирован в ESI в прологе функции.
EDI = 0x0008BB9C: возможно, адрес временного буфера или строки.
faulting = 0x6 = 0 + 6 — подтверждает null-deref со смещением поля +6.

Значения других регистров штатные (EBX=0x001A86C0 — похоже на стек
вызывающего, EDX=0x12981F04 — возможный адрес строки имени arg_0).

7. ПЕРВОПРИЧИНА (обновлено с учётом показаний игрока)
================================================================================
Точка вызова (sub_113FAD0+0x460 = 0x0113FF30):

  loc_113FF0C:                ; case 1 (jump table 0x113FF05)
    mov  edi, [esi+4]         ; edi = ID участника из пакета
    mov  ecx, dword_2163AE4   ; ecx = глобальный контейнер слотов
    push edi
    call sub_1135B40          ; поиск слота по ID
    mov  ecx, eax             ; ecx = найденный слот (или 0)
    test ecx, ecx
    jnz  loc_113FF29          ; если найден -> идти на инициализацию
    push edi
    call sub_1135BA0          ; fallback: поиск слота с id=0 (свободный)
    mov  ecx, eax             ; ecx = результат fallback (или 0)
    ; *** ЗДЕСЬ НЕТ проверки ecx == 0 ***

  loc_113FF29:
    push dword ptr [esi+2Ch]  ; arg_4 = флаг
    lea  eax, [esi+1Ch]       ; arg_0 = строка (имя участника)
    push eax
    call sub_1135AA0          ; sub_1135AA0(this=ecx, ...)

Функция sub_1135BA0 возвращает NULL, если ни один из 8 слотов не
имеет ID = 0 (нет свободного слота).

ПЕРВОПРИЧИНА (двухуровневая):
  1) НЕПОСРЕДСТВЕННАЯ: в вызывающем коде (sub_113FAD0, case 1)
  ОТСУТСТВУЕТ проверка результата поиска после fallback-вызова
  sub_1135BA0. Когда таблица не содержит слота с ID=0, sub_1135BA0
  возвращает 0, и этот ноль без проверки передаётся как this в
  sub_1135AA0, что приводит к разыменованию нулевого указателя.

  2) КОРЕННАЯ: Таблица (dword_2163AE4) выделяется через
  operator new(0xBB), который НЕ обнуляет память. Конструктор
  sub_1135B10 — пустой (mov eax, ecx; retn). Сброс через
  sub_1135B20 (memset) вызывается УСЛОВНО (см. ниже).
  В результате 8-й слот (неиспользованный) содержит
  НЕИНИЦИАЛИЗИРОВАННЫЙ МУСОР КУЧИ в поле ID вместо 0.

  По показаниям игрока: 6 из 8 слотов занято, изначально было 7,
  один вышел. Выход игрока НЕ привёл к освобождению слота
  (сообщение 0x3F6 не было отправлено/обработано при отключении).
  Следовательно, в таблице: 7 слотов с валидными ID (6 активных +
  1 zombie ушедшего) + 1 слот с мусором (не 0).
  Поиск ID=0 не находит ни одного → sub_1135BA0 возвращает NULL.

  Три точки сброса memset (sub_1135B20) и их условия:
    - sub_113E1C0: только если ds:69959Ch == 0
    - sub_1137DB0: только если sub_109C1B0() == 0
    - sub_113BA50: безусловно, но не гарантирован на пути
  Если условия не соблюдены, таблица никогда не обнуляется.

Проверка родственных функций (sibling): после вызова sub_1135B40 (первый
поиск) проверка `test ecx,ecx; jnz` ЕСТЬ. После второго поиска
(sub_1135BA0) такая проверка ОТСУТСТВУЕТ — это доказывает пропуск
проверки именно в этой функции (sub_113FAD0).

Дополнительные находки (исследование по реестру зацепок):

  П1. Функция освобождения слота (sub_1135BE0): вызывается при получении
  сообщения 0x3F6 (1014). Корректно обнуляет весь слот (от +0 до +16h) и
  декрементирует счётчик. Сама по себе работоспособна, но пути её вызова
  (0x3F6) не пересекаются с путём регистрации (0x8A2/2210). Освобождение
  происходит ТОЛЬКО по явному сообщению 0x3F6; если это сообщение не
  приходит (например, при разрыве соединения до отправки 0x3F6), слот
  остаётся занятым навсегда.

  П2. ВТОРОЙ ТАКОЙ ЖЕ БАГ в функции sub_1138690 (строка 528611):
  sub_1138690 также обрабатывает сообщение 0x8A2/2210, подтип 1, и
  содержит идентичный фрагмент кода:
    call sub_1135B40  ; поиск по ID
    test eax, eax
    jnz  loc_1138714  ; если найден -> инициализация
    push edi
    call sub_1135BA0  ; fallback
    ; *** НЕТ проверки eax == 0 ***
  loc_1138714:
    push dword ptr [esi+2Ch]
    lea  ecx, [esi+1Ch]
    mov  ecx, eax
    call sub_1135AA0  ; ECX может быть 0 -> тот же null-deref
  Это свидетельствует о систематическом паттерне: обе функции копированы
  из общего шаблона с одинаковым пропуском проверки.

  П3. Сравнение с соседними подтипами (case 3/4/6/7 в sub_113FAD0):
    - Case 3: сбрасывает глобальный флаг, таблицу не трогает.
    - Case 6: обрабатывает строку через sub_1135D50, таблицу не трогает.
    - Case 7: делает поиск ID через sub_1135B40; при NULL (не найдено)
      уходит на default (бездействие). Это ПРАВИЛЬНЫЙ паттерн —
      "не нашли → ничего не делаем" — в отличие от case 1, где попытка
      fallback-создания не защищена проверкой результата.
    - Case 4: обрабатывает данные через callback, таблицу не трогает.
    Таким образом, баг изолирован в case 1 и sub_1138690 — только там
    результат sub_1135BA0 используется как this без проверки.

  П4. Другие callers sub_1135BA0 (безопасные):
    - sub_1137DB0 (строка 528009): результат отбрасывается (mov ecx, 0).
    - sub_113FAD0+0x410 (строка 539911): результат НЕ используется как
      this, служит только для побочного эффекта (создание записи).

  П5. Анализ дампа процесса (.dmp): минидамп (MemoryList, без кучи).
  Глобальный контейнер dword_2163AE4 находится на куче и в дампе
  отсутствует (47 диапазонов памяти, всего 0 МБ кучи). Содержимое
  таблицы (число занятых слотов, какие ID) по статике не восстановимо.
  Для чтения таблицы нужен полный дамп (с кучей), включаемый настройкой
  <Sys.FullDump> = 1 в HD-моде.

10. ПРИЗНАКИ ПОИСКА В СТАРЫХ ВЕРСИЯХ HOTA / В SOD
================================================================================
Context-инварианты:
  - ESI = ECX = 0 (this = NULL) — основной инвариант.
  - faulting = 0x6 — запись по нулевому this на смещение +6.

Текстовые сигнатуры по дизассемблеру:
  - Паттерн: `mov esi, ecx; mov [esi+6], eax` при ECX = 0 в
    функции инициализации записи.
  - Участок вызова: `call sub_1135BA0; mov ecx, eax` БЕЗ `test ecx, ecx`
    после `call`.
  - Вторая копия бага: в sub_1138690 (HD_HOTA.dll-5.7 R20, строка
    528611, caller на 528665) — идентичный фрагмент:
    `call sub_1135BA0; mov ecx, eax; call sub_1135AA0` без проверки.

Стабильные адреса h3hota.exe (ядро цепочки):
  - A0_DialogFunction_sub_5FFAC0 (0x005FFAC0)
  - A0_WndMgr_Run_MapSelectDlg_sub_602D60
  - A0_MapSelectDlg_Run_sub_584EC0

Сравнение с родственными функциями:
  - В той же sub_113FAD0 поиск через sub_1135B40 (ПЕРВЫЙ вызов)
    защищён проверкой `test ecx,ecx; jnz`. Второй вызов (sub_1135BA0)
    такой проверки НЕ ИМЕЕТ. Это доказывает пропуск проверки
    во втором пути.

Оценка присутствия: код находится в HD_HOTA.dll (HD-мод), поэтому
в SoD отсутствует. Для старых версий HotA/HD-мода сигнатура
должна быть стабильной, так как функции sub_113FAD0 и sub_1138690 —
часть инфраструктуры сетевого лобби HD-мода.

Признак кросс-версионности: искать в листингах HD_HOTA.dll старых
версий паттерн `call sub_1135BA0; mov ecx, eax; <без test ecx,ecx>;
call sub_1135AA0` — обе функции (sub_113FAD0 case 1 и sub_1138690).
Наличие такого паттерна в любой версии HD-мода означает наличие того
же бага.


11. ВОСПРОИЗВЕДЕНИЕ ВЫЛЕТА
================================================================================
Детерминированность бага: УСЛОВНО ДЕТЕРМИНИРОВАННАЯ — срабатывает
при заполнении таблицы участников (8/8 слотов) и получении сообщения
подтипа 1 для несуществующего в таблице ID.

Условие активации пути (дизассемблер, sub_113FAD0):
  1) [ESI+8] == 0x8A2 (2210) — тип сообщения.
  2) [ESI+18h] == 1 — подтип (case 1 в jump table).
  3) Поиск ID по sub_1135B40 и sub_1135BA0 даёт NULL (таблица полна).

Сценарий воспроизведения:
  1) Запустить сетевую игру HoMM3 HotA 1.8.0 с HD-модом 5.7 R20.
  2) В лобби подключить 8 участников (максимум).
  3) Выбрать карту (любую) и перейти к согласованию старта.
  4) Нажать "Принять" (подтверждение условий).
  -> Вылет при обработке повторного или дополнительного сообщения
     о согласовании, когда таблица участников уже заполнена.

Вероятностный путь (если 8 участников недоступны):
   - Провести несколько сессий согласования/отмены старта игры
     (чтобы исчерпать слоты таблицы через накопление временных
     записей).
   - Или модифицировать сетевой трафик, создав ситуацию, когда
     таблица участников переполнена.


ВЫВОД
================================================================================
Вылет — результат двух сцепленных дефектов HD_HOTA.dll (HD-мод 5.7 R20):

  1) Пропущенная проверка NULL после sub_1135BA0 в sub_113FAD0 (case 1)
     и sub_1138690 — результат fallback-поиска свободного слота не
     проверяется перед вызовом sub_1135AA0.
