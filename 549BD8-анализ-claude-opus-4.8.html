Визуализация вылета #

Обход реки идёт «назад» по ссылкам на предыдущую ячейку (prev) и должен остановиться на ячейке-истоке (стоимость == 0). Если конечная ячейка реки не была достигнута поиском пути, она хранит исходные служебные значения (prev = (-1,-1,-1), стоимость = 0x7D00), обход не распознаёт их и считает по -1 отрицательный индекс клетки.

Findex < 0
Bindex 0
массив клеток0 .. W*H-1
конецW*H-1

запись по index*48 уходит ЛЕВЕЕ базы массива (index отрицательный) — мимо выделенной памяти

B B — база массива клеток (регистр EAX в момент сбоя, поле [edi+8]).
F F — адрес записи base + index*48 + 0x28 при index = -(1 + W*(1+H)); для карты 108×108 index = -11773, адрес левее базы.

Как возникает служебный prev: бит стока реки на ячейке (поле +0x28) и её стоимость/prev (поля +0x1C/+0x10) — НЕЗАВИСИМЫ. Бит стока реки ставится раньше (периметр / предыдущие реки) и НЕ сбрасывается инициализацией. Если волна поиска пути не дошла до ячейки-стока (изолированный водоём), та проходит условие «cost ≠ 0», но её prev остаётся (-1,-1,-1):

initcost=0x7D00, prev=-1
волнаcost+prev вместе
рекаCreateAndStartRiver
обход prevбез guard -> СБОЙ

фазы внутри функции: служебный prev рождается на фазе волны (не дошла), а срабатывает на фазе обхода

Цепочка обхода: пока стоимость ячейки ненулевая, код берёт её prev и переходит к ячейке-предку. На исправной карте цепочка приходит в исток; при недостигнутой конечной ячейке первый же prev служебный (-1,-1,-1):

стокcost ≠ 0
prevшаг назад
недостигнутаcost = 0x7D00
prev = (-1,-1,-1)-> index < 0

служебная ячейка проходит условие «cost ≠ 0», но guard'а на (-1,-1,-1) нет — обход шагает за карту

S S — ожидаемый исток: стоимость с младшими 16 битами == 0, нормальный конец обхода.
X X — недостигнутая конечная ячейка: остаётся в исходном состоянии, её prev служебный и уводит индекс в отрицательную область.

Краткое резюме¶

Игра падает во время ГЕНЕРАЦИИ СЛУЧАЙНОЙ КАРТЫ, на этапе прокладки рек. Код идёт «назад» по цепочке предыдущих ячеек реки, помечая каждую флагом. Конец цепочки он ожидает узнать по ячейке-истоку (поле стоимости обнулено). Но если волна поиска пути НЕ дошла до выбранной конечной ячейки, та остаётся в исходном состоянии: стоимость = 0x7D00 («недостижимо»), а координаты предка = (-1,-1,-1). Обход не распознаёт это служебное значение, подставляет -1 в формулу индекса клетки и получает большой ОТРИЦАТЕЛЬНЫЙ индекс — запись уходит далеко за пределы массива клеток.

Тип исключения: EXCEPTION_ACCESS_VIOLATION (запись). Инструкция сбоя: or dword ptr [esi+28h], 40000000h по адресу 0x00549BD8 в h3hota.exe. Корневой регистр ESI = база массива клеток + индекс*48; во всех логах индекс отрицательный, потому что взят из служебного предка (-1,-1,-1).

Сопоставление адресов¶

Формула: IDA_addr = runtime_addr - load_base + imagebase. Для h3hota.exe: load_base = 0x00400000 (секция Modules лога), imagebase = 0x00400000 (шапка листинга). Значит для этого модуля IDA-адрес равен runtime-адресу напрямую.

Механика отказа: цикл читает координаты предка из поля по смещению +0x10 (prev_xyz), считает линейный индекс клетки index = prev_x + x_size*(prev_y + y_size*prev_z), переводит в байтовое смещение (index*48), прибавляет базу массива клеток и пишет флаг 0x40000000 в поле +0x28. Когда предок служебный (-1,-1,-1), index = -1 + x_size*(-1 + y_size*(-1)) — большое отрицательное число, и адрес esi+0x28 попадает мимо буфера.

Регистровые инварианты¶

EDX = 0xFFFFFFFF во всех логах. EDX загружается на 549BB1 как prev_y = [esi+4] и сразу участвует в расчёте индекса. EDX = -1 прямо доказывает, что обход взял служебного предка (prev_y=-1).

EBX = 0x0000FFFF во всех логах. Это константа маски из кода (mov ebx, 0FFFFh на 549B9D), которой проверяются младшие 16 бит поля стоимости. Инвариант подтверждает, что упавший кадр — именно этот цикл.

ESI = cells_base + index*48. Восстановление индекса: index = (ESI - EAX) / 48, где EAX на момент сбоя = cells_base ([edi+8], загружен на 549BD0). Для лога 00b2eaa0: (0x15C720B4 - 0x15CFC024) / 48 = -565104 / 48 = -11773.

Связь индекса со служебным предком (-1,-1,-1): index = prev_x + w*(prev_y + h*prev_z) = -1 + w*(-1 + h*(-1)) = -(1 + w + w*h) = -(1 + w*(1+h)). Для -11773: w*(1+h) = 11772 = 108*109 -> w=108, h=108. Для лога 1355151c: index=-20881, w*(1+h)=20880=144*145 -> w=h=144. Разные карты — один и тот же служебный предок.

faulting_addr - ESI = 0x28 во всех логах (поле, в которое идёт запись флага). Адресная проверка сошлась: ESI + 0x28 == write of address.

Первопричина¶

Поставщик данных (инициализация ячеек) — в начале той же функции, 0x549343..0x54937A: цикл проставляет КАЖДОЙ ячейке стоимость с младшими битами 0x7D00 (32000, «недостижимо») и координаты предка (-1,-1,-1):

.text:00549352   mov  edx, [eax]          ; поле стоимости
.text:00549354   and  edx, 0FFFF7D00h     ; low16 := 0x7D00
.text:0054935A   or   dh, 7Dh
.text:0054935D   mov  [eax], edx
.text:00549365   mov  [edx], ecx          ; prev_x = -1
.text:0054936A   mov  [edx+4], ecx         ; prev_y = -1
.text:0054936D   mov  [edx+8], ...         ; prev_z

Далее ячейка-исток получает стоимость с младшими 16 битами = 0 (терминатор обхода), а волна поиска пути должна заполнить достижимым ячейкам корректные prev и стоимость.

Сама волна поиска пути (цикл 0x549502..0x549797 той же функции) — это поиск в ширину со случайными весами: соседней ячейке назначается стоимость (rand()&0x1F) + стоимость_текущей + 1 (0x549643), и ВМЕСТЕ с ней — prev = координаты текущей (0x5496A8..0x5496CF). То есть корректный prev появляется ТОЛЬКО у ячейки, до которой дошла волна; недостигнутая ячейка сохраняет служебный prev=(-1,-1,-1). Примечательно, что сама волна ПРОВЕРЯЕТ границы соседа перед обращением к ячейке:

.text:005495C8   jl   ...   ; x < 0       -> пропуск
.text:005495D4   jge  ...   ; x >= x_size  -> пропуск
.text:005495DC   jl   ...   ; y < 0       -> пропуск
.text:005495E5   jge  ...   ; y >= y_size  -> пропуск

А обратный обход prev (0x549BA6) такой проверки НЕ делает.

Конечные ячейки (стоки рек) размечает соседняя функция A0_ARandMapGen_MarkRiverSinks_sub_549160 битом 0x40000000 в поле +0x28 в ДВУХ местах: (1) фаза внутренних устьев (0x5491A6 -> CheckMarkAvailRiverDelta_sub_548F30) метит сухопутные клетки-устья у подходящей береговой геометрии (биты направления 0x40000000 на 0x54913B..0x549146); (2) фаза периметра (циклы 0x549211 и 0x549267) безусловно метит ВСЕ краевые ячейки (x=0 / x=max и y=0 / y=max). Этот же бит 0x40000000 проверяет волна как признак достигнутого стока (бит 30 в 0x54974F): найдя такую ячейку, волна останавливается. Важно: инициализирующий цикл (0x549343) сбрасывает стоимость (+0x1C) и prev (+0x10), но НЕ трогает поле +0x28 — биты стоков/флаги переживают между вызовами размещения рек.

ТОЧНЫЙ ДЕФЕКТ — пост-проверка после волны (0x549797). Когда очередь волны опустела, функция решает, начинать ли обход, исключительно по биту стока реки:

.text:00549797   mov  eax, [esi+28h]
.text:0054979A   mov  ecx, eax
.text:0054979C   shr  ecx, 1Eh
.text:0054979F   test cl, 1            ; бит 30 (сток) ?
.text:005497A2   jnz  short loc_549805 ; да -> начать обход

Проверяется ТОЛЬКО бит стока реки (0x40000000), НО НЕ проверяется, была ли ячейка РЕАЛЬНО ДОСТИГНУТА волной (cost.low16 == 0). Это и есть дефект: бит стока реки — это «ячейка ПОМЕЧЕНА как сток реки», а НЕ «сток достигнут».

Почему помеченный, но недостигнутый сток возможен. Общее условие: выбранный сток не лежит в связной области истока, поэтому волна его не релаксирует. Два гарантированных подкласса: (а) сток на клетке периметра типа вода/скала — волна их В ПРИНЦИПЕ не проходит (гейты приёма соседа):

.text:0054960A   cmp  eax, 8           ; terr == 8 (вода) ?
.text:0054960D   jz   loc_54975A       ; да -> не релаксировать
.text:00549613   cmp  eax, 9           ; terr == 9 (скала) ?
.text:00549616   jz   loc_54975A       ; да -> не релаксировать

(б) любой сток (периметр или внутреннее устье от 548F30) в зоне, ОТРЕЗАННОЙ от истока водой/рельефом. В обоих подклассах помеченная ячейка никогда не релаксируется и сохраняет cost.low16=0x7D00 и prev=(-1,-1,-1). Когда именно такая клетка оказывается конечной, пост-проверка 0x549797 видит пометку стока реки и запускает обход от неё.

Потребитель (цикл 0x549B9A) завершается ТОЛЬКО по условию cost.low16 == 0. У нерелаксированной ячейки cost.low16=0x7D00, поэтому условие никогда не выполняется сразу: первая же итерация берёт prev=(-1,-1,-1) (EDX=-1 во всех дампах) и считает отрицательный индекс — запись мимо массива. Дампы подтверждают, что выбранный сток был нерелаксирован (prev_y=-1, индекс раскладывается точно в prev=(-1,-1,-1)), т.е. пост-проверка 0x549797 пропустила недостигнутую ячейку в обход.

Между волной и обходом функция вызывает размещение реки (A0_RiverPlacer_CreateAndStartRiver_sub_55F080, 0x549892). Инициализирующий цикл 0x549343 сбрасывает cost (+0x1C) и prev (+0x10), но НЕ трогает поле +0x28 — бит стока реки переживает; это и делает пометку стока реки ненадёжным признаком достижимости.

Родственное подтверждение «своего» паттерна границ: соседняя функция того же генератора A0_ARandMapGen_MarkRiverSources_sub_549C90 ПЕРЕД вычислением адреса клетки проверяет координаты на диапазон:

.text:00549D09   test edx, edx
.text:00549D0B   jl   short loc_549D3A    ; x < 0  -> пропуск
.text:00549D0D   cmp  edx, [ecx+18h]
.text:00549D10   jge  short loc_549D3A    ; x >= w -> пропуск
.text:00549D12   test esi, esi
.text:00549D14   jl   short loc_549D3A    ; y < 0  -> пропуск
.text:00549D19   cmp  esi, eax
.text:00549D1B   jge  short loc_549D3A    ; y >= h -> пропуск

То есть в самой подсистеме есть образец проверки координат перед обращением к клетке, но обход дельты реки (0x549B9A) его не делает.

Вывод о первопричине: падающий цикл логически корректен ПРИ соблюдении контракта «у каждой ячейки цепочки prev есть достижимый исток (cost==0)». Контракт нарушается там, где волна поиска пути останавливается на ячейке с битом стока реки (0x40000000), НО эта ячейка не была релаксирована волной и сохраняет служебный prev=(-1,-1,-1). Такое сочетание возможно, потому что бит стока реки (поле +0x28) и стоимость/prev (поля +0x1C / +0x10) — НЕЗАВИСИМЫЕ данные: бит стока реки выставлен раньше (по периметру/при предыдущих реках) и НЕ сбрасывается инициализацией, а волна под некоторыми зёрнами не доходит до этой ячейки. При этом сама волна и соседняя MarkRiverSources ГРАНИЦЫ ПРОВЕРЯЮТ, а падающий обход prev — нет; то есть в подсистеме есть собственный образец контроля границ, которого обходу не хватает.

Дефект локализован точно: это пост-проверка 0x549797, которая решает «начинать ли обход» по БИТУ СТОКА, а должна была бы по ФАКТУ ДОСТИЖИМОСТИ (cost.low16==0). Достаточное условие сбоя: конечный сток попадает на водяную/скальную клетку периметра (terr 8 или 9), которую волна по своим же гейтам никогда не релаксирует. Какой именно расклад карты/зерно выбирает такой сток (изолированный водоём у края, блокировка рельефом) — динамика, по статике до конца не выводится (см. раздел 9), но сама пропущенная проверка (достижимость вместо пометки) — конкретная.

Условия, спровоцировавшие сбой¶

Из логов: Game Type преимущественно 0 (одиночная игра), встречаются 2 и 3; Network Game как 0, так и 1; Day = 1 11 во всех логах (момент старта только что сгенерированной карты). Сбой происходит на этапе генерации карты, поэтому игровые поля «Map file»/тип игры на путь сбоя не влияют — значим лишь факт запуска генератора случайной карты.

Зерно генератора (Last RMG Seed) в логах разное (1768144658, 1767694387, 1775300020, ...). Это согласуется с зависимостью сбоя от зерна: только часть зёрен/конфигураций даёт недостижимую конечную ячейку реки.

Достоверность и ограничения¶

Верификация листинга: модуль h3hota.exe сопоставлен с Модули/h3hota.exe.txt по imagebase 0x400000; адрес сбоя найден точно. h3hota.exe — неизменяемый SoD-EXE, его адреса стабильны между версиями HotA, поэтому рассинхрона версии для ПАДАЮЩЕГО модуля нет.

Не смапплены кадры стека из диапазона 0x0292xxxx и записи HW_HOTA.dll/HD_HOTA.dll — это переходники patcher_x86 и вход HD-мода; листинга для JIT-переходников не существует, на диагноз они не влияют (виновный код в h3hota.exe).

Корреляция: 136 уникальных логов с одинаковыми инвариантами — надёжная база, единичным дампом диагноз не ограничен.

Обоснование вердикта полезности — ВЫСОКАЯ: место сбоя доказано арифметически (ESI+0x28, отрицательный индекс из служебного предка), класс установлен, а дефект локализован до конкретной инструкции В САМОЙ падающей функции. Пост-проверка 0x549797 решает «начинать ли обход» по БИТУ СТОКА (поле +0x28, бит 30), тогда как корректный критерий — ФАКТ ДОСТИЖИМОСТИ (cost.low16==0). Бит стока реки выставлен MarkRiverSinks (внутренние устья через 548F30 и безусловно весь периметр) НЕЗАВИСИМО от достижимости, а сток вне связной области истока (вода/скалы terr 8/9 или отрезанная зона) волной никогда не релаксируется — значит помеченный, но недостигнутый сток возможен. Это вычисленный самим кодом неверный предикат, а не просто «нет guard на недоверенный вход»: функция сама производит неверное решение о начале обхода. Точечный фикс (сверять достижимость, а не пометку) идентифицируем и находится именно в этой функции. Ограничение: конкретный расклад карты/зерно, дающий недостижимый сток-на-воде, по статике не воспроизводится (это не мешает фиксу, т.к. проверка достижимости закрывает любой такой расклад).

Признаки поиска в старых версиях HotA / в SoD¶

Контекст-инварианты (по логам): EDX = 0xFFFFFFFF (служебный prev_y = -1), EBX = 0x0000FFFF (маска младших 16 бит из кода), запись по адресу ESI+0x28, отрицательный индекс клетки, (faulting - cells_base) кратно 48.

Текстовая сигнатура по дизассемблеру (для листинга любой версии): обратный обход с расчётом индекса клетки без клампинга —

mov  edx, [esi+4]          ; prev_y
imul eax, [edi+10h]        ; * y_size
add  eax, edx
imul eax, [edi+0Ch]        ; * x_size
add  eax, ecx              ; index
lea  esi, [eax+eax*2]
shl  esi, 4                ; index*48
add  esi, [edi+8]          ; + cells_base
or   dword ptr [esi+28h], 40000000h

Стабильные адреса h3hota.exe (общие для всех версий HotA, т.к. EXE не меняется): цикл сбоя 0x549BD8, функция 0x5492E0 (CreateRiverWithDelta), вызывающая 0x549D60 (CreateRivers), соседняя проверка координат в 0x549C90 (MarkRiverSources), разметка стоков периметра в 0x549160 (MarkRiverSinks).

Родственные пути без guard'а: в листинге искать всех вызывающих функцию размещения дельты реки и проверять, есть ли перед расчётом индекса проверка служебного предка / границ (в обходе 0x549B9A её нет; в 0x549C90 и во внутренней волне поиска 0x5495C8..0x5495E5 — есть).

О присутствии в SoD / старых версиях (ПОДТВЕРЖДЕНО дампами): весь виновный код расположен в главном EXE (h3hota.exe — неизменяемый под HotA SoD-производный EXE), а не в hota.dll. Найдены краш-логи С ТЕМ ЖЕ адресом сбоя 0x00549BD8 в режиме SoD (модуль Heroes3 HD.exe / HEROES~1., база образа 0x400000) с датами 2022 года — то есть задолго до HotA 1.8.0. В этих логах те же регистровые инварианты (EDX=0xFFFFFFFF, EBX=0x0000FFFF, faulting=ESI+0x28) и та же цепочка вызовов (CreateRivers 0x549D60 -> CreateRiverWithDelta 0x5492E0 -> 0x549BD8). Часть старых логов (HD-мод <= 5.4) пишет тот же адрес в offset-форме HEROES~1.+0x149BD8 (0x400000 + 0x149BD8 = 0x549BD8). Поскольку base образа = imagebase = 0x400000, эти логи мапятся на тот же листинг h3hota.exe напрямую. Вывод: дефект — часть базового RMG, унаследованного от SoD, а не HotA-специфика. Ограничение: отдельного листинга чистого SoD-EXE (без HD-мода) нет, но совпадение base 0x400000 и адресов делает маппинг на имеющийся листинг корректным.

Воспроизведение вылета¶

Уровень воспроизводимости: НЕДЕТЕРМИНИРОВАННЫЙ (зависит от зерна генератора). Гарантированной последовательности «шаг -> краш» нет; нужен массовый прогон генерации карт.

Роль 1 — тестер за игрой (без отладчика): 1) Запустить HotA 1.8.0, режим «Сгенерировать случайную карту». 2) Брать БОЛЬШИЕ размеры карт (Large/XL/Huge/Giant — восстановленные размеры 108..252), на которых реки длиннее и чаще тянут дельты. 3) Многократно генерировать карты, перебирая разные зёрна (в т.ч. шаблоны с обилием воды/рек). «Что ловить» = сам факт вылета с адресом 0x00549BD8. 4) Фиксировать: зерно (Last RMG Seed), размер карты, шаблон, повторяемость.

Роль 2 — подтверждение причины (с отладчиком / минидампом):

  • Точка останова на 0x00549BD8. Причина подтверждена, если EDX = 0xFFFFFFFF (prev_y = -1) и cells_base + index*48 + 0x28 == faulting_addr при отрицательном index = (ESI - cells_base)/48.
  • Условный останов, пропускающий итерацию при prev == (-1,-1,-1), должен убрать именно этот вылет — это подтверждает, что обход следует за служебным предком.

Дифференциальный тест причины: на зерне-репродьюсере поставить останов на 0x549797 и снять у выбранной ячейки-стока поле стоимости (+0x1C): если cost.low16 != 0 (напр. 0x7D00) при выставленном бите стока реки — ячейка помечена, но не достигнута, и именно на этом входе обход уйдёт за массив. Контрольный патч (заменить проверку «бит стока реки 0x40000000» на «бит стока реки И cost.low16 == 0» либо добавить в обход проверку prev != (-1,-1,-1)) должен убрать именно этот вылет — это подтверждает первопричину.

Вывод¶

Виновник — генератор случайных карт игры (h3hota.exe), функция размещения дельты реки sub_5492E0. Перед обратным обходом реки функция проверяет лишь пометку стока реки ячейки (0x549797), а не факт её достижимости поиском пути (стоимость == 0). Поскольку пометка стока реки проставлена на всём периметре карты (включая воду/скалы, которые поиск пути принципиально не обходит), помеченная, но недостигнутая ячейка-сток сохраняет служебный prev=(-1,-1,-1). Тогда обход вычисляет большой отрицательный индекс клетки и записывает флаг далеко за пределами массива клеток — отсюда нарушение доступа на запись. Дефект локализован в самой функции: проверяется пометка стока реки вместо факта достижимости. Дефект — часть базового генератора, унаследованного от SoD (подтверждён дампами SoD-режима 2022 года), а не HotA-специфика.
