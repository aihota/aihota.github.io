Визуализация вылета #

Два независимых пути HotA/SoD приходят к одному дефекту: результат выделения памяти (форма без исключения) не проверяется на NULL и сразу используется как приёмник _memset. Под давлением 32-битной кучи выделение возвращает 0 — запись уходит по адресу 0.

A — Ресурсы / DEF (~78): загрузка спрайтов и DEF; нижний кадр A0_Load_Def. Размер — размер ресурса, ESI хранит реальный указатель.
B — Расчёт ИИ (~31): буфер _AIMapMoveInfo_ (30 байт на клетку); размер = 30 * сторона карты (M/L/XL/H/XH/G), ESI = 0, ход ИИ.

Краткое резюме¶

В 32-битной игре заканчивается непрерывный блок памяти. Очередной запрос на выделение буфера возвращает "пусто" (указатель 0), но код не проверяет этот результат и сразу пытается обнулить несуществующий буфер. Отсюда запись по адресу 0.

Code : EXCEPTION_ACCESS_VIOLATION Information : write of address: 0x00000000 Где : hota.dll+0x205E34 == CRT _memset, инструкция rep stosb Корневой : EAX=0 (результат выделения), EDI=0 (приёмник _memset), ECX==EDX>0 (размер обнуления)

Это memset(NULL, 0, N). Во всех 109 уникальных вылетах низкоуровневый инвариант один. Сходятся два независимых пути HotA/SoD к одному и тому же дефекту, что и даёт устойчивый адрес hota.dll+0x205E34. Один из вылетов впервые сопровождён минидампом процесса: регистры совпали с текстовым логом байт-в-байт, а сырой стек дал чистую цепочку вызовов группы A (ресурсы/DEF), которую зашумлённый текстовый лог не показывал.

Контекст исключения совпал с текстовым логом во всех регистрах (EAX=0, EDI=0, ECX=EDX=EBP=0x2A701D, ESI=0x0C3CAF08), EIP = hota.dll+0x205E34. Сырой стек по ESP дал чистую цепочку, которой не было в зашумлённом логе:

[ESP+04] = 0x030B1675 адрес возврата → hota.dll+0x161675 (инструкция сразу за call _memset, ветка 1) [ESP+08] = 0x00000000 void * приёмник (результат new = NULL) [ESP+0C] = 0x00000000 Val [ESP+10] = 0x002A701D Size (совпадает с ECX/EDX)

Адрес возврата 0x030B1675 однозначно указывает на ветку 1 функции sub_101615B0 (выделение буфера через [esi+8], размер хранится в EBP), а выше по стеку — A0_Load_Def (h3hota+0x55CB64). Текстовый лог локализовать вылет не мог, а минидамп подтвердил путь группы A на рантайме.

Кадры стека ВЫШЕ A0_Load_Def (восстановлены из минидампа, в текстовом логе их не было) уточняют контекст именно этого экземпляра группы A:

[ESP+1C/38] = 0x006600F8  -> строка .data 'secsk82.def'
                          (иконка вторичного навыка, 82 px)
[ESP+00/24] = 0x0069DBCC  -> в .data рядом с деревом ресурсов
[ESP+26C]   = 0x0069E560  -> A0_GameResourceTree_dword_69E560

(глобальное дерево ресурсов игры)

Строка 'secsk82.def' по XREF грузится из A0_MsgDlgItem_Init_sub_4F5540 (инициализация элемента диалогового окна сообщения); рядом в .data лежат imrl82.def, spellScr.def, resour82.def — иконки popup-диалогов. То есть объект-владелец буфера [edi+108h] — это ГЛОБАЛЬНОЕ дерево ресурсов в .data (не heap-объект — поэтому saved EDI=0x69DBCC указывает в .data), а конкретный триггер этого вылета — открытие popup-диалога с загрузкой его DEF-иконки. Оговорка: адреса .data читаются из образа точно, но их присутствие на стеке — это наблюдение по одному экземпляру (кучи в минидампе нет, цепочку объектов не проследить); вся группа A охватывает ~78 разных ресурсов, не только диалоговые иконки.

Различия (карты, дни, игроки, версии HD-мода, размеры буфера) при одинаковом низкоуровневом инварианте подтверждают системный шаблон, а не локальный дефект.

Функция _memset

.text:10205E10 _memset proc near
.text:10205E10     mov   edx, [esp+Size]
.text:10205E14     mov   ecx, [esp+arg_0]
.text:10205E18     test  edx, edx
.text:10205E1A     jz    short toend
.text:10205E1C     movzx eax, byte ptr [esp+Val]
.text:10205E21     bt    dword_14676BA8, 1
.text:10205E29     jnb   short SSE2CheckA
.text:10205E2B     mov   ecx, [esp+Size]
.text:10205E2F     push  edi
.text:10205E30     mov   edi, [esp+4+arg_0]
.text:10205E34     rep stosb               <- СБОЙ

На входе фактически memset(arg_0=NULL, Val=0, Size=ECX). Приёмник EDI загружается из arg_0 (= NULL) и rep stosb пишет первый байт в адрес 0. EDX и ECX оба берутся из Size, поэтому в Context ECX == EDX == размер.

Регистровые инварианты¶

EAX = 0 Ноль распространился без проверки. EDI = 0 приёмник _memset (arg_0). Подтверждает адресную проверку: destination == faulting address == 0. ECX = EDX > 0 размер. В _memset Size читается дважды (в EDX и ECX), отсюда равенство.

Группа A (ресурсы): размер = размер загружаемого ресурса (DEF-спрайт, LOD-данные), переменный, привязки к геометрии карты нет.

Группа B (ИИ): размер кратен 30 и равен 30 * сторона карты. Шесть наблюдаемых значений отображаются один к одному на стандартные стороны карт HoMM3:

0x870 = 30 * 72 (M) 0xCA8 = 30 * 108 (L) 0x10E0 = 30 * 144 (XL) 0x1518 = 30 * 180 (H, только HotA) 0x1950 = 30 * 216 (XH, только HotA) 0x1D88 = 30 * 252 (G, только HotA)

30 байт на клетку соответствует структуре _AIMapMoveInfo_ AI-планировщика.

Буфер группы B размером 30 * сторона карты соответствует структуре _AIMapMoveInfo_ (30 байт на клетку). В чистом движке SoD такой буфер выделяет A0_AIMgr_ClearCellsOrHexes_sub_4B11C0 (h3hota.exe). HotA перехватывает функции AI-пути (sub_42EDD0, sub_42F570, точку вызова h3hota+0x42EE3E) трамплинами patcher_x86: по листингу hota.dll этих адресов нет, поэтому точный адрес реализации аллокатора HotA по листингу не виден. Кадры hota.dll выше _memset в Call stack V2 логов группы B (sub_100179B0, sub_100181B0) при проверке оказались строковым форматировщиком, не связанным с этим выделением, — это искажение битого стека после rep stosb, а не настоящая цепочка.

Минидамп процесса подтвердил цепочку группы A на рантайме: адрес возврата на стеке (hota.dll+0x161675) попадает ровно за call _memset ветки 1 sub_101615B0, выше — A0_Load_Def (h3hota+0x55CB64). Раскладка стека приведена в разделе 2.

Кадры HD_HOTA.dll и raw-адреса вида 0x02ADxxxx в этих логах — внешний цикл сообщений и трамплины patcher_x86, а не отдельная причина: фолт происходит ниже, в коде HotA/SoD.

Первопричина¶

Один дефект на нескольких call-site: результат выделения памяти не проверяется на NULL перед использованием как приёмник _memset.

Группа A, дефектный фрагмент HotA (sub_101615B0), ветка 1:

.text:10161655     mov   ebp, [esi+8]
.text:10161658     xor   ebp, eax
.text:1016165A     mov   eax, 617492h        ; operator new(uint)
.text:1016165F     push  ebp
.text:10161664     call  eax
.text:10161666     push  ebp             ; Size
.text:10161667     push  0               ; Val
.text:10161669     push  eax             ; void *  (= результат new)
.text:1016166A     mov   [edi+108h], eax
.text:10161670     call  _memset             <- нет test eax,eax

Ветка 2 той же функции имеет тот же дефект:

.text:10161746     push  dword ptr [esi+14h]
.text:10161749     mov   ebx, 617492h        ; operator new(uint)
.text:1016174E     call  ebx
.text:10161750     mov   [edi+108h], eax
.text:10161756     push  dword ptr [esi+14h] ; Size
.text:10161759     push  0               ; Val
.text:1016175B     push  eax             ; void *
.text:1016175C     call  _memset             <- нет test eax,eax

Аллокатор 0x617492 (h3hota.exe) — форма без исключения (nothrow):

.text:00617492     push  1
.text:00617494     push  [esp+4+a1]
.text:00617498     call  __nh_malloc
.text:0061749F     retn

__nh_malloc при нехватке памяти возвращает NULL, а не бросает исключение. Между call new и call _memset нет проверки test eax,eax / jz. NULL сохраняется в [edi+108h] и сразу идёт в _memset как приёмник.

Внутренняя несимметрия как доказательство: та же sub_101615B0 ПЕРЕД повторным использованием буфера проверяет существующий указатель (.text:10161647 cmp dword ptr [edi+108h], 0; jnz ...; и .text:1016173D аналогично), но результат СВЕЖЕГО выделения в memset не проверяет. То есть проверка указателя в подсистеме есть, но в горячем пути выделения пропущена.

Группа B (ИИ): путь A0_AI_Hero_Setup_CellsFromAnotherHeroesMinWayCosts переписан в hota.dll (модуль сбоя = hota.dll при возврате в h3hota). Буфер _AIMapMoveInfo_ (30 * сторона карты) выделяется и обнуляется тем же шаблоном new → _memset; при отказе выделения приёмник = NULL.

Доказательство от родственного кода SoD. Оригинал того же буфера в движке SoD — A0_AIMgr_ClearCellsOrHexes_sub_4B11C0 (h3hota.exe):

.text:004B1228     lea   eax, [esi+esi*2]    ; eax = 3 * esi
.text:004B122B     lea   ecx, [eax+eax*4]    ; ecx = 15 * eax
.text:004B122E     shl   ecx, 1              ; ecx = 30 * esi (размер)
.text:004B1230     push  ecx
.text:004B1231     call  ??2@YAPAXI@Z        ; operator new(uint)
.text:004B1239     cmp   eax, ebx            ; ЕСТЬ проверка результата
.text:004B123B     jz    short loc_4B1243

Здесь esi = 2 * X * Y * (L+1), поэтому 30 * esi даёт ровно те значения размера, что видны в логах группы B (30 * сторона карты). Оригинал SoD ПОСЛЕ выделения сверяет результат с нулём, а деструктор и очистка _AIMgr_ (sub_4B1140, sub_4B1260) проверяют указатель перед delete. То есть в SoD проверки на месте, а путь HotA роняется по NULL там, где оригинал проверку имеет, — тот же системный шаблон, что и в группе A.

Технический корень семейства: контракт-рассогласование. Код рассчитан на бросающий operator new (как в шаблонах STL), но реально подключён nothrow-аллокатор SoD, возвращающий NULL. Поэтому в горячих путях систематически отсутствует проверка результата на NULL.

Условия, спровоцировавшие сбой¶

Условие ---------------------------------------------------------------- HotA 1.8.0 во всех логах 32-битный процесс (h3hota HD.exe) да Memory.Mod = 0 в распарсенных логах Нехватка непрерывной памяти главный триггер Ход ИИ (Active Player != Me) обязателен для группы B Большие/ресурсные карты часто, но не единственное

Главный триггер — недоступность непрерывного блока памяти в 32-битном адресном пространстве на момент очередного выделения, а не конкретная карта. Длинная партия, крупные карты, многопоточная графика и апскейл усиливают фрагментацию кучи. Из ini одного из AI-логов: Graphics.Mode=6, OpenGL.Upscaling=6, Graphics.Threads=1, Memory.Mod=0, Quick Combat=0.

Новый ресурсный лог (с минидампом) показывает обратную крайность: Graphics.Mode=5, OpenGL.Upscaling=0, одиночная игра, ход самого игрока, ранний Day 2 — и всё равно отказ выделения ~2.65 МБ под ресурс. То есть путь группы A срабатывает и без тяжёлой графики и без длинной партии: определяющим является доступность непрерывного блока памяти, а не конкретный графический режим или стадия игры.

Достоверность и ограничения¶

Подтверждено / высокая достоверность
  • Новый минидамп процесса — без полной кучи (MemoryList), поэтому ветка 1 группы A подтверждена по стеку и регистрам, но содержимое буфера и объект-владелец из дампа не читаются.
  • Кадры HD_HOTA.dll — внешний цикл/трамплины, не отдельная причина; листинги HD 5.6 не требуются (модуль сбоя hota.dll проверен).
  • Все логи — HotA 1.8.0. По набору нельзя утверждать, что дефект впервые появился именно в 1.8.0.
Ограничения и неизвестное
  • Точный адрес выделения+memset для AI-пути по листингу hota.dll не локализуется: HotA перехватывает SoD-функции AI-пути (sub_42EDD0, sub_42F570, h3hota+0x42EE3E) трамплинами patcher_x86, этих адресов в листинге нет, а кадры hota.dll выше _memset в Call stack V2 группы B — искажённые (оказались строковым форматировщиком). Путь и механизм подтверждены цепочкой вызовов, инвариантом размера (30 * сторона карты) и наличием проверки в оригинале SoD (sub_4B11C0).

Признаки поиска в старых версиях HotA / в SoD¶

Адрес hota.dll+0x205E34 между версиями не стабилен

Context-инварианты: Code : EXCEPTION_ACCESS_VIOLATION Information : write of address: 0x00000000 EAX = 0, EDI = 0, ECX == EDX > 0

Текстовая сигнатура точки сбоя (любая версия): _memset:

mov edi, [esp+...+arg_0]
rep stosb

Текстовая сигнатура call-site (ресурсный путь HotA):

mov   eax, 617492h        ; operator new
call  eax
push  size
push  0
push  eax
mov   [obj+108h], eax
call  _memset             ; нет test eax,eax между new и memset

Сигнатура группы ИИ: размер _memset кратен 30 и равен 30 * сторона карты (M/L/XL/H/XH/G), ход ИИ (Active Player != Me).

Стабильные адреса h3hota.exe для поиска как call <addr> в любой версии: 0x00617492 operator new (nothrow) / __nh_malloc 0x0055CB64 ресурсный/DEF путь около A0_Load_Def 0x00526C90 A0_AI_Hero_MakeTurnPart 0x0042E0B0 A0_AI_Get_BestCellProp_ForMovingTo 0x0042EDD0 A0_AI_Hero_Setup_CellVisitProp_List 0x0042F570 A0_AI_Hero_Setup_CellsFromAnotherHeroesMinWayCosts

В SoD прямой адрес hota.dll отсутствует (код в HotA-модуле), но nothrow-формы operator new унаследованы от SoD; искать по сигнатуре new → memset/copy без проверки результата.

Воспроизведение вылета¶

Вылет недетерминирован 32-битном процессе. Гарантированной последовательности шагов нет; ниже вероятностный путь для поимки минидампа.

Условие активации пути: Любое выделение в горячем пути (загрузка ресурса/DEF либо подготовка AI-буфера хода) возвращает NULL под давлением кучи, и результат сразу идёт в _memset.

Базовый сценарий (повышает вероятность): 1) Крупная карта (H/XH/G) с подземельем, много игроков ИИ. 2) Длинная партия (большой день) — куча фрагментирована. 3) Многопоточная графика и высокий апскейл (Graphics.Mode=6 OpenGL, OpenGL.Upscaling высокий) — больше крупных непрерывных буферов. 4) Дождаться хода ИИ (Active Player != Me) для группы B; для группы A — массовая загрузка спрайтов/ресурсов (вход в бой, экраны, прокрутка карты с разнообразными объектами). Подтверждённый минидампом конкретный путь группы A — открытие popup-диалога сообщения с иконкой (вторичный навык/награда): диалог грузит свою DEF-иконку (напр. secsk82.def) через дерево ресурсов. 5) Прогонять многократно на разных размерах карт и зёрнах ГСЧ.

Что ловить (подтверждение того же дефекта): EAX=0, EDI=0, ECX==EDX>0, write of 0x0 на hota.dll+0x205E34. Для группы B ECX = 30 * сторона карты; для группы A — размер ресурса и ненулевой ESI.

Дифференциальный тест причины: Снизить давление памяти (меньшая карта, меньше потоков/апскейла, короткая партия) — частота вылета падает или он исчезает, что подтверждает отказ выделения как активатор. Брейкпойнт на hota.dll+0x205E34: причина подтверждена, если приёмник (EDI) равен 0, то есть предшествующее выделение вернуло NULL.

Вывод¶

hota.dll+0x205E34 — общий CRT _memset, в который сходятся два пути HotA/SoD с одним дефектом: результат nothrow-выделения памяти не проверяется на NULL перед использованием как приёмник _memset, и при отказе выделения происходит запись по нулевому адресу. Подтверждённые пути в HotA 1.8.0 — ресурсный буфер sub_101615B0 (hota.dll, нижний кадр A0_Load_Def) и подготовка AI-буфера хода (реализация HotA поверх SoD-функции расчёта путей; размер строго 30 * сторона карты).
