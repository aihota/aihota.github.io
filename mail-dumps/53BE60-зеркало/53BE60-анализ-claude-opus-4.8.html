Визуализация вылета #

Связь зоны хранит указатель на настройки зоны-соседа. Два пути заполняют этот указатель по-разному, а один и тот же потребитель ему доверяет.

A Текстовый разборщик шаблона sub_5384E0 · 0x538751..0x53875B проверяет резолвер на NULL и отбрасывает связь — NULL в массив не кладёт
B Загрузчик связей зон HotA hota.dll · sub_101C6300 · 0x101C644F для токена Zone2=-1 НЕ отбрасывает связь, а сохраняет её с NULL-указателем партнёра
1 Запись связи (28 байт) connections.data + off · поле 0 another_zone_settings = edx
2 Потребитель доверяет входу sub_53BE60 · 0x53BF92 mov ecx, [edx] — читает id соседа без проверки на NULL
3 Сбой EXCEPTION_ACCESS_VIOLATION edx = 0 → чтение по адресу 0x0
A Штатный путь загрузки: контракт «у связи всегда валидный партнёр» соблюдён — NULL-связи отброшены, поэтому через загрузку шаблона сбой недостижим.
B Найденный поставщик: загрузчик связей HotA (sub_101C6300) для токена Zone2=-1 строит связь с NULL-партнёром (0x101C644F), тогда как текстовый разборщик SoD её отбрасывает. Это ОБЩИЙ парсер; зеркало включает флаг Mirror==2 (0x101C684B в загрузчике sub_101C6800): число зон делится пополам и достраивается отражением, поэтому NULL-партнёр доходит до потребителя у зеркальных mt_*/h3dm*. По комментариям в дампах падают именно самодельные зеркальные шаблоны (стоковый mt_Jebus не падает). Источник первопричины.
1 Потребитель корректно отсчитывает число связей (end−base)/28 и берёт указатель партнёра из записи — за границу массива не выходит.
2 3 По 57 дампам: EDX=0, чтение 0x0, EAX = число зон (1..0x11). Второй симптом той же зоны — деление на ноль на 0x53C059 (ESI=0, пустой список координат).

Краткое резюме¶

Генератор случайных карт перед тем, как поставить центр зоны, перебирает её связи с другими зонами: для каждой связи он берёт указатель на настройки зоны-соседа и читает у неё номер (id), чтобы притянуть центр поближе к уже размещённым соседям. Здесь и падает: у одной из связей указатель на зону-соседа равен NULL, и чтение её первого поля происходит по нулевому адресу.

Связи между зонами заполняются при загрузке шаблона. Текстовый разборщик SoD делает это аккуратно: если зона-сосед по номеру не найдена (функция поиска вернула NULL), связь целиком пропускается — NULL в массив связей он не кладёт. А собственный загрузчик связей HotA (sub_101C6300) для специального токена Zone2 = -1, наоборот, СОХРАНЯЕТ связь с NULL-партнёром. Именно этот NULL позже и разыменовывается.

ВАЖНОЕ УТОЧНЕНИЕ по комментариям самих отправителей в дампах: падают НЕ стоковые зеркальные шаблоны, а САМОДЕЛЬНЫЕ, сделанные на их основе. Дамп 0x02746163 (1.7.1): «При запуске своего шаблона, сделаного на подобии mt_Jebus, выдаёт ошибку. При запуске просто mt_Jebus никакой ошибки нет». Дамп 0x00D78F74 (1.6.1): «попытка сделать зеркальный шаблон, так как по старинке в редакторе через 1 и 101 зоны и т.п. почему-то не работает». То есть первопричина — НЕСБАЛАНСИРОВАННЫЙ зеркальный шаблон: связь ссылается на зону-двойник (по словам пользователя — «1 и 101», т.е. id и id+100), который в отражённом наборе отсутствует; резолвер возвращает NULL, и загрузчик HotA сохраняет битую связь. Стоковый mt_Jebus сбалансирован — потому не падает. Поле Map file в дампах (сценарии .h3m) и Last RMG Seed — шум (§14); зеркальная атрибуция доказана комментариями + цепочкой, а один дамп даже явно содержит строку mt_Jebus.

Исключение EXCEPTION_ACCESS_VIOLATION (чтение по адресу 0x00000000) происходит на инструкции

.text:0053BF92  mov  ecx, [edx]    ; <-- СБОЙ (чтение), edx = NULL

Здесь EDX — указатель another_zone_set_v14 на настройки зоны-соседа, взятый из записи связи; EAX — число зон в шаблоне (zones_count_v13). По всем 57 дампам с этим исключением EDX = 0 и адрес чтения = 0, а EAX принимает малые значения 1..0x11 (счётчик зон) — арифметика совпадает точно.

Внутри той же функции есть и второй, более редкий выход (2 дампа): деление на ноль на инструкции 0x53C059 div esi, где ESI = число кандидатных координат центра; когда список кандидатов пуст, ESI = 0. Это иной непосредственный симптом, но та же подсистема и тот же зеркальный сценарий (см. раздел 7).

Виновный модуль — hota.dll (загрузчик связей зон sub_101C6300): при ненайденном двойнике (токен Zone2=-1) он сохраняет связь с NULL-партнёром, нарушая контракт «у связи всегда валидный указатель», на который опирается унаследованная от SoD функция расстановки центра зоны (sub_53BE60 в h3hota.exe). Сама SoD-функция исправна — она лишь доверяет входу, как и в оригинале. Непосредственный триггер — невалидный (самодельный) зеркальный шаблон с несбалансированной связью (недоверенный вход). Исправление — на стороне hota.dll: отбрасывать связь с NULL-партнёром (как это делает текстовый разборщик SoD) либо валиди-ровать зеркальный шаблон при загрузке (предупреждать о битой связи вместо падения генератора).

Сопоставление адресов¶

IDA_addr=runtime_addr−load_base+imagebase.

Нативные кадры (h3hota HD, load_base = imagebase = 0x400000): runtime 0x0053C059 → IDA 0x0053C059 (div esi) — совпадает runtime 0x0053BE60 → IDA 0x0053BE60 (PlaceZoneCenter) — V1-anchor runtime 0x0053C2D2 → IDA 0x0053C2D2 (возврат в Create_Zones) runtime 0x00549FCE → IDA 0x00549FCE (возврат в ARandMapGen_Main)

Кадр faulting EIP (часть дампов 1.8.0): runtime 0x00F208AB / 0x02B308AB и т. п. — это НЕ адрес внутри h3hota.exe и не внутри листинга какого-либо модуля. Регион (≈0x00F00000 / 0x02B00000) выделяется patcher_x86 под codecave; HW_HOTA хукает RMG-функции (в HW_HOTA.dll видно push 549E20h — хук ARandMapGen_Main, и push 538000h и др.). Сопрограммная схема HotA: исполнение SoD-функции идёт через перенесённые в codecave инструкции, поэтому стек-уокер привязывает кадр к логическому SoD-адресу 0x53BE60 (строка «[ 0x0053BE60 ] called from [ 0x0053C2D2 ]»), а EIP указывает в трамплин. Логика инструкции сбоя при этом — ровно 0x53BF92 (mov ecx,[edx]); это подтверждают регистры (EDX=0, EAX=кол-во зон).

Листинга для трамплина не существует по определению (§17, §12) — запрос не требуется.

Точка сбоя — разбор инструкций¶

Симптом 1 (57 дампов), 0x53BF92 mov ecx, [edx]: edx = another_zone_set_v14 — указатель на _RmgZoneSettings_ зоны-соседа, прочитанный из записи связи: edx = *(void**)(zone_settings→connections.data + connect_off). Запись связи имеет шаг 28 байт (0x1C); первое поле (off 0) — указатель на настройки зоны-партнёра. Падение: edx = 0 → чтение [0] (поле id_dword0) → AV read 0x0. Адресная проверка (§4): faulting_addr = EDX = 0. op = чтение. ✓ EAX (zones_count_v13) = 1..0x11 — целиком согласуется с числом зон в шаблоне (sar eax,2 от размера вектора зон). ✓

Симптом 2 (2 дампа), 0x53C059 div esi: Перед делением: esi = v18 = (end − data)/12 — число элементов списка кандидатных координат центра (aval_center_xyz_list_v25). v21 = data[ 12 * (rand() % v18) ]. Если список пуст, v18 = 0 → div esi (esi=0) → INT_DIVIDE_BY_ZERO. Адресная проверка (§4, для div): делитель ESI = 0. ✓ Контекст (дамп 0x53C059): ESI=0, EDX=0 (обнулён перед div), EAX=0x4915 (делимое = 12*rand()). ✓ Поставщик списка (уточнёно чтением sub_53BE60 и sub_53B7E0): список aval_center_xyz_list наполняет GetCoordsToPlaceZoneNearIt (sub_53B370) В ДВА ПРОХОДА внутри sub_53BE60: сначала по зонам СО СВЯЗЯМИ (0x53BF98), и ЕСЛИ пусто — fallback-цикл по ВСЕМ зонам (0x53BFE7..0x53C01B). Затем SelectBestPlaces (sub_53B7E0, вызов 0x53C028) НЕ отбирает по связям, а СУЖАЕТ список по Z-уровню («на каждом уровне — хотя бы 1 зона», 0x53B848). div ловит пустоту ИТОГОВОГО списка. Связь с первопричиной НЕПРЯМАЯ: из-за fallback по всем зонам пустота НЕ сводится просто к «нет связей» — это вырожденная зона без допустимых координат вообще (геометрия/переполнение карты при зеркале). Поэтому симптом 2 — ТОГО ЖЕ зеркального сценария, но его связь с NULL-связью НЕ ДОКАЗАНА статикой (вероятна, но нестрогая гипотеза; может быть независимым путём).

Регистровые инварианты¶

По всем 57 AV-дампам: EDX = 0x00000000 — NULL-указатель зоны-соседа (= faulting) read of address: 0x0 — 57/57 EAX ∈ {1,3,4,5,6,7,8,9,0xB,0x11} — zones_count_v13 (число зон шаблона); распределение: 5→20, 9→8, 3→7, 6→5, 1→5, 0xB→4, 7→3, 0x11→2, 8→1, 4→1. Малые значения, как и должно быть для числа зон RMG. ESP=0x00196348, EBP=0x00196380 — устойчивы (один и тот же кадр). По 2 div-дампам: ESI = 0x00000000 — пустой список кандидатов (= делитель)

Вывод: регистровые инварианты однозначно фиксируют один дефект (NULL-связь) с двумя непосредственными проявлениями в одной функции.

Первопричина¶

7.1. Потребитель (где падает) доверяет входу. sub_53BE60 (h3hota.exe, унаследован от SoD) в цикле связей читает указатель на зону-соседа и сразу разыменовывает его (0x53BF92), не проверяя на NULL. Граница цикла связей берётся из размера вектора connections: count = (end − base)/28 (0x53BF4B..0x53BF6B). То есть потребитель проходит ровно по числу реально лежащих в массиве записей — за границу он НЕ выходит. Значит, NULL приходит из содержимого корректно отсчитанной записи, а не из перебора мусора.

7.2. Поставщик №1 — текстовый разборщик шаблона — NULL НЕ кладёт. sub_5384E0 (Load_rmg_txt), создание связи, 0x53873A..0x538761:

0x53873C call GetZoneSettings_By_Id ; rz_1 (зона A) → eax 0x538744 mov [ebp+rz_1_v61], eax 0x538747 call GetZoneSettings_By_Id ; rz_2 (зона B) → eax 0x538751 test edi, edi ; rz_1 == NULL? 0x538753 jz loc_538849 ; да → связь пропустить 0x538759 test ebx, ebx ; rz_2 == NULL? 0x53875B jz loc_538849 ; да → связь пропустить 0x538761 mov [ebp+connection_v50], ebx ; только теперь сохранить

Резолвер GetZoneSettings_By_Id (sub_533570) при ненайденном id возвращает NULL (loc_5335A6: xor eax,eax). Текстовый путь это проверяет и НЕ создаёт связь с NULL-партнёром. Следовательно, через обычную загрузку шаблона NULL-связь структурно недостижима, и граница потребителя согласована с поставщиком (§19, развилка 1: «упавший элемент недостижим корректной логикой данного пути»).

7.3. Поставщик №2 — НАЙДЕН: загрузчик связей зон HotA (sub_101C6300 в hota.dll). HotA своим хуком ставит собственную процедуру разбора связей зон; она, как и текстовый разборщик SoD, вызывает ТОТ ЖЕ резолвер зоны по номеру (call 533570h — GetZoneSettings_By_Id), но обрабатывает «не найдено» ИНАЧЕ:

.text:101C6435  test ebp, ebp
.text:101C6437  jnz  short loc_101C6442   ; B≠NULL → строить
.text:101C6439  cmp  ebx, 0FFFFFFFFh      ; токен зоны B == -1 ?
.text:101C643C  jnz  loc_101C6729        ; B=NULL и токен≠-1 → пропустить
.text:101C6442  ...                      ; B=NULL и токен==-1 → СТРОИТЬ
.text:101C644F  mov  [esp+78h+var_2C], ebp ; СОХРАНИТЬ партнёра = NULL

То есть для специального токена -1 (Zone2 = -1) загрузчик HotA НЕ отбрасывает связь, как это делает текстовый разборщик SoD (0x538751..0x53875B), а строит запись связи с НУЛЕВЫМ указателем партнёра (ebp=NULL сохраняется в запись). Именно этот NULL позже разыменовывает sub_53BE60 на 0x53BF92. Резолвер 0x533570 на ненайденном id штатно возвращает NULL (0x5335A6 xor eax,eax). Ветка присутствует ИДЕНТИЧНО в hota.dll 1.6.1, 1.7.3 и 1.8.0 (две call 533570h + cmp ebx,0FFFFFFFFh), что согласуется с дампами всех этих версий. Зон-мирроринга в h3hota.exe нет (там только MagicMirror боя и terr/road/river-зеркало клеток — не зоны), поэтому поставщик — именно hota.dll.

Где именно включается зеркало (разрешено). sub_101C6300 — ЭТО ОБЩИЙ парсер связей: у него единственный вызывающий — загрузчик шаблона sub_101C6800 (читает "rmg.txt", 0x101C6883). Зеркальный режим включается ИМЕННО в sub_101C6800 по полю Mirror шаблона:

.text:101C684B  cmp  dword ptr [ebx+10B8h], 2   ; Mirror == 2 ?
.text:101C6856  jnz  short loc_101C687E
.text:101C6858  mov  eax, edi                   ; да: число зон
.text:101C6865  sar  eax, 1                      ;      делится пополам

То же поле [..+10B8h]==2 проверяется в mirror-коде 0x101AE20D (внутри хук-сопрограммы sub_101AE0A0, callback из sub_101B1200): она подменяет расчёт размера карты/зоны в Create_Zones (пишет адрес возобновления 0x53C257 — регион min_map_size/min_zone_size), через __aulldiv/fsqrt получает измерение и при Mirror==2 УДВАИВАЕТ результат (0x101AE221 add eax,eax). То есть отражается ГЕОМЕТРИЯ карты (удвоение измерения), парно к делению числа зон пополам (sar eax,1) в загрузчике; это НЕ переиндексатор связей -1→двойник. Кроме того, в hota.dll есть строка (0x101A5840): "Mirror templates currently can be generated only with HW_Rulez pl..." — зеркальные шаблоны требуют плагин HW_Rulez, и именно HW_HOTA.dll виден в стеке этого вылета. Это связывает «Mirror==2» с наблюдаемыми логами.

Итог: ветка «Zone2=-1 → NULL-партнёр» (0x101C644F) общая, но безвредна без зеркала: без Mirror==2 отражённых зон нет и «фиктивный» партнёр не встречается при расстановке центра. При Mirror==2 число зон делится пополам и достраивается отражением — и NULL-партнёр доходит до sub_53BE60. Что остаётся не символизированным — точный код достройки отражённой половины (как -1-связь переиндексируется в несуществующую зону); его подтверждение — точкой останова (раздел 11).

7.3.1. Детектор зеркальности в HW_HOTA.dll (по ИМЕНИ шаблона). Флаг Mirror в hota.dll выставляется не произвольно: HW_HOTA.dll имеет функцию fTNameIsMirror (sub_10019250), явно экспонированную через patcher как "HW.fTNameIsMirror" (0x1001B432). Она сравнивает префикс имени шаблона с несколькими вариантами "mt" (aMt = "mt", и байтовые шаблоны рядом) и возвращает 1 при совпадении; вызывающие — sub_100184B0 и sub_100190C0 (регистрируются в sub_1001B420). Это согласуется с правилом именования mt_* — но это именно код игры, ветвящийся по префиксу (явное подтверждение, исключение §14), а не догадка по имени папки. Сам флаг [zone+10B8h] пишется косвенно (при копировании структуры шаблона), прямого mov-setter’а по статике нет; читателей флага два: 0x101C684B (загрузчик) и 0x101AE20D (mirror).

7.3.2. У потребителя НЕТ проверки типа/фиктивности связи. Цикл связей в sub_53BE60 — это while(1) с 0x53BF41: берёт каждую запись по счётчику count=(end−base)/28 и сразу разыменовывает партнёра (0x53BF92), НЕ смотря на поле is_created_byteA (off 0xA) или фиктивность. Понятие "fictive" в h3hota.exe вообще отсутствует (греп пуст) — вся обработка -1/фиктивных связей живёт только в hota.dll. Поэтому band-aid-фикс (пропускать связь с партнёром==NULL) лёг бы ровно между 0x53BF76 и 0x53BF92; правильный же фикс — у поставщика (hota.dll).

7.4. Тест «детерминированность vs частота» (§19). Вылет интермиттентный (часть розыгрышей RMG), а не «у всех и сразу». Это согласуется: NULL-связь появляется лишь при определённой раскладке зон/тегов отражения и при конкретном зерне ГСЧ, когда путь расстановки центра доходит до дефектной связи. Механизм — не порча памяти (адрес строго 0x0, регистры чистые), а детерминированный по входным данным NULL, маскируемый случайностью конфигурации. Симптом 2 (div esi=0) — другое проявление вырожденной зоны (пустой список координат ПОСЛЕ fallback-прохода по всем зонам, не только со связями); связь с NULL-связью вероятна, но статикой не доказана (см. раздел 4).

7.5. Почему именно зеркальные шаблоны. Текстовый разборщик фильтрует NULL-связи; единственный путь, где связь с NULL-партнёром реально попадает в массив — размножение зеркала. Поэтому пользовательское наблюдение «вылет на зеркальном шаблоне» прямо указывает на код-поставщик в hota.dll.

Условия, спровоцировавшие сбой¶

  • Запущена ГЕНЕРАЦИЯ случайной карты по САМОДЕЛЬНОМУ ЗЕРКАЛЬНОМУ шаблону (Mirror; обычно на основе mt_*/h3dm*). По комментариям отправителей: стоковый mt_Jebus НЕ падает, падает именно самодельный шаблон на его основе.
  • В шаблоне есть связь с Zone2 = -1 (зеркальный/односторонний двойник), а зоны-двойника в наборе НЕТ (несбалансированная конфигурация) → резолвер возвращает NULL, и загрузчик HotA сохраняет связь с NULL-партнёром (0x101C644F).
  • Расстановка центра зоны (PlaceZoneCenter) доходит до этой связи и разыменовывает NULL (0x53BF92) — либо у зоны не остаётся валидных кандидатных координат → деление на ноль (0x53C059).
  • Зависит от зерна ГСЧ и геометрии — отсюда непостоянство.
  • НО: сама ветка Zone2=-1→NULL НЕ гейтится флагом Mirror (в теле sub_101C6300 проверок [+10B8h] нет). Зеркало — это контекст, в котором такая связь штатна и достижима при расстановке.

Достоверность и ограничения¶

Подтверждено: • Точка сбоя 0x53BF92 (mov ecx,[edx]) — арифметика по всем 57 AV дампам: EDX=0, чтение 0x0, EAX=число зон. ✓ • Второй симптом 0x53C059 (div esi=0) — по 2 дампам: ESI=0. ✓ • Цепочка ARandMapGen_Main → Create_Zones → PlaceZoneCenter — по XREF листинга h3hota.exe и совпадению адресов возвратов. ✓ • ЗЕРКАЛЬНАЯ АТРИБУЦИЯ и «самодельный шаблон» — по комментариям в самих дампах (не только из внешнего репорта): 0x02746163 — «mt_Jebus не падает, падает свой шаблон на его основе»; 0x00D78F74 — «попытка сделать зеркальный шаблон... через 1 и 101 зоны». Один дамп явно содержит строку mt_Jebus. ✓ • Текстовый разборщик шаблона null-guard'ит связи (0x538751.. 0x53875B) — поставщик №1 исключён. ✓ • Зон-мирроринг в h3hota.exe отсутствует → поставщик в hota.dll. ✓ • Поставщик НАЙДЕН в hota.dll: sub_101C6300 (ОБЩИЙ загрузчик связей зон, единственный вызывающий — sub_101C6800) для токена Zone2=-1 строит связь с NULL-партнёром (0x101C6435..0x101C644F), тогда как текстовый разборщик SoD её отбрасывает. Ветка идентична в 1.6.1/1.7.3/1.8.0. ✓ • Зеркальный режим включается флагом Mirror==2 в загрузчике sub_101C6800 (0x101C684B), а зеркальность шаблона определяет HW_HOTA.dll по имени (fTNameIsMirror, sub_10019250, префикс "mt"). ✓ • Потребитель sub_53BE60 не проверяет фиктивность/is_created связи (while(1) с 0x53BF41) — разыменовывает безусловно. ✓

Ограничения: • Инструкция, где NULL сохраняется в запись связи, ЛОКАЛИЗОВАНА: hota.dll sub_101C6300, 0x101C644F (mov [..],ebp при ebp=NULL), ветка токена Zone2=-1 (0x101C6435..0x101C643C). Вопрос «общий загрузчик или зеркало» РАЗРЕШЁН: это общий парсер, а зеркало включается флагом Mirror==2 (см. 7.3, 7.3.1). Зеркало ГЕОМЕТРИИ тоже локализовано: хук-сопрограмма sub_101AE0A0 при Mirror==2 удваивает измерение карты (0x101AE20D cmp [esi+10B8h],2 → 0x101AE221 add eax,eax) и возобновляет Create_Zones на 0x53C257. Что ПО СТАТИКЕ НЕ ЗАКРЫТО: точный код переиндексации СВЯЗЕЙ отражённой половины (как -1-связь сопоставляется с зоной-двойником) и прямой mov-setter флага [zone+10B8h]=2 (пишется косвенно при копировании структуры). Схема «id+100» — формулировка отправителя, кодом не подтверждена (константы 100/64h в mirror-коде нет; см. раздел 11, negative result). СУЖЕНИЕ ПОИСКА (раскрутка sub_101AE0A0): этот колбэк — один из шагов сопрограммной таблицы RMG, регистрируемой в sub_101B1200 (+236 = 0x101B1436; мастер-регистратор из sub_100E49D0+163B). Проверка ВСЕХ resume-адресов этой таблицы (mov [..+20h],5xxxxxh): ни один шаг НЕ возобновляет регион парсера связей 0x538xxx — т.е. переиндексатор связей В ЭТОЙ СОПРОГРАММНОЙ ТАБЛИЦЕ ОТСУТСТВУЕТ (negative result); сопрограммы касаются геометрии/ расстановки (0x53Cxxx), а связи строятся отдельным путём sub_101C6800 → sub_101C6300. Искать переиндексацию среди co-routine шагов бесполезно. Для развязки нужен дамп с точкой останова (раздел 11). • По §17: листинг h3hota.exe (1.8.0) совпадает с дампами 1.8.0; для дампов 1.6.1 — 1.7.3 функция sub_53BE60 унаследована от SoD и стабильна по адресу 0x53BE60 (тот же Imagebase 0x400000), что подтверждается одинаковой цепочкой во всех версиях. Рассинхрон версии — в ПЕРИФЕРИЙНЫХ модулях (HW_HOTA/HD_HOTA), он не блокирует диагноз падающего модуля.

Почему полезность СРЕДНЯЯ (§20): падающая функция (sub_53BE60) ИСПРАВНА — она доверяет входу так же, как оригинал SoD. Точечный фикс в точке сбоя (добавить NULL-проверку перед 0x53BF92) — это band-aid; первопричина у поставщика выше по стеку (код зеркал hota.dll создаёт связь с несуществующей зоной). Лог полезен и указывает на подсистему, но фикс не в точке сбоя. Место поставщика теперь локализовано (hota.dll sub_101C6300, ветка Zone2=-1, 0x101C644F); до ВЫСОКОЙ полезность не поднята, т. к. по статике не доказано, является ли -1 штатным «фиктивным» признаком (тогда фикс — отброс NULL-связи у потребителя/загрузчика) или ошибкой драйвера зеркал.

Признаки поиска в старых версиях HotA / в SoD¶

Были до 1.8.0✅ — есть дампы 1.6.1, 1.7.0, 1.7.1, 1.7.2, 1.7.3 с той же функцией, цепочкой и инвариантом (EDX=0). Дефект не нов для 1.8.0.
Были в SoD❌ — НЕ подтверждено. Сама функция sub_53BE60 унаследована от SoD и доверяет входу, НО источник NULL-связи — зеркальное размножение зон, которого в SoD нет (это возможность HotA). В чистом SoD без HotA-зеркал этот NULL не производится. (❌ = «не подтверждено по имеющимся данным», а не «точно отсутствует».)

Воспроизведение вылета¶

Условие активации: ГЕНЕРАЦИЯ случайной карты по зеркальному шаблону. Обычная игра на готовой карте путь не задействует.

Базовый сценарий:

  • Взять зеркальный шаблон (mt_Jebus, mt_TeamJebus, mt_Diamond, mt_Andromeda, mt_Antares, mt_Firewalk, h3dm1 и т. п.).
  • Массово генерировать карты, перебирая зёрна ГСЧ; приоритет — конфигурации с числом игроков/позиций на границах диапазонов связей (поля min/max human/total positions у связей), при которых часть связей отбраковывается асимметрично между оригиналом и зеркалом.
  • Ожидаемый вылет: AV чтения 0x0 на 0x53BF92 (EDX=0, EAX=число зон) либо INT_DIVIDE_BY_ZERO на 0x53C059 (ESI=0).

Создание шаблона, провоцирующего вылет (НА ОСНОВЕ СТРУКТУРЫ РЕАЛЬНЫХ ШАБЛОНОВ из HotA_RMGTemplates):

  • База — РАБОЧИЙ зеркальный mt_Jebus.h3t: у него в зоне Id=1 флаг Mirror=[x] (колонка 12), а связи (колонки 128/129 Zone1/Zone2): 1→2, затем 2→-1, 2→-1, 2→-1, 2→-1. Токен Zone2=-1 здесь ШТАТНЫЙ и САМ ПО СЕБЕ НЕ РОНЯЕТ (стоковый mt_Jebus работает) — HotA корректно резолвит двойника при сбалансированной раскладке.
  • Чтобы уронить: взять такой шаблон и РАЗБАЛАНСИРОВАТЬ зеркало (как у отправителей: «свой шаблон на основе mt_Jebus», «через 1 и 101 зоны»): сделать так, чтобы -1-связь указывала на двойника, который в отражённом наборе не материализуется (лишняя/непарная зеркальная связь, рассинхрон номеров зон). Тогда резолвер вернёт NULL, и загрузчик сохранит битую связь. ПРИМЕЧАНИЕ про «1 и 101»: схема «двойник = id+100» — это формулировка ОТПРАВИТЕЛЯ из комментария к дампу, А НЕ подтверждённая кодом арифметика: константы 100 (64h) в mirror-функциях hota.dll (sub_101AE0A0, mirror-ветка sub_101C6800) ПО СТАТИКЕ НЕТ (negative result). Точный шаг нумерации двойника остаётся неподтверждённым (см. раздел 9, ограничения).
  • Проверять на крупных картах с максимумом зон — выше шанс дойти до дефектной связи при расстановке центров.

АНТИ-РЕПРО (важно, чтобы не повторять ошибку): файлы CrashEdgeStress*.h3t в HotA_RMGTemplates — НЕ зеркальные (во всех зонах Mirror пуст, связи ссылаются на конкретные id 2..48, без Zone2=-1). Они НЕ воспроизводят этот механизм (тестируют другое — edge/stress по числу зон). Обязательный признак корректного репро — Mirror=[x] И связь Zone2=-1. (Файл 532D1F.h3t — другой адрес краша, к этому вылету не относится.)

Дифференциальный тест (подтверждение причины):

  • Точка останова на 0x53BF76 (mov edx,[ecx+edx]) — снять edx. Причина подтверждена, если edx = 0 (запись связи содержит NULL-указатель партнёра), при этом счётчик связей count=(end−base)/28 > 0 и индекс в пределах массива (т.е. перебор корректен, NULL — в содержимом записи).
  • Контроль: если у всех связей партнёр != NULL — вылета нет.

Подтверждение поставщика (закрывает открытый вопрос 7.3):

  • Точка останова в hota.dll на рантайм-адресе 0x101C644F (mov [esp+..],ebp) — попадание сюда с ebp=0 доказывает, что связь с NULL-партнёром сохраняет именно загрузчик HotA. (Рантайм-адрес = load_base hota.dll + (0x101C644F − 0x10000000).)
  • Проверка «общий загрузчик или зеркало»: сравнить, бьёт ли эта точка на НЕзеркальном шаблоне с Zone2=-1 (напр. 1deaL, где есть фиктивные связи): если ebp≠0 всегда — -1 корректно резолвится вне зеркала, и NULL рождает именно зеркальное отражение.

Вывод¶

Виновник — hota.dll (подсистема зеркальных шаблонов генератора случайных карт): при размножении отражённой половины карты создаётся связь зоны, целевой указатель которой = NULL (зона по своему номеру не нашлась в отражённом наборе). Унаследованная от SoD функция расстановки центра зоны (sub_53BE60 в h3hota.exe) штатно перебирает связи и разыменовывает указатель партнёра без проверки — на 0x53BF92 (mov ecx,[edx]) происходит чтение по нулевому адресу. По всем 57 дампам этого исключения EDX=0 и адрес чтения 0x0, EAX = число зон шаблона — арифметика совпадает. Ещё 2 дампа дают деление на ноль на 0x53C059 (пустой список кандидатных координат вырожденной зоны) — иной симптом того же зеркального сценария (связь с NULL-связью вероятна, но статикой не доказана — см. раздел 4).

Сама SoD-функция исправна: она доверяет входу так же, как в оригинале, а контракт «у связи всегда валидный партнёр» соблюдает текстовый разборщик шаблона (sub_5384E0), который NULL-связи отбрасывает (0x538751..0x53875B). Нарушает контракт именно загрузчик связей HotA (найден: hota.dll sub_101C6300, ветка токена Zone2=-1, 0x101C644F — сохраняет NULL-партнёра) — в h3hota.exe зон-мирроринга нет. Поэтому вылет воспроизводится ТОЛЬКО на зеркальных шаблонах, а исправление должно быть на стороне поставщика (hota.dll): не создавать связь с несуществующей зоной либо отбрасывать связь с NULL-партнёром, как это делает текстовый разборщик.

Дефект не нов для 1.8.0 — подтверждён дампами 1.6.1 — 1.8.0. Непостоянство вылета объясняется не порчей памяти, а зависимостью от зерна ГСЧ и геометрии зеркальной раскладки.
