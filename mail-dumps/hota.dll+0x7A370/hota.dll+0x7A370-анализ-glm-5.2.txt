========================================================================
АНАЛИЗ ВЫЛЕТА hota.dll+0x7A370 (спрайт книги заклинаний)
========================================================================

Модель: GLM 5.2
Дата: 15.07.2026
Заключение: В hota.dll при инициализации книги заклинаний в цикле из
32 шагов накапливается до 32 пиксельных растров; движковый nothrow-
аллокатор отказывает выделить один из них на итерации 3..29, фабрика
растрового кэша возвращает NULL, а вызвавший код использует этот
возврат как объект без проверки и падает на разыменовании нулевого
this (чтение поля +0x30).

КРАТКОЕ РЕЗЮМЕ
--------------------

Игра падает в момент подготовки интерфейсного спрайта книги закланий
(SpelScBk.pcx) в самом начале партии (день 0, тестовая карта). Точка
сбоя одна и та же во всех 26 уникальных логах: чтение по адресу
0x00000030 — это разыменование поля со смещением +0x30 у НУЛЕВОГО
объекта (this==NULL). Нулевой this появляется потому, что вспомогательная
фабрика растрового кэша честно возвращает NULL при невозможности
выделить пиксельный буфер, а вызвавший её код забирает возврат как
готовый объект и без всякой проверки передаёт в функцию заполнения
растра. Дефект локализован: в той же функции проверка возврата другой
аллокации (operator new) присутствует, а после фабрики — отсутствует.

Отказ аллокатора вызван накоплением: вызывающий код в цикле создаёт
до 32 одинаковых растров спрайта и складирует их в глобальный вектор,
не освобождая; на одной из итераций (в дампах — с 3-й по 29-ю)
движковый nothrow-аллокатор отказывает и возвращает NULL. Тот факт,
что первые итерации успешны, доказывает: это давление кучи от
накопления, а не «битый» размер растра (тогда отказ был бы на первой
итерации). Сам пропуск проверки — детерминированная ошибка кода,
воспроизводимая на всём диапазоне версий HD-мода (5.6 R15 — 5.7 R30)
при HotA 1.8.0.

Класс сбоя: разыменование нулевого указателя. Исключение — нарушение
права доступа при чтении (EXCEPTION_ACCESS_VIOLATION, чтение).

2. Корреляция дампов
--------------------

26 уникальных вылетов (точных дублей нет). Группировка — по одной
точке сбоя и набору регистровых инвариантов; caller'ы в V2-стеке
неименованные, поэтому группировка ведётся по падающей функции.

Регистровые инварианты (абсолютно стабильны во всех 26 логах):
   - EAX = 0x00000000
   - ECX = 0x00000000
   - операция: чтение
   - адрес сбоя (faulting) = 0x00000030

Что меняется от дампа к дампу (и что это значит):
   - EDX, EBX — это адреса/размеры растровых объектов конкретной
     сессии (значения вида 0x05DC900C, 0x09D4200C и т. п. — разные
     адреса загруженных спрайтов в куче у разных игроков);
   - EBP — малое число 0x03..0x1D (вероятно, индекс/счётчик в цикле
     подготовки спрайта);
   - ESI, EDI, ESP — стековые/указательные значения конкретного кадра;
   - load_base модуля hota.dll плавает (0x031C0000, 0x06450000,
     0x50F60000 и т. д.) — ASLR-рандомизация базы DLL, к диагнозу
     отношения не имеет.

Состояние игры одинаковое во всех 26 логах: одиночная игра (Network
Game = 0), ход игрока (Me = Active Player = 0), день 0/00 (начало
партии), карта test.h3m. Пять дампов (№18-20, 22) сопровождаются
минидампами Windows (MDMP), которые подтверждают тот же EIP/ECX.

Поскольку во ВСЕХ логах адрес сбоя, операция и EAX/ECX идентичны,
а различаются только плавающие значения кучи и ASLR-база — это одна
сигнатура сбоя, и она носит детерминированный характер в точке кода
(подробнее — в разделах 4 и 7).

3. Сопоставление адресов
------------------------

Формула: IDA_адрес = runtime_адрес - адрес_загрузки + базовый_адрес_образа

Для дампа №18 (есть минидамп):
   runtime EIP (из CONTEXT минидампа) = 0x0323A370
   адрес загрузки hota.dll            = 0x031C0000
   базовый адрес образа (листинг)      = 0x10000000
   IDA-адрес = 0x0323A370 - 0x031C0000 + 0x10000000 = 0x1007A370

Смещение в модуле: 0x0323A370 - 0x031C0000 = +0x7A370, что совпадает
с полем лога "Address: [ hota.dll+0x7A370 ]". Проверка пройдена.

Адрес сбоя 0x00000030 — это НЕ адрес инструкции, а адрес ЧТЕНИЯ
(Information: read of address: 0x00000030). Инструкция, выполнявшая
чтение, находится по IDA-адресу 0x1007A370 (см. раздел 4).

4. Точка сбоя
------------------------

Падающая функция — sub_1007A370 (заполнение 32-битного растра
константой). Это функция без кадра SEH, соглашение stdcall (retn 4).
Её первая инструкция и есть точка сбоя:

    .text:1007A370 sub_1007A370 proc near          ; CODE XREF: sub_10103400+73D
    .text:1007A370                 mov     edx, [ecx+30h]   ; <<< СБОЙ: чтение [0+30h]

Проверка арифметики: ECX = 0x00000000 (поле лога/CONTEXT), смещение
поля = +0x30. Тогда адрес чтения = ECX + 0x30 = 0x00000000 + 0x30 =
0x00000030. Это в точности равно faulting address из лога. Проверка
пройдена: падение происходит именно при первой попытке прочитать поле
объекта, на который указывает ECX. Поскольку ECX == 0, объект нулевой
(this == NULL).

Полная структура функции sub_1007A370 (дизассемблер дословно):

    .text:1007A370  mov  edx, [ecx+30h]   ; данные растра (массив строк)
    .text:1007A373  push ebx
    .text:1007A374  mov  ebx, [ecx+24h]   ; ширина (число dword в строке)
    .text:1007A377  push ebp
    .text:1007A378  mov  ebp, [ecx+2Ch]   ; шаг строки (stride)
    .text:1007A37B  push esi
    .text:1007A37C  mov  esi, [ecx+28h]   ; высота (число строк)
    .text:1007A37F  lea  eax, [ebx*4]
    .text:1007A386  sub  ebp, eax
    .text:1007A388  test esi, esi
    .text:1007A38A  jle  short loc_1007A3AD
    .text:1007A38C  push edi
    .text:1007A390 loc_1007A390:
    .text:1007A390  test ebx, ebx
    .text:1007A392  jle  short loc_1007A3A7
    .text:1007A394  mov  eax, [esp+10h+arg_0]
    .text:1007A398  mov  edi, edx
    .text:1007A39A  mov  ecx, ebx
    .text:1007A39C  rep stosd              ; заполнение строки цветом
    .text:1007A39E  lea  eax, [ebx*4]
    .text:1007A3A5  add  edx, eax
    .text:1007A3A7 loc_1007A3A7:
    .text:1007A3A7  add  edx, ebp          ; переход на следующую строку
    .text:1007A3A9  dec  esi
    .text:1007A3AA  jnz  short loc_1007A390
    .text:1007A3AC  pop  edi
    .text:1007A3AD loc_1007A3AD:
    .text:1007A3AD  pop  esi
    .text:1007A3AE  pop  ebp
    .text:1007A3AF  pop  ebx
    .text:1007A3B0  retn 4

Функция читает из дескриптора растра поля: +0x30 (буфер пикселей),
+0x24 (ширина), +0x28 (высота), +0x2C (шаг строки), затем циклом
rep stosd заполняет весь растр константой (arg_0 — цвет заливки).
Сбой возникает на самом первом чтении +0x30, потому что ECX == 0.
Объект, переданный как this, полностью нулевой.

5. Регистровые инварианты (размеры / счётчики / индексы)
--------------------------------------------------------

Инвариант, общий для ВСЕХ 26 уникальных дампов:
   EAX = 0x00000000
   ECX = 0x00000000   <-- регистр this (thiscall), это и есть нулевой this

Регистр ECX в вызывающей функции sub_10103400 перед падающим вызовом
загружается прямо из возврата фабрики sub_10079970 (строка листинга
0x10103B37: mov ecx, eax). То есть ECX == 0 означает: фабрика вернула
NULL. Это единственный корневой регистр, объясняющий адрес сбоя:
   base(this) + смещение_поля = 0 + 0x30 = 0x00000030 == faulting address

Остальные регистры варьируются по дампам и не образуют инварианта:
   EBP = 3..0x1D (3..29) — это СЧЁТЧИК ИТЕРАЦИЙ цикла накопления
        растров в вызывающей функции sub_10103400 (цикл из 32 шагов,
        ebp 0..0x1F; подробнее в разделе 7). Разные значения EBP в
        разных дампов означают, на какой итерации цикла отказала
        аллокация: первые растры (до ebp=3) выделились успешно, отказ
        наступил позже. Это исключает «битый размер одного растра»
        (тогда отказ был бы на ebp=0) и подтверждает накопительный
        характер отказа кучи;
   EDX, EBX, ESI, EDI — адреса объектов растрового кэша и кучи, разные
                        в каждом дампе (ASLR + состояние аллокатора);
   ESP — адреса стека в диапазоне 0x001A/0x0019Fxxx (ASLR).

Проверки размера буфера здесь НЕ требуется: сбой — это разыменование
поля у нулевого указателя, а не выход за границу или массовая запись
по NULL. Поэтому формулы размера буфера (через ECX/EDX) к данному
адресу сбоя неприменимы — ECX == 0 это не размер, а this.

6. Восстановленная цепочка вызовов
----------------------------------

По стеку V2 (дампы 1, 18 и др.) восстановлена следующая цепочка
(адреса IDA листинга, базы выровнены по 0x10000000):

   sub_1007A370   <-- ТОЧКА СБОЯ, заполнение растра цветом
      this (ECX) = NULL, пришёл из возврата фабрики
   sub_10103400   <-- инициализация спрайта "SpelScBk.pcx" (книга
                      заклинаний); в .text:10103B30 вызывает
                      фабрику sub_10079970, затем .text:10103B37
                      заносит её возврат в ECX и .text:10103B3D
                      вызывает sub_1007A370 БЕЗ проверки на NULL
   sub_101055C0   <-- имя в логе "MainProc" (кадр 0x...6585)
   sub_100E49D0   <-- имя в логе "MainProc" (кадр 0x...3600B)
   (trampoline)   <-- кадр 0x011DDB48 вне модулей (стаб хука)

Имя символа "MainProc" относится к верхним кадрам; падающий модуль
во всех случаях — hota.dll. Кадр-трамплин вне модулей (raw-адрес
вида 0x011DDB48 / 0x0BCD874) — это runtime-стаб перехвата patcher_x86,
в листинге не мапится, через него «перешагиваем» к кадру внутри hota.dll.

7. Первопричина с доказательством
---------------------------------

Источник нулевого this — фабрика растрового кэша sub_10079970.
Её тело (дизассемблер, .text:10079970):

    .text:10079993  push    35h ; '5'                 ; размер объекта 0x35
    .text:10079995  mov     eax, 617492h               ; движковый аллокатор SoD
    .text:1007999A  call    eax                          ; выделить сам объект
    .text:1007999C  add     esp, 4
    .text:1007999F  mov     [esp+18h+var_14], eax
    .text:100799A3  test    eax, eax
    .text:100799A5  jz      short loc_100799DF          ; отказ 1: объект не выделен
    .text:100799A7  mov     [esp+18h+var_10], eax
    .text:100799AB  push    [esp+18h+arg_8]
    .text:100799AF  mov     ecx, eax                    ; this для конструктора
    ...
    .text:100799C1  call    sub_100785E0                ; конструктор: ставит
                                                        ;  буфер в [this+30h]
    ...
    .text:100799CE  cmp     dword ptr [eax+30h], 0      ; буфер НЕ выделен?
    .text:100799D2  jnz    short loc_100799E1           ; если есть -> вернуть объект
    .text:100799D4  push    eax                          ; иначе: удалить объект
    .text:100799D5  mov     eax, 60B0F0h                ;   (деструктор/оператор delete)
    .text:100799DA  call    eax
    .text:100799DC  add     esp, 4
    .text:100799DF loc_100799DF:                         ; отказ 2: нет пиксельного буфера
    .text:100799DF  xor     eax, eax                     ; ВЕРНУТЬ NULL
    .text:100799E1 loc_100799E1:
    .text:100799E1  ...                                   ; эпилог, retn 0Ch

Фабрика корректна: она возвращает NULL в двух случаях — (1) не удалось
выделить сам объект-дескриптор, (2) конструктор не смог выделить
пиксельный буфер (поле +0x30). Конструктор sub_100785E0 кладёт в
[this+0x30] результат того же аллокатора SoD (617492h) без проверки:

    .text:10078654  push    edx                          ; размер = stride*height
    .text:1007865E  call    eax                           ; call 617492h (nothrow)
    .text:10078667  mov     [esi+30h], eax               ; буфер <- результат (может быть 0)

Аллокатор 617492h (это функция в h3hota.exe, вызывается косвенно через
mov eax,617492h; call eax) — NOTHROW: при нехватке памяти он возвращает
NULL, а не бросает исключение. Поэтому поле +0x30 остаётся нулевым,
фабрика видит это (cmp [eax+30h],0) и честно возвращает NULL.

ПЕРВИЧНАЯ ПРИЧИНА — в вызывающем коде sub_10103400. В точке вызова
фабрики НЕТ проверки её возврата:

    .text:10103B30  call    sub_10079970                 ; фабрика (вернёт NULL)
    .text:10103B35  push    0                            ; arg: цвет заливки
    .text:10103B37  mov     ecx, eax                     ; this = возврат (= 0)
    .text:10103B39  mov     [esp+0E8h+var_A4], eax       ; (сохранён, но не проверен)
    .text:10103B3D  call    sub_1007A370                 ; СБОЙ: [ecx+30h], ECX=0

Между возвратом фабрики (0x10103B35) и использованием результата как
this (0x10103B3D) нет ни test eax,eax, ни jz. Возврат NULL
«проглатывается» как валидный объект.

Доказательство систематичности дефекта (сравнение с родственными
функциями). В начале той же функции sub_10103400 проверка возврата
аллокации ПРИСУТСТВУЕТ:

    .text:10103414  push    0Ch ; Size
    .text:10103416  call    ??2@YAPAXI@Z                  ; operator new(uint)
    .text:1010341B  add     esp, 4
    .text:1010341E  test    eax, eax                      ; <<< ПРОВЕРКА ЕСТЬ
    .text:10103420  jz      short loc_10103438           ;     и переход при NULL

То есть автор кода знает о необходимости проверять аллокации и делает
это для operator new, но НЕ делает для фабрики sub_10079970, которая
по контракту тоже может вернуть NULL. Это и есть локализованный
дефект: пропущенная проверка возврата фабрики в точке 0x10103B3D.
Фабрика sub_10079970 вызывается из sub_10103400 в семи местах
(0x101034C8, 0x101034E0, 0x101034F0, 0x10103629, 0x101036DD,
0x101037D7, 0x10103B30), и ни в одном из них её возврат не
проверяется — значит, дефект не локален в один вызов, а является
общим шаблоном в этой функции.

Сопоставление поставщика и потребителя данных (механизм). Поставщик
поля +0x30 — конструктор sub_100785E0 (через nothrow-аллокатор SoD);
потребитель — падающая sub_1007A370. Границы потребителя (заполнение
всего растра: ширина из +0x24, высота из +0x28, шаг из +0x2C, буфер
+0x30) согласованы с границами поставщика (конструктор выделяет ровно
stride*height байт для +0x30 и записывает те же размеры в +0x24/+0x28/
+0x2C). Рассинхрона границ НЕТ — это не off-by-one и не перехлёст.
Следовательно, корень именно в непроверенном возврате, а не в ошибке
геометрии буфера. Состояние кучи (отказ nothrow-аллокатора) — лишь
условие активации; само по себе оно не привело бы к падению, если бы
возврат фабрики проверялся.

ПРИЧИНА ОТКАЗА АЛЛОКАТОРА: накопление растров в цикле. Вызов фабрики
в точке 0x10103B30 находится ВНУТРИ цикла из 32 итераций:

    .text:10103B25 loc_10103B25:               ; НАЧАЛО цикла
    .text:10103B30  call   sub_10079970         ; создать растр (итерация N)
    ...
    .text:10103CFC  cmp    ebp, 20h             ; ebp = счётчик 0..0x1F (32 шага)
    .text:10103CFC  jl     loc_10103B25         ; ПОВТОР

На каждой итерации создаётся НОВЫЙ растр и помещается в глобальный
вектор dword_10634F10 (рост capacity через sub_101FC820). Источник
размеров — поля [ebx+24h]/[ebx+28h] объекта-спрайта, причём регистр
ebx ВНУТРИ цикла не меняется, то есть ВСЕ 32 растра одного и того же
размера (~4*width*height байт каждый).

Это разрешает ключевую развилку "битый размер" vs "отказ кучи".
Регистр EBP в дампах принимает значения 0x03..0x1D (3..29), и EBP
в момент сбоя — это именно счётчик итераций (он не переприсваивается
между сохранением в 0x10103B1D и вызовом фабрики в 0x10103B30). Если
бы причина была в невалидном ("битом") размере растра, фабрика
отказала бы на ПЕРВОЙ итерации (ebp=0). Но сбой ловится на итерациях
3..29 — первые итерации УСПЕШНО выделяют растры, и отказ происходит
по мере накопления. Это прямое доказательство heap-pressure внутри
цикла, а НЕ битого поля размера.

Замечание о масштабе. Точный размер одного растра (поля width/height
объекта-спрайта) лежит в куче и в сокращённых минидамппах недоступен,
поэтому суммарный объём 32 растров оценочно — от единиц до десятков
МБ. Но для механизма это не существенно: решающее доказательство
heap-pressure — не величина, а сам факт "та же аллокация сначала
успешна, потом отказывает" (сбой на итерациях 1..29, никогда на 0 при
постоянном размере). Это сигнатура деградации кучи (заполнение/
фрагментация) ВНУТРИ цикла накопления, а не одного невалидного
запроса. Вектор sub_101FC820 при росте освобождает лишь свой буфер
указателей (free старого массива), а НЕ растровые объекты — их адреса
копируются в новый буфер, поэтому накопление растров реально.

ВТОРОЕ, более сильное обоснование: монотонный рост кэша через сессию.
Глобальный вектор dword_10634F10 (растровый кэш книги заклинаний)
просмотрен по всем 8 обращениям в коде (0x100FD6D0, 0x100FDF83,
0x100FEAFA, 0x1010343C, 0x10103BF2, 0x10108A37, 0x1010A64A). Все они
— ЧТЕНИЕ вектора для индексации/отрисовки; ни в одном нет цикла
освобождения элементов (нет push элемента; call delete/60B0F0h).
Единственная ЗАПИСЬ в вектор — создание (0x1010343C) и push_back в
цикле (0x10103BF2). Значит кэш растров МОНОТОННО растёт: повторные
построения интерфейса книги заклинаний (каждое добавляет до 32 растров)
НИКОГДА не освобождают ранее добавленные. Таким образом heap-pressure
складывается не только внутри одного цикла, но и ЧЕРЕЗ ВСЮ сессию —
поверх буферов HD-мода, локализации и прочих подсистем. Это объясняет,
почему вылет стабильно ловится в начале партии (Day=0): к моменту
первого построения книги уже накоплены большие буферы инициализации,
а кэш растров добавляет к ним непрерывные блоки.

8. Условия, спровоцировавшие сбой
---------------------------------

Фактическое состояние из логов (одинаково во всех 26 дампах):

  - HotA version: 1.8.0; HOMM3 HD version: от 5.6 R15 до 5.7 R30.
  - Процесс 32-битный (Heroes3 HD.exe, база 0x400000), лимит адресного
    пространства 2 ГБ.
  - Network Game = 0 (одиночная игра); Me = Active Player = 0 (ход
    игрока); Day = 0/00 (первый день партии); карта test.h3m.
  - Memory.Mod = 0 (HotA-оптимизация памяти отключена).

Сигнальные ini-настройки, ускоряющие давление 32-битной кучи
(что и есть условие активации — отказ nothrow-аллокатора SoD):
  - Graphics.Threads = 4; Graphics.AdvMap.Threads = -1 (по числу ядер)
    — дополнительные рабочие буферы потоков отрисовки;
  - OpenGL.Upscaling = 2 — тайл-буферы масштабирования;
  - Misc.BattleSaver = 1 и Misc.TournamentSaver = 1 — фоновые автосейвы
    занимают непрерывные блоки памяти;
  - Memory.Mod = 0 — отключённая оптимизация памяти HotA.

Механизм: при подготовке интерфейса книги заклинаний движок в цикле
из 32 итераций (0x10103B25..0x10103CFC, cmp ebp,20h) выделяет пиксельные
буферы спрайта SpelScBk.pcx и накапливает их в глобальном векторе
(dword_10634F10, рост capacity через sub_101FC820). Освобождения внутри
цикла нет — только накопление. На перегруженной 32-битной куче
движковый nothrow-аллокатор (617492h) отказывает на одной из средних
итераций (EBP в дампах = 3..29) — возвращает NULL; фабрика sub_10079970
корректно транслирует это в свой NULL; вызывающий код игнорирует NULL и
падает. Это согласуется с тем, что вылет ловится в начале партии, на
тестовой карте, в одиночной игре — в момент первичной инициализации
интерфейсных спрайтов, когда до 32 одинаковых растров накапливаются
в векторе, не освобождаясь внутри цикла, поверх уже загруженных
HD-мода и локализации. Точный размер одного растра (поля
[ebx+24h]/[ebx+28h] лежат в куче, в сокращённых минидамппах
недоступны) статикой не определяется; но развилка «битый размер»
отпадает независимо от величины — отказ на итерации 1..29 при
постоянном размере доказывает деградацию кучи в ходе цикла.

9. Достоверность и ограничения
------------------------------

Верификация листинга: модуль сбоя hota.dll соответствует листингу
HotA.dll-180.txt (базовый адрес образа 0x10000000; версия модуля из
лога 1.8.0 совпадает с версией листинга). Уровень достоверности
маппинга — высокий.

Адресная проверка точки сбоя пройдена дважды: и по регистрам лога
(ECX=0 даёт [0+0x30]=0x30), и по CONTEXT минидампа (EIP=0x0323A370 =
+0x7A370 в hota.dll, ECX=0). Пять минидампов (дампы 18-20, 22)
подтверждают тот же EIP/ECX, что и текстовые логи.

Корреляция: 26 уникальных логов с идентичной сигнатурой (EAX=0,
ECX=0, faulting=0x30) — это не единичный вылет, а устойчивая
сигнатура. Стабильность регистров по серии усиливает диагноз
(нулевое this из возврата фабрики), но механизм — прочтённый в коде
пропуск проверки — доказан статикой, а не только повторяемостью.

Ограничения:
  - Стек V1 битый (кадры "called from before" с мусором); цепочка
    восстановлена по V2 + маппинг. Верхние кадры V2 неименованные
    ("?"/"?"), их функциональная роль ("MainProc", диспетчер
    интерфейса) установлена по соседним кодовым адресам, а не по
    символам листинга.
  - Минидампы сокращённые (формат MiniDump без кучи: код + стек),
    поэтому содержимое объектов кучи в момент сбоя недоступно. Однако
    для данного диагноза это не требуется: причина — в статически
    доказанном пропуске проверки, а не в содержимом объекта.
  - Точное условие отказа nothrow-аллокатора (фрагментация кучи vs
    исчерпание) без полного дампа кучи не определяется; это
    рантайм-условие активации, не влияющее на локализацию дефекта в
    коде.

Обоснование вердикта полезности — ВЫСОКАЯ. Лог ведёт к конкретному,
локализованному и легко исправимому дефекту кода: пропущенной
проверке возврата фабрики sub_10079970 в точке 0x10103B3D (и
аналогично в шести других местах вызова этой же фабрики). Дефект
сопровождается прямым доказательством: в той же функции проверка
возврата operator new присутствует (0x1010341E: test eax,eax; jz),
значит отсутствие проверки после фабрики — именно пропуск, а не
сознательное решение. Это не рантайм-коррупция без локализуемого
корня: корень найден и подтверждён чтением кода (поставщик данных
— конструктор sub_100785E0, его границы согласованы с потребителем
sub_1007A370; отказ даёт только условие активации).

10. Признаки поиска вылета в старых версиях HotA / в SoD
--------------------------------------------------------

Адрес hota.dll+0x7A370 НЕ стабилен между версиями HotA и не применим
к SoD напрямую (код лежит в HotA-специфичном модуле hota.dll, а не в
унаследованном h3hota.exe). Для опознания бага в любой версии
использовать следующие стабильные признаки.

Context-инварианты (значения в краш-логе, не зависящие от версии):
  - ECX = 0x00000000, EAX = 0x00000000 во всех дампах серии;
  - операция: чтение, faulting address = 0x00000030 (смещение поля
    пиксельного буфера в дескрипторе растра);
  - Day = 0, одиночная игра (Network Game = 0), Me = Active Player.

Текстовые сигнатуры по дизассемблеру (для листинга любой версии
hota.dll):
  - точка сбоя — функция, первая инструкция которой
    "mov edx, [ecx+30h]" с последующим циклом "rep stosd" по строкам
    (заполнение растра константой);
  - её вызывающий (MainProc-подобный инициализатор спрайта) содержит
    mov ecx, offset aSpelscbkPcx ("SpelScBk.pcx") и вызов фабрики
    (push 35h в 1.8.0 / push 38h в 1.5.3; mov eax,617492h; call eax)
    БЕЗ test eax,eax; jz перед вызовом падающей функции заполнения.

Присутствие в старых версиях HotA. Проверен листинг HotA.dll-153.txt
(версия 1.5.3): структурный аналог дефекта ПРИСУТСТВУЕТ.
  - Спрайт "SpelScBk.pcx" используется в 1.5.3 (строка-константа
    aSpelscbkPcx в .rdata, загружается в sub_10099CD0).
  - Аналог фабрики — sub_100643D0 (.text:100643D0): тот же движковый
    аллокатор (mov edi,617492h; call edi) и размещение пиксельного
    буфера в [this+30h]; при отказе аллокации фабрика возвращает NULL
    (loc_10064444: xor eax,eax). Размер объекта в 1.5.3 — 0x38 (push
    38h), в 1.8.0 сжался до 0x35 (push 35h), но структура идентична.
  Следовательно, класс дефекта (непроверенный возврат фабрики
  растрового кэша) существует и в версии до 1.8.0. Конкретный адрес
  точки сбоя в 1.5.3 другой (смещения функций иные), но та же
  сигнатура поиска его обнаружит.

Присутствие в SoD: не применимо. Виновный код (инициализация спрайта
SpelScBk.pcx, фабрика с nothrow-аллокатором и падающее заполнение
растра) целиком лежит в hota.dll — HotA-специфичном модуле, которого
в чистом SoD нет. В h3hota.exe (унаследованный SoD-движок) эта
подсистема интерфейса книги заклинаний отсутствует как отдельный
HotA-кэш, поэтому флаг «Были в SoD» = НЕТ по факту расположения кода.

Сиблинги без проверки: родственная функция sub_100797B0 (также
вызывается из sub_10103400 с результатом фабрики) тоже начинает с
разыменования [ecx+24h] без проверки this (0x100797B8: cmp eax,
[ecx+24h]). При том же нулевом возврате фабрики сбой проявился бы в
ней (по смещению +0x24). В листинге любой версии проверять все семь
вызовов фабрики sub_10079970 (или её аналога) внутри MainProc-
инициализатора спрайта на наличие test eax,eax; jz перед
дальнейшим использованием this.

11. Воспроизведение вылета
--------------------------

Условие активации пути (из дизассемблера). Чтобы прийти к точке
0x10103B3D, требуется: (а) первичная инициализация интерфейсного
спрайта книги заклинаний SpelScBk.pcx в начале партии (Day=0), при
которой (б) движковый nothrow-аллокатор SoD (617492h) отказывает в
выделении пиксельного буфера и возвращает NULL. Тогда фабрика
sub_10079970 вернёт NULL, а непроверенный вызов sub_1007A370
упадёт на [ecx+30h].

Базовый сценарий для тестера (вероятностный — зависит от состояния
кучи). Цель — создать давление 32-битной кучи и/или фрагментацию в
момент инициализации интерфейса книги заклинаний:

  1) Запустить одиночную игру на любой карте (серия ловилась на
     test.h3m, но имя карты к причине не относится) в HotA 1.8.0 под
     HD-модом 5.6 R15 — 5.7 R30; удерживать Memory.Mod = 0.
  2) Удерживать условия, ускоряющие давление кучи: Graphics.Threads = 4,
     Graphics.AdvMap.Threads = -1, OpenGL.Upscaling = 2, одновременно
     Misc.BattleSaver = 1 и Misc.TournamentSaver = 1 (фоновые автосейвы).
  3) В первый день партии открыть экран героя и экран книги
     заклинаний (триггер загрузки спрайта SpelScBk.pcx и построения
     его растрового кэша). Повторять (новая партия — открытие книги),
     пока аллокатор не откажет в нужный момент.

Что ловить (контрольные признаки): краш-лог с Module: hota.dll,
Address: [hota.dll+0x7A370], Code: EXCEPTION_ACCESS_VIOLATION,
Information: read of address: 0x00000030, Context ECX=0, EAX=0; Map
file произвольный, Day=0, Network Game=0. Появление этих значений =
дефект воспроизведён.

Дифференциальный тест (диагностический, для подтверждения причины).
Это проверка диагноза, а не обходной путь:
  - Воспроизвести вылет как выше, затем повторить ТОТ ЖЕ сценарий с
    увеличенным доступным адресным пространством процесса (включить
    флаг large-address-aware у исполняемого файла) — при росте
    свободной памяти nothrow-аллокатор перестанет отказывать, и
    вылет пропадёт. Исчезновение вылета именно при снятии давления
    кучи подтверждает, что отказ аллокатора — условие активации, а
    непроверенный возврат фабрики — первопричина.
  - Для надёжной фиксации причины рекомендуется снять ПОЛНЫЙ дамп
    (со всей кучей) при включённом твике Sys.FullDump = 1 в hota.ini:
    он позволит в отладчике убедиться, что поле [this+0x30] == 0 в
    момент сбоя, и что фабрика sub_10079970 действительно вернула
    NULL (EAX=0 после 0x10103B30).

Воспроизводимость: вероятностная (зависит от состояния кучи), но
сигнатура строго детерминирована. Гарантированной пошаговой
последовательности без давления кучи не существует — это свойство
самого класса дефекта (отказ nothrow-аллокатора непредсказуем во
времени). Тем не менее 26 независимых репортов за 2.5 месяца
свидетельствуют, что сценарий стабильно ловится у реальных игроков
именно в первый день одиночной партии.

ВЫВОД
-----

Виновник — HotA (код в hota.dll). Суть дефекта: при инициализации
интерфейсного спрайта книги заклинаний (SpelScBk.pcx) вызывающий код
не проверяет возврат фабрики растрового кэша sub_10079970, которая
корректно возвращает NULL, когда движковый nothrow-аллокатор (617492h)
не может выделить пиксельный буфер. Нулевой возврат используется как
указатель-объект (this) и передаётся в функцию заполнения растра
sub_1007A370, где первая же инструкция падает на разыменовании поля
+0x30 у нулевого this (чтение адреса 0x00000030). Пропуск проверки
доказан сравнением с соседним кодом: в той же функции возврат
operator new проверяется (test eax,eax; jz), а возврат фабрики — нет;
тот же шаблон повторён в семи точках вызова этой фабрики. Класс
дефекта присутствует и в версии до 1.8.0 (подтверждено на листинге
HotA.dll-153.txt, фабрика sub_100643D0), в SoD не применим — код
HotA-специфичен.
