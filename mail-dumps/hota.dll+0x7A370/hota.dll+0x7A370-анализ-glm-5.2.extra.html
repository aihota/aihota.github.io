Визуализация вылета #

Сбой происходит при инициализации интерфейсного спрайта книги заклинаний SpelScBk.pcx. Вызывающий код в цикле из 32 шагов создаёт растровые кэши и складывает их в глобальный вектор, не освобождая. На одной из средних итераций движковый аллокатор 617492h отказывает выделить пиксельный буфер, фабрика честно возвращает NULL, а вызвавший код передаёт этот NULL в функцию заполнения растра без проверки.

Путь данных от аллокатора до точки сбоя
1. Аллокатор SoD 617492h
выделить пиксельный буфер растра
отказ
return NULL (nothrow)
2. Фабрика sub_10079970
видит [obj+0x30] == 0
корректно вернула NULL
3. Инициализатор sub_10103400
mov ecx, eax
проверки нет
вызов sub_1007A370
4. Точка сбоя 0x1007A370
mov edx, [ecx+30h]
ECX = 0, this = NULL
чтение по адресу 0x00000030
Дескриптор растра, который функция sub_1007A370 пытается читать через ECX
this = ECX

Почему аллокатор отказывает: накопление растров в цикле
итерация 0..2
выделено
итерация 3..29 (EBP в дампах)
отказ на одной из них
итерация 30..31
не достигнуты

Первые итерации успешны, отказ наступает в середине цикла — это сигнатура деградации кучи (заполнение/фрагментация) внутри цикла накопления, а не битый размер одного растра (тогда отказ был бы на итерации 0). Растры складываются в глобальный вектор dword_10634F10, который между построениями не освобождается — накопление идёт и через всю игровую сессию.

сбой / отказ = отказ аллокатора, нулевой возврат, точка падения
корректно вернула NULL = фабрика честно транслирует отказ аллокатора
выделено = успешные итерации до отказа

В той же функции-инициализаторе проверка возврата другой аллокации (operator new) присутствует — значит отсутствие проверки после фабрики именно пропуск, а не сознательное решение.
