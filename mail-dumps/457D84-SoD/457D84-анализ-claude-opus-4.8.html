Визуализация вылета ¶

Тот же дефект, что в HotA-серии, но в чистом движке SoD (Heroes3.exe, базовый адрес 0x400000). Внутри менеджера игры по смещению +0x1F879 лежит массив команд игроков -- 8 байт (валидный индекс 0..7). Код читает его по номеру слота (EDI, в пределах) и по "выбранному игроку" (ECX). Перед вторым чтением проверяется только знак (id >= 0), верхняя граница -- нет. "Выбранный игрок" содержит мусор, поэтому адрес base + ECX + 0x1F879 уходит далеко за пределы массива.

BEAX = база
team[0..7]+0x1F879, 8 байт
Tteam[EDI] -- норма
...за массивом
Fteam[ECX] -- СБОЙ

чтение по "выбранному игроку" уходит далеко правее массива из 8 элементов

B база менеджера игры (EAX), выровнена по 64K -- валидна.
T штатное чтение team[EDI] по номеру слота (0..7) на 0x00457D7D -- проходит во всех вылетах.
F адрес сбоя base + ECX + 0x1F879, где ECX ("выбранный игрок") вне диапазона 0..7 -- проверена только нижняя граница.

Краткое резюме¶

В режиме кампании при клике на сценарий (по показаниям игроков -- "Spoils of War", при загрузке сохранённой кампании) игра падает в диалоге выбора сценария, пытаясь прочитать байт команды (альянса) "выбранного игрока" из таблицы заголовка карты. Номер выбранного игрока возвращает виртуальный метод объекта сценария кампании и используется как индекс в массив из 8 элементов; перед использованием проверяется только знак (не отрицательный), но НЕ верхняя граница. В логах номер выходит за пределы 0..7, поэтому адрес чтения уходит за пределы структуры менеджера игры.

Это та же точка сбоя и та же механика, что в серии вылетов HotA 1.8.0 по адресу 457D84, но здесь -- чистый движок SoD (поле HotA version отсутствует, модуль Heroes3.exe). Совпадение адреса, смещения поля и цепочки вызовов доказывает, что дефект унаследован от SoD.

Тип исключения: EXCEPTION_ACCESS_VIOLATION, чтение (read). Где: Heroes3.exe / Heroes3 HD.exe, адрес 0x00457D84, инструкция mov bl,[eax+ecx+1F879h]. Корневой регистр: ECX = "выбранный игрок" (вне диапазона) при валидной базе EAX.

Корреляция дампов (сравнительная таблица)¶

Все 41 вылет имеют один адрес сбоя (0x00457D84), одну операцию (read), один путь вызова (0x004578E4 → 0x00457D84) и общий набор инвариантных регистров. Различается лишь "выбранный игрок" (ECX) и производный от него адрес чтения.

Профиль "большой индекс" (36 из 41 вылетов): ECX -- большое произвольное значение, меняется от вылета к вылету. Примеры:

Группировка: один путь вызова, одна падающая функция, один шаблон отказа. Это системный путь, а не локальный баг конкретной карты (карты и сборки HD-мода разные при одном адресе сбоя).

Сопоставление с HotA-серией (папка 457D84): профиль "структурный индекс" в SoD -- ECX=0x00080006, в HotA -- ECX=0x00080004 (оба = значение вида 0x0008000X, старшее слово 0x0008). Это один и тот же шаблон отказа в обеих сборках; структурный вид согласуется с возвратом целого dword из виртуального метода (см. раздел 7), а не зажатого байта игрока.

Сопоставление адресов¶

Главный EXE (Heroes3.exe / Heroes3 HD.exe): базовый адрес образа = адрес загрузки = 0x00400000, поэтому адрес из лога совпадает с адресом листинга напрямую (runtime_addr — load_base + imagebase = runtime_addr).

Адрес сбоя 0x00457D84 → листинг .text:00457D84 (точное совпадение).

Адресная проверка инструкции сбоя (mov bl,[eax+ecx+1F879h]): faulting = EAX + ECX + 0x1F879

Вылет 10x09000000 + 0x1BDD86D2 + 0x1F879 = 0x24DF7F4B (совпало) Вылет 16: 0x0AB70000 + 0x00080006 + 0x1F879 = 0x0AC0F87F (совпало) Вылет 41: 0x0A6B0000 + 0x3353B775 + 0x1F879 = 0x3DC0AFEE (совпало)

Совпадение во всех вылетах подтверждает: база EAX валидна, смещение поля +0x1F879 фиксировано, за границы выводит именно ECX.

Идентичность смещения +0x1F879 с HotA-серией доказывает, что layout структуры менеджера игры (поле команды игрока) и сама инструкция по 0x00457D84 в SoD и в HotA одинаковы -- код унаследован без изменений.

Точка сбоя — разбор инструкций¶

Точка сбоя по адресу 0x00457D84 совпадает с разобранной в HotA-серии (папка 457D84). Функция -- расстановка союзников и врагов в диалоге выбора сценария кампании (по адресам HotA-листинга: sub_457CF0, начало 0x00457CF0). В SoD-режиме отдельный листинг чистого Heroes3.exe не требуется: совпадают адрес, смещение поля и цепочка вызовов, поэтому маппинг ведётся по тем же адресам главного EXE (базовый адрес 0x400000 общий для Heroes3.exe и h3hota.exe).

Восстановленная по HotA-листингу схема (адреса главного EXE стабильны):

.text:00457D06  mov  eax, [eax+1F468h]      ; arg = camp_info.selected_bonus_ix
.text:00457D10  mov  ecx, [ecx+edx*4]       ; scenarios_list[scenario_ix]
.text:00457D13  mov  ecx, [ecx+0A4h]        ; scenario->start_options
.text:00457D19  mov  edx, [ecx]             ; vtable
.text:00457D1B  call dword ptr [edx+20h]    ; Get_PlayerID(arg) -> id игрока
.text:00457D1E  mov  <SelectedPlayerID>, eax ; сохранение БЕЗ проверки
                                             ; верхней границы
...
.text:00457D5F  mov  eax, <GameMgr>          ; база менеджера игры
.text:00457D64  mov  cl, [eax+edi+1F6D0h]    ; флаг наличия игрока (слот EDI)
.text:00457D6D  jl   short <skip>            ; флаг<0 -> слот пропущен
.text:00457D71  jl   short <skip>            ; v1<0 -> пропуск
.text:00457D79  test ecx, ecx
.text:00457D7B  jl   short <skip>            ; id<0 -> пропуск (только знак!)
.text:00457D7D  mov  dl, [eax+edi+1F879h]    ; team[слот]   (норма)
.text:00457D84  mov  bl, [eax+ecx+1F879h]    ; team[id] <- СБОЙ
.text:00457D8B  cmp  dl, bl                  ; сравнение команд

Механика отказа: единственная проверка "выбранного игрока" перед использованием как индекса -- проверка знака (test ecx,ecx; jl). Верхней границы (id < 8) нет. EAX -- валидная база менеджера игры; +0x1F879 -- фиксированное смещение поля команды игрока в заголовке карты. При значении вне диапазона 0..7 адрес EAX + ECX + 0x1F879 уходит за пределы выделенной памяти, и чтение байта падает.

Поставщик значения (виртуальный вызов 0x00457D1B) разрешён по HotA-листингу: объект scenario→start_options (поле +0xA4) -- полиморфный _CampScenarioStartOptions_, слот его таблицы виртуальных функций +0x20 во всех конкретных типах указывает на метод Get_PlayerID. Подробный разбор типов и аргумента -- в разделе 7.

Регистровые инварианты¶

Постоянные во всех 41 вылете: EDI = 0x00000000 -- цикл падает на слоте игрока №0. (В HotA-серии было EDI=1: там слот №0 пропускался по флагу наличия, падало на слоте №1. Это лишь разные итерации одного и того же цикла -- механика идентична.) EDX = 0xFFFFFF00 -- младший байт DL = 0: team[слот №0], успешно прочитанный на 0x00457D7D перед сбоем (соседнее чтение по валидному индексу проходит). EBX = 0x00000071 -- рабочий байт = 0x71 + слот (lea ebx,[edi+71h] при EDI=0). В HotA было 0x72 при EDI=1 -- та же формула.

Переменные: EAX -- база менеджера игры, выровнена по 64K (0x..0000); своя аллокация на каждой машине -- штатно. ECX -- "выбранный игрок": в 36 вылетах большое произвольное значение (0x0A540847 ... 0x7CB8FCF8), в 5 вылетах структурное 0x00080006. Оба профиля -- значения, далеко выходящие за диапазон 0..7. Адрес чтения = EAX + ECX + 0x1F879 -- следствие выхода ECX за границу.

Контракт нарушен на величину, невозможную для исправного объекта: метод Get_PlayerID по контракту возвращает игрока в диапазоне байта (0..255, см. раздел 7), а в логах ECX -- сотни мегабайт либо структурный dword 0x00080006. Значит вернулось не валидное поле игрока, а целый dword из чужого/неверно интерпретированного объекта. Произвольность ECX между вылетами (в т.ч. на одной карте test.h3m) подтверждает, что значение не является стабильной величиной из файла сценария.

Восстановленная цепочка вызовов¶

Восстановлено по Call stack / Call stack V2 (адреса главного EXE маппятся напрямую, базовый адрес 0x400000):

_hd3_.dll / HD_SOD.dll (цикл событий интерфейса / диспетчер диалога)
0x0045A089 обработка диалога выбора сценария
0x00457670 обработчик выбора сценария кампании
EXE0x004578E4 call sub_457CF0 (расстановка союзников/врагов)

Кадры HD_SOD.dll / _hd3_.dll и трамплины patcher_x86 (0x020Axxxx) -- это драйвер интерфейса HD-мода, прокачивающий событие диалога. Виновный код лежит в главном EXE (движок SoD); HD-мод лишь доставляет событие открытия/обновления экрана выбора сценария кампании. Путь вызова идентичен HotA-серии. Кадр 0x004EF37C в стеке -- это путь главного меню A0_MainMenu, через который кампания продолжается из сохранения (см. раздел 7).

Первопричина¶

Непосредственный дефект (точка сбоя): функция расстановки союзников и врагов берёт "выбранного игрока" виртуальным методом объекта сценария кампании и использует его как индекс в массив команд игроков (8 элементов) без проверки верхней границы. Из проверок перед чтением (флаг наличия слота, знак номера слота, знак "выбранного игрока") ни одна не ограничивает "выбранного игрока" сверху значением 8.

Поставщик значения разрешён по листингу. Объект scenario→start_options (поле +0xA4) -- полиморфный _CampScenarioStartOptions_, создаётся при чтении файла кампании (A0_CampScenario_Read_sub_487980) одним из трёх типов по тег-байту из файла: 1 → Bonus, 2 → Player, 3 → Hero, иначе start_options остаётся NULL. Слот таблицы виртуальных функций +0x20 во всех трёх типах -- это метод Get_PlayerID. Его реализации НЕ эквивалентны по диапазону результата:

Player (sub_4852C0)movsx eax, byte ptr [list+arg*2] -- знаковый байт, диапазон -128..127;
Bonus (sub_484CB0)mov eax, [ecx+4] -- dword, записанный как (x & 0xFF), диапазон 0..255;
Hero (sub_485680)mov eax, [[ecx+8] + arg*8] -- ЦЕЛЫЙ dword по индексу arg, БЕЗ клампа.

Аргумент метода (arg) -- это camp_info.selected_bonus_ix (поле менеджера игры +0x1F468), считываемое на 0x00457D06 и тоже не проверяемое сверху.

Отсюда два уровня отсутствующих проверок верхней границы, оба на индексах, восстановленных из сохранённого состояния кампании: 1) scenario_ix (this+0x6C) -- выбор объекта scenarios_list[scenario_ix]; 2) selected_bonus_ix (camp_info +0x1F468) -- аргумент Get_PlayerID, используемый Hero-вариантом как [base + arg*8] и Player-вариантом как [list + arg*2]. Структурное значение ECX=0x00080006 (старшее слово 0x0008) согласуется именно с Hero-вариантом: возвращён целый dword, а не зажатый байт игрока. Этот dword функция sub_457CF0 принимает за номер игрока 0..7 и без верхней проверки кладёт в индекс таблицы команд.

Почему объект-поставщик оказывается негодным. Список сценариев и их объекты start_options НЕ сериализуются в сейв: при продолжении кампании из сохранения путь главного меню A0_MainMenu_sub_4EEA70 (кадр 0x004EF37C в стеке этой серии) заново строит scenarios_list из файла .h3c вызовом A0_CampInfo_Load_sub_4883C0 (число сценариев = таблица A0_CampMapInfos[campaing_id].region_count). Из сейва восстанавливаются только индексы (scenario_ix, selected_bonus_ix). Если сохранённые индексы не соответствуют заново загруженному списку (другая версия или мод кампании, иной campaing_id, иное число сценариев), scenario[scenario_ix] и/или Get_PlayerID(selected_bonus_ix) читают за границей → возвращается мусор → чтение team[id] падает. Тот же паттерн "индекс из сейва → Get_PlayerID → индекс игрока без проверки" повторно встречается в A0_MainMenu_sub_4EEA70 (0x4EFB81, 0x4EFFCD), что доказывает системность контракта, а не единичную опечатку. Это согласуется с показаниями игроков ("loading saved campaign", "Spoils of War").

Сравнение с соседними путями (доказательство пропущенной проверки): в главном EXE около 120 чтений того же поля команды team[+0x1F879], и почти все парные сравнения берут оба индекса из циклов 0..7. Путь 0x00457D7D/0x00457D84 -- единственный, где один индекс приходит из виртуального метода, а не из ограниченного цикла; именно у него нет проверки верхней границы.

Наследование от SoD: дефект присутствует в чистом движке SoD (модуль Heroes3.exe, 25 из 41 вылета, без HotA). Совпадение адреса, смещения поля +0x1F879, цепочки вызовов и регистровых инвариантов с HotA-серией доказывает, что HotA унаследовала эту функцию от SoD без правки границы.

Вывод по первопричине: место сбоя -- симптом отсутствующей проверки верхней границы индекса. Поставщик значения разрешён (Get_PlayerID полиморфного объекта start_options), и механизм понятен: два индекса из сохранённого состояния кампании используются без верхней границы поверх списка сценариев, заново загружаемого из .h3c. КОНКРЕТНЫЙ корраптор (какой именно фактор разводит сохранённые индексы с перезагруженным файлом -- версия/мод кампании, повреждение сейва) по статике не фиксируется: для него нужен сейв игрока или минидамп.

Условия, спровоцировавшие сбой¶

  • Режим кампании (одиночная игра): Game Type = 0, Network Game = 0, Day = 1, до начала партии.
  • Показания игроков (3 независимых лога) прямо указывают сценарий: клик на сценарий кампании "Spoils of War" при загрузке сохранённой кампании; один лог -- "End of the Inferno senario". Повторяемость по словам игрока высокая ("happend 3 times and keeps happening").
  • Путь сбоя исполняется при выборе сценария в диалоге кампании, в т.ч. при продолжении сохранённой кампании через главное меню (кадр 0x004EF37C).
  • Настройки графики/боя/памяти на путь сбоя не влияют -- они различаются между вылетами при одном адресе.
  • Карты и сборки HD-мода разные; общий знаменатель -- сам экран выбора сценария кампании, а не конкретная карта или сборка.

Достоверность и ограничения¶

Верификация листинга: модуль сбоя -- главный EXE движка SoD (Heroes3.exe / Heroes3 HD.exe), базовый адрес образа 0x00400000. Отдельный листинг чистого Heroes3.exe не привлекался: адрес 0x00457D84, смещение поля +0x1F879, цепочка вызовов и регистровые инварианты совпадают с разобранной HotA-серией, где инструкция по 0x00457D84 (mov bl,[eax+ecx+1F879h]) сверена с листингом Модули/h3hota.exe.txt байт-в-байт. Heroes3.exe и h3hota.exe имеют общий базовый адрес 0x400000, поэтому унаследованный код лежит по тем же адресам. Маппинг прямой.

Достоверность сигнатуры -- очень высокая: 41 вылет, один адрес, один путь вызова, устойчивые инварианты (EDI=0, EDX=0xFFFFFF00, EBX=0x71). Достоверность МЕХАНИЗМА -- высокая: поставщик значения разрешён по листингу (виртуальный Get_PlayerID полиморфного объекта start_options), доказаны два непроверенных индекса (scenario_ix, selected_bonus_ix) из сохранённого состояния кампании поверх списка, заново загружаемого из .h3c; механизм подтверждён показаниями игроков о загрузке сохранённой кампании. Достоверность КОНКРЕТНОГО КОРРАПТОРА -- низкая: какой фактор разводит сохранённые индексы с перезагруженным файлом (версия/мод кампании, повреждение сейва), по статике не определить -- нужен сейв игрока или минидамп.

Полезность лога -- СРЕДНЯЯ: место сбоя и контракт поставщика локализованы точно, но первопричина (рассинхрон индексов сейва с перезагруженным списком сценариев) находится выше по стеку и по статике не сводится к одному фиксу в точке сбоя. Проверка верхней границы индекса в точке сбоя (и/или в Get_PlayerID) убрала бы падение, но это защита от симптома; полноценный фикс -- согласование сохранённых индексов с заново загружаемой кампанией.

Ограничение по чистому листингу SoD: листинг отдельного Heroes3.exe в проекте отсутствует. Разбор поставщика (Get_PlayerID, типы start_options, путь A0_MainMenu/A0_CampInfo_Load) выполнен по листингу h3hota.exe; перенос на чистый SoD обоснован общим базовым адресом 0x400000 и совпадением всех сигнатур. Для трассировки в точности по чистому SoD понадобился бы листинг Heroes3.exe нужной сборки.

Кадры HD_SOD.dll / _hd3_.dll и трамплины patcher_x86 в стеке не маппятся в листинг (драйвер интерфейса HD-мода и runtime-стабы хуков); на диагноз виновного модуля это не влияет. Рассинхрон версий HD-мода (5.7 R7/R8/R9/R12/R13) -- только в периферийном модуле; падающий главный EXE постоянен по базовому адресу, поэтому рассинхрон диагноз не блокирует.

Признаки поиска в старых версиях HotA / в SoD¶

Код места сбоя находится в главном EXE движка Shadow of Death. Эта серия сама по себе доказывает присутствие дефекта в чистом SoD (модуль Heroes3.exe без HotA), поэтому баг не специфичен для HotA и существовал до версии 1.8.0.

Стабильные сигнатуры для опознания вылета независимо от версии:

  • Context-инварианты: EDX=0xFFFFFF00 (DL=0), EBX=0x71 (при EDI=0) или 0x72 (при EDI=1); адрес сбоя -- чтение по EAX + ECX + 0x1F879 при валидном EAX и выходящем за диапазон ECX.
  • Текстовая сигнатура по дизассемблеру (поиск в листинге любой версии): mov ecx, <SelectedPlayerID-глобал> test ecx, ecx jl short <skip> mov dl, [eax+edi+1F879h] mov bl, [eax+ecx+1F879h] <- инструкция сбоя cmp dl, bl Характерно отсутствие сравнения индекса с верхней границей (нет cmp с 8) между проверкой знака и чтением.
  • Стабильные адреса главного EXE: функция расстановки союзников/врагов 0x00457CF0, вызов из обработчика выбора сценария 0x004578E4, поставщик значения -- виртуальный метод Get_PlayerID объекта start_options (слот таблицы виртуальных функций +0x20). Эти адреса общие для Heroes3.exe (SoD) и h3hota.exe (HotA), так как оба имеют базовый адрес 0x400000.
  • Сиблинг: соседнее чтение 0x00457D7D (того же поля команды по индексу слота с проверкой знака слота) -- проверки верхней границы там также нет, потому что слот ограничен циклом 0..7; для "выбранного игрока" такого ограничения нет.

Воспроизведение вылета¶

Уровень воспроизводимости: по показаниям игроков -- высокий ("happend 3 times and keeps happening"), хотя само значение "выбранного игрока" в логах выходит за диапазон. Путь сбоя детерминирован (открытие/выбор сценария в диалоге кампании всегда доходит до чтения по индексу "выбранного игрока"), а попадание за границу определяется рассинхроном сохранённого состояния кампании с заново загружаемым файлом .h3c.

Условие активации пути (из кода): функция расстановки союзников/врагов вызывается при выборе сценария в диалоге кампании; до чтения по индексу "выбранного игрока" перебирается слот игрока (в логах SoD падает на слоте №0).

Роль 1 -- тестер за игрой (без отладчика):

  • Запустить SoD-режим под HD-модом (по логам -- HD-мод 5.7 R7...R13; встречается и чистый Heroes3.exe, и Heroes3 HD.exe).
  • Войти в режим кампании и ЗАГРУЗИТЬ сохранённую кампанию, затем кликнуть на сценарий. По показаниям игроков воспроизводится на сценарии "Spoils of War" (упоминается также "End of the Inferno"); стоит проверить и другие сценарии.
  • Повышает шанс: сейв кампании, сделанный в другой версии или сборке кампании, чем установленная сейчас (рассинхрон индексов сохранённой кампании с перезагружаемым файлом .h3c).
  • Что ловить: сам факт вылета с адресом 0x00457D84. По словам игрока происходит повторяемо при клике на сценарий загруженной кампании.
  • Варьировать: разные кампании и сценарии внутри них, повторные загрузки сохранённой кампании, открытие диалога выбора сценария.

Роль 2 -- подтверждение причины (с отладчиком / минидампом):

  • Точка останова на 0x00457D84. Подтвердить, что ECX ("выбранный игрок") вне диапазона 0..7 и что faulting = EAX + ECX + 0x1F879.
  • Подняться к виртуальному вызову 0x00457D1B: снять тип объекта start_options ([ecx] -- какая из таблиц Player/Bonus/Hero), значение аргумента selected_bonus_ix (camp_info +0x1F468) и возврат метода.
  • Сверить scenario_ix (this+0x6C) и реальный размер scenarios_list с тем, что даёт перезагрузка .h3c (число = A0_CampMapInfos[campaing_id]. region_count): расхождение подтверждает рассинхрон индексов сейва.
  • Дифференциальный тест: условный останов/патч, пропускающий путь при "выбранном игроке" >= 8, должен убрать именно этот вылет -- это подтверждает, что за границы выводит индекс, а не база.
  • Минимальные данные для репорта разработчикам: краш-лог + минидамп, тип объекта start_options, значения scenario_ix и selected_bonus_ix, имя кампании/сценария ("Spoils of War") и сейв кампании.

Вывод¶

Виновник -- код главного EXE движка Shadow of Death (диалог выбора сценария кампании, функция расстановки союзников и врагов). "Выбранный игрок" возвращает виртуальный метод Get_PlayerID объекта сценария кампании и используется как индекс в массив команд игроков из 8 элементов с проверкой только знака и без проверки верхней границы. Механизм разрешён по статике: два индекса из сохранённого состояния кампании (выбор сценария и аргумент Get_PlayerID) применяются без верхней границы поверх списка сценариев, заново загружаемого из файла .h3c; при их рассинхроне метод возвращает значение вне диапазона 0..7, и чтение байта команды уходит за пределы памяти. Дефект присутствует в чистом SoD (Heroes3.exe) и унаследован HotA без изменений -- это та же точка сбоя, что в серии вылетов HotA 1.8.0 по адресу 457D84. Точка сбоя и контракт поставщика подтверждены арифметикой и листингом во всех 41 вылете; конкретный фактор рассинхрона (версия/мод кампании, повреждение сейва) по статике не локализуется -- для него нужен сейв игрока или минидамп.
