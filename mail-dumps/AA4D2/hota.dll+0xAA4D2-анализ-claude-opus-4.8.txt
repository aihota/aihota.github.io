Заключение: Виновник - hota.dll. Обработчик подсистемы карты приключений

КРАТКОЕ РЕЗЮМЕ
--------------
При действии игрока на карте приключений (обработка щелчка мышью, идущая
через HD-мод обратно в hota.dll) вызывается функция hota.dll sub_100AA4B0.
Она берёт небольшой идентификатор объекта (0..4 во всех логах), умножает
его на размер записи 0x13 (19 байт) и прибавляет базовый адрес таблицы
записей подсистемы. Базовый адрес лежит в поле "подсистема+0x10" и во всех
12 логах равен нулю. В результате чтение поля записи по адресу
"0 + id*19 + 0x0D" падает по нулевому указателю.

Тип исключения: EXCEPTION_ACCESS_VIOLATION, чтение.
Адрес чтения: маленький (0x0D, 0x20, 0x33, 0x46, 0x59) - это id*19 + 0x0D.
Где: hota.dll, функция sub_100AA4B0, инструкция cmp byte ptr [esi+0Dh], 0.
Корневой регистр: ESI = id*19 (id из {0,1,2,3,4}); базовый адрес таблицы,
прибавляемый к ESI, равен нулю.

Сам объект подсистемы выделен корректно (регистр EAX во всех логах -
правдоподобный адрес кучи, чтение его поля +0x10 проходит и возвращает 0).
Отсутствует именно таблица записей, на которую это поле должно указывать.

Уточнение по структуре (установлено по коду): "таблица" - это обычный
растущий вектор. Его заголовок лежит прямо в объекте подсистемы:
поле +0x10 = begin (начало), +0x14 = end (конец), +0x18 = capacity
(ёмкость). begin = NULL во всех 12 логах означает, что вектор ПУСТ -
это штатное начальное состояние пустого вектора, а НЕ повреждённый
("мусорный") указатель. Подсистема - именованный обработчик движка
HotA "HotA.CanHaveBattles" ("может ли объект иметь бои"); его и вызывает
HD-мод при обработке действия по карте.

То есть sub_100B5C60 возвращает адрес поля внутри GameMgr по фиксированному
смещению dword_10634AC8. По этому полю лежит указатель на объект подсистемы.
Чтение "mov eax,[eax]" (разыменование указателя подсистемы) проходит
успешно - объект подсистемы выделен. Падает следующее звено: поле
"подсистема+0x10" хранит базовый адрес таблицы записей, и он равен нулю.

Механика отказа: чтение поля записи [begin + id*19 + 0x0D] при begin = NULL.
Поле +0x0D - это признак валидности записи (см. раздел 7). Размер записи
0x13 = 19 байт. Поля заголовка вектора: подсистема+0x10 = begin,
подсистема+0x14 = end, подсистема+0x18 = capacity; число записей
count = (end - begin) / 19. Падающая функция НЕ проверяет ни begin на
ноль, ни id против count - сразу считает адрес записи.


5. РЕГИСТРОВЫЕ ИНВАРИАНТЫ
-------------------------
Постоянные по всем 12 логам:
   - адрес чтения = ESI + 0x0D (поле записи), где ESI = id*0x13;
   - id = (*arg_0 & 0x3FF) принимает только малые значения {0,1,2,3,4};
   - базовый адрес таблицы (подсистема+0x10) = 0 (доказано в разделе 3:
     адрес чтения не содержит слагаемого-базы);
   - EAX = подсистема + 0x10 - правдоподобный адрес кучи (объект цел);
   - вызывающий код один и тот же: HD_HOTA.dll+0x134B74.

EDI, EBX, ECX, EDX, EBP/ESP в адресной арифметике сбоя не участвуют.
EDI в логах принимает разные малые значения и к индексу записи отношения
не имеет (совпадение EDI с адресом чтения - лишь в логе N1).

Размеры буферов в ECX/EDX к формуле буфера не привязаны: размер записи
фиксирован (0x13), а вылет вызван отсутствием самой таблицы, а не её
размером. Обратную оценку размера карты по ECX применять нельзя.

Малые допустимые id и стабильно чистый ноль базы (а не "мусор" и не
маркер освобождённой памяти 0xFEEEFEEE) указывают, что таблица НЕ
ВЫДЕЛЕНА, а не повреждена/освобождена.

7. ПЕРВОПРИЧИНА
---------------
Падающая функция sub_100AA4B0 обращается к таблице записей подсистемы по
формуле "база + id*19" и сразу читает поле +0x0D без проверки базы на ноль.

Сравнение с родственными функциями той же подсистемы (та же связка
"and esi,3FFh; call sub_100B5C60; imul esi,13h; mov eax,[eax];
add esi,[eax+10h]; cmp byte ptr [esi+0Dh],0"):
   - sub_1007FA70 (.text:1007FA89)
   - функция при .text:1007FC14
   - sub_1007FE20 (.text:1007FE39)
   - sub_100A84C0 (.text:100A84EE)
Все четыре используют ИДЕНТИЧНЫЙ шаблон и НИ ОДНА не проверяет базовый
адрес таблицы (поле +0x10) на ноль перед чтением записи. То есть вся
подсистема построена на договорённости "таблица всегда выделена" - это
наследованный контракт, а не единичная забытая проверка.

Путь к сбою единственный и незащищённый: sub_100AA4B0 нигде не вызывается
напрямую - она только один раз регистрируется как обработчик
"HotA.CanHaveBattles" и дёргается через мост HD-мода. Защищённого
варианта вызова (с проверкой begin) в hota.dll нет.

Поставщик данных (наполнение вектора): загрузчики sub_100A8860 и
sub_100A8AA0 (оба - именованные обработчики того же реестра). Они
десериализуют 19-байтовую запись через sub_100B5FA0 (чтение полей из
потока call [edi+4], версионная градация по dword_10634D20: поля при
версии >=4, >=6, >=9; поле +0x0D, на котором падаем, читается при
версии >= 9), сбрасывают её (байт +0x0D = 0, +0x0E = 0xFFFFFFFF
"нет объекта", байт +0x12 = 0) и ВСТАВЛЯЮТ в вектор через sub_100ABD20.
Вставка - типичный рост вектора: при end == capacity вызывается
расширение sub_100ABC20, затем 19 байт записи копируются и end
увеличивается на 0x13. Значит begin становится НЕ-нулевым только
после того, как в вектор добавлена хотя бы одна запись этим путём.

Что доказано кодом и арифметикой:
   - падение - чтение записи из ПУСТОГО вектора (begin = NULL);
   - объект подсистемы при этом валиден (EAX - корректный адрес кучи),
     id корректен (0..4), повреждения самого объекта/индекса нет;
   - begin = NULL - это штатное начальное состояние пустого вектора
     (стандартный заголовок begin/end/capacity), а НЕ повреждённый
     указатель: версия "редкой порчи поля базы" исключена;
   - падающая функция не проверяет НИ begin на ноль, НИ id против
     count = (end-begin)/19; все четыре родственных обработчика
     подсистемы тоже не проверяют - это сквозной контракт "вектор уже
     наполнен".

Итог по первопричине: дефект в hota.dll - обработчик "HotA.CanHaveBattles"
читает запись из вектора, который к моменту запроса пуст. Поскольку
пустой вектор - это нормальное состояние контейнера (не порча), вопрос
сводится к логике ВЫШЕ точки чтения: либо запрос "CanHaveBattles"
штатно может прийти для объекта, у которого записей нет (тогда нужна
проверка begin/count в самом обработчике - её отсутствие и есть дефект),
либо нарушен порядок: загрузчик (sub_100A8860/sub_100A8AA0) ещё не
наполнил вектор к моменту обращения. Разделить эти два варианта по
статике нельзя - требуется минидамп или точка останова на загрузчике
(см. раздел 11). Сам факт чтения из пустого контейнера без проверки
доказан кодом.

пустого вектора (стандартный заголовок begin/end/capacity по
подсистема+0x10/+0x14/+0x18), а не повреждённый указатель: версия
рантайм-порчи поля базы ИСКЛЮЧЕНА. Найден и поставщик данных: вектор
наполняется загрузчиками sub_100A8860/sub_100A8AA0 через вставку
sub_100ABD20; begin становится не-нулевым только после первой вставки.
Падающая функция следует сквозному контракту подсистемы "вектор уже
наполнен" (одинаково у всех её обработчиков, проверки begin/count нет ни
у одного), поэтому точечная правка на месте сбоя - лишь маскировка.
Первопричина - выше точки чтения: либо запрос "CanHaveBattles" штатно
приходит для объекта без записей (нужна проверка begin/count в
обработчике), либо вектор не наполнен загрузчиком к моменту обращения.
Разделить эти два варианта по статике нельзя - нужен минидамп (проверить
begin/end по подсистема+0x10 и был ли отработан загрузчик) или точка
останова на вставке sub_100ABD20. Это и удерживает вердикт на уровне
СРЕДНЯЯ, а не ВЫСОКАЯ.

Ограничения маппинга: ряд кадров Call stack - трамплины patcher_x86, в
листинге не представлены (по построению). Это не мешает диагнозу: точка
сбоя и непосредственный вызывающий код определены по V1+V2 и совпадают.

Листинги, не относящиеся к диагнозу и потому не запрошенные: _hd3_.dll
5.7 R13, системные DLL. Для текущего диагноза достаточно hota.dll 1.8.0
и HD_HOTA.dll 5.7 R13.


10. ПРИЗНАКИ ПОИСКА В СТАРЫХ ВЕРСИЯХ HotA / В SoD
-------------------------------------------------
Адрес hota.dll+0xAA4D2 между версиями HotA не стабилен. Виновный код лежит
в hota.dll (HotA-специфичная подсистема), в чистом SoD-движке h3hota.exe
его нет.

Стабильные сигнатуры для опознания дефекта независимо от версии:
   - Регистровые инварианты: адрес чтения = ESI + 0x0D, ESI кратен 0x13
     (19), базовый адрес таблицы (подсистема+0x10) = 0, объект подсистемы
     валиден; вызов всегда с собственного хода игрока (Me == Active Player).
   - Текстовая сигнатура по дизассемблеру (ищется в hota.dll любой версии):
        and  esi, 3FFh
        call <вспом. функция доступа к полю GameMgr>
        imul esi, 13h
        mov  eax, [eax]
        add  esi, [eax+10h]
        cmp  byte ptr [esi+0Dh], 0
   - Родственные обработчики той же подсистемы с тем же шаблоном
     (sub_1007FA70, функция при +0x7FC14, sub_1007FE20, sub_100A84C0) -
     все без проверки базы таблицы на ноль.
   - Признак поставщика: функция-десериализатор записей (аналог
     sub_100B5FA0) с версионной градацией полей и сбросом
     запись+0x0D = 0, запись+0x0E = 0xFFFFFFFF.
   - Текстовая сигнатура по строкам (стабильна между версиями):
     имя обработчика "HotA.CanHaveBattles" в реестре функций HotA
     (рядом push offset падающей функции и call [reg+30h]); тот же
     литерал "HotA.CanHaveBattles" в мосте HD_HOTA.dll.
   - Признак вставки в вектор: helper роста (аналог sub_100ABD20),
     копирующий 19 байт записи (dword*3 + word + byte) и делающий
     add [vector+4], 13h.


11. ВОСПРОИЗВЕДЕНИЕ ВЫЛЕТА
--------------------------
Условие активации пути (из кода): обработка взаимодействия игрока с
объектом карты приключений (A0_AdvMgr_MapClick -> хук HD-мода ->
sub_100AA4B0), на СОБСТВЕННОМ ходу игрока, когда у подсистемы ещё/уже
не выделена таблица записей (поле подсистема+0x10 = NULL), а запрос идёт
по малому id объекта (0..4).

Роль 1 - тестер за игрой (без отладчика):
   - Окружение из логов: HotA 1.8.0 + HD-мод 5.7 R13.
   - Наиболее перспективные исходные данные (повторились у независимых
     игроков): карта "The Battle of Daeyan' ..." (логи N2, N4, N12; пара
     N2/N12 - один день и тот же id=1) и "Peaceful Ending.h3m" (логи
     N9/N10, соседние дни, id=0). Если эти карты/сейвы доступны, начинать
     с них.
   - Сценарий: на своём ходу взаимодействовать (щелчок/наведение по пути
     обработки клика) с объектами карты приключений в начале партии
     (во всех логах ранняя стадия: день из лога), повторяя действия с
     первыми объектами (малые id). Что ловить тестеру - сам факт вылета
     с тем же адресом hota.dll+0xAA4D2.
   - Варьировать: одиночная и сетевая игра (вылет встречается в обоих),
     разные карты ранней стадии, повтор взаимодействия с первыми
     объектами. Бой/быстрый бой не нужны - путь от них не зависит.
   - Воспроизводимость честно: вылет НЕ на каждом клике (иначе игра
     падала бы у всех сразу) - есть неустановленное условие, при котором
     таблица к моменту запроса пуста. Поэтому нужен прогон по нескольким
     ранним состояниям/картам, а не одна попытка.

Роль 2 - подтверждение причины (с отладчиком / по минидампу):
   - Точка останова на инструкции сбоя (hota.dll+0xAA4D2,
     cmp byte ptr [esi+0Dh], 0).
   - Снять EAX (это подсистема+0x10) и значение по нему: причина
     подтверждается, если [EAX] == 0 (базовый адрес таблицы = NULL),
     при этом сам EAX - валидный адрес кучи (объект подсистемы цел),
     а ESI == id*0x13 при малом id.
   - Проверить формулу: адрес чтения == ESI + 0x0D и == id*0x13 + 0x0D
     (без слагаемого-базы). Совпадение доказывает нулевую базу таблицы.
   - Дифференциальный тест первопричины: поставить точку останова на
     вставке в вектор sub_100ABD20 и на загрузчиках sub_100A8860/
     sub_100A8AA0; убедиться, что в упавших сессиях к моменту запроса
     "CanHaveBattles" вставки в этот вектор не было (begin так и остался
     NULL, end == begin). Если обеспечить наполнение вектора до запроса,
     путь сбоя перестаёт достигаться - это подтверждает, что корень
     выше точки чтения (порядок наполнения / запрос к объекту без
     записей), а не сама инструкция сбоя.


ВЫВОД
-----
Виновник - hota.dll. Обработчик "HotA.CanHaveBattles" (sub_100AA4B0)
читает запись по корректному малому id объекта из вектора записей, но
вектор пуст: его начало (поле подсистема+0x10) равно нулю. Это штатное
состояние пустого вектора, а не порча памяти - объект подсистемы цел,
индекс валиден. Все родственные обработчики подсистемы одинаково
доверяют тому, что вектор наполнен, и не проверяют ни его начало, ни
границу. Поэтому первопричина не в точке чтения, а выше: либо запрос
приходит для объекта без записей (нужна проверка в обработчике), либо
вектор не наполнен загрузчиком к моменту обращения.
