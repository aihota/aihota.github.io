АНАЛИЗ ВЫЛЕТА HD_TC2.dll+0x6805 (отрисовка Pcx8)
======================================================
Модель: Claude Opus 4.8
Дата: 22.06.2026
Заключение: Вылет происходит в 32-битном порте функции Pcx8_Draw:
в источник изображения приходит нулевой указатель. Сам цикл HD_TC2
повторяет контракт исходной функции SoD и показывает симптом; дальнейшая
проверка сужает поставщика до массива small_portraits_dwordC8 диалога
выбора карты и загрузчика A0_Load_Pcx8_sub_55AA10.

КРАТКОЕ РЕЗЮМЕ
--------------
Порт рисовалки Pcx8 в HD_TC2 читает первый байт исходного изображения
через EDX, но во всех семи уникальных логах EDX равен 0. Поэтому
инструкция чтения падает на адресе 0x00000000.

Функция HD_TC2 зарегистрирована как замена исходной функции SoD
h3hota.exe:0044F940. Сравнение показывает, что исходная функция SoD
тоже не проверяет источник перед чтением: это контракт рисовалки
"вызывающий код передает готовую поверхность". Значит, место сбоя
не является доказанным корнем. Более вероятный корень расположен выше:
вызвавший код передал Pcx8-источник с нулевой базой или рассогласованной
структурой.

Дополнительная проверка поставщика сужает механизм: в подтверждённом
пути отрисовки параметров игрока индекс портрета берётся из данных
выбранного героя/заголовка карты, затем без проверки границ используется
как индекс массива small_portraits_dwordC8. Сам массив заполняется
загрузчиком A0_Load_Pcx8_sub_55AA10 из HPS-портретов; этот загрузчик может
вернуть созданный Pcx8-объект с нулевым буфером после ошибки чтения PCX.

Стек указывает на перерисовку диалога выбора карты и параметров игрока.
Поля Map file и Day приведены в логах дословно, но не доказывают, что
именно эти карты или день игры являются причиной.

3. СОПОСТАВЛЕНИЕ АДРЕСОВ
------------------------
Для первого лога:

    Адрес загрузки HD_TC2.dll: 0x07CB0000
    Базовый адрес образа листинга: 0x02000000
    Адрес во время работы: 0x07CB6805

Расчет:

    0x07CB6805 - 0x07CB0000 + 0x02000000 = 0x02006805

Смещение модуля:

    0x07CB6805 - 0x07CB0000 = 0x6805

Смещение совпадает с записью лога HD_TC2.dll+0x6805. Листинг содержит
падающую инструкцию по адресу 0x02006805.

Шапка листинга HD_TC2.dll:

    .text:02001000 ; File Name   : HD_TC2.dll
    .text:02001000 ; Imagebase   : 2000000
    .text:02001000 ; Timestamp   : 69F84603 (Mon May 04 07:08:51 2026 UTC)
    .text:02001000 ; PDB File Name : HD_True32.pdb

Во всех семи логах размер загруженного HD_TC2.dll одинаков:
0x003CF000. В текстовом логе нет timestamp или хэша модуля, поэтому
совпадение версии листинга подтверждается по единственному доступному
листингу, базовому адресу образа, постоянному размеру модуля и совпавшему
смещению функции. Это достаточно для разбора точки сбоя, но не закрывает
вопрос точного хэша сборки.

не применяется: адрес сбоя равен самому регистру-источнику.

5. РЕГИСТРОВЫЕ ИНВАРИАНТЫ
-------------------------
Главный инвариант - нулевая база источника:

    EDX = 0 во всех семи вылетах.
    EDI = 0 во всех семи вылетах.

По коду EDI является рассчитанным началом текущей строки источника:

    source_row = [Pcx8+0x30] + [Pcx8+0x2C] * y + x
    EDX = source_row

Раз EDI и EDX равны 0 в момент чтения, вызывающий путь передал функцию
с источником, у которого рассчитанная строка начинается с нулевого адреса.
Наиболее простой вариант - поле базы изображения [Pcx8+0x30] равно 0 при
нулевых x/y-смещениях. Другие варианты, например рассогласованные аргументы,
тоже находятся выше по цепочке вызовов; сама инструкция только фиксирует
нулевой источник.

EAX и ESI в момент сбоя содержат ширину текущей строки:

    0x30 в шести вылетах
    0x252 в одном вылете

Это не адрес и не размер выделения. Значение показывает, что цикл успел
подготовить количество пикселей для копирования и упал на первом чтении
строки источника.

Уточнение поставщика данных.

Дальнейшая проверка показала, что для пути с шириной 0x30 источник
приходит из массива маленьких портретов диалога выбора карты.

В отрисовке параметров игрока код получает индекс портрета и передаёт
соответствующий Pcx8-объект в рисовалку:

    .text:0058DD3E ; 367:   port_id_v68 =
    .text:0058DD3E ;        A0_MapSelectDlg_Player_Get_SelectedHero_Portrait_sub_58D3B0(this_v6, player_ix);
    .text:0058DD41                 call    A0_MapSelectDlg_Player_Get_SelectedHero_Portrait_sub_58D3B0
    .text:0058DDE6                 cmp     eax, 0FFFFFFFFh
    .text:0058DDE9                 jnz     loc_58DEFA
    .text:0058DF02 ; 452:       this_v6->small_portraits_dwordC8[port_id_v68],
    .text:0058DF02 ; 455:       this_v6->small_portraits_dwordC8[0]->width,
    .text:0058DF02 ; 456:       this_v6->small_portraits_dwordC8[0]->height,
    .text:0058DF33                 mov     ecx, [esi+eax*4+0C8h] ; this
    .text:0058DF3C                 call    A0_Pcx8_Draw_To_Pcx16_sub_44FA80

Это объясняет важную деталь: ширина 0x30 в шести логах может быть взята
из эталонного слота small_portraits_dwordC8[0], а не из повреждённого
портрета. Поэтому повреждённый Pcx8-объект может иметь нулевой буфер
[+0x30], но цикл всё равно получает ненулевую ширину 0x30 от слота 0.

Функция выбора портрета проверяет только sentinel -1, но не проверяет
верхнюю границу индекса и не проверяет валидность загруженного Pcx8:

    .text:0058D40F                 mov     eax, [eax+20h]
    .text:0058D412                 cmp     eax, 0FFFFFFFFh
    .text:0058D415                 jnz     loc_58D4B2
    .text:0058D442                 mov     eax, [edx+20h]
    .text:0058D445                 cmp     eax, 0FFFFFFFFh
    .text:0058D448                 jnz     short loc_58D454
    .text:0058D454                 mov     edx, [edx+eax*4+2Ch]
    .text:0058D4A9                 cmp     ecx, 0FFFFFFFFh
    .text:0058D4AE                 mov     edx, ecx
    .text:0058D4B0                 mov     eax, edx

Массив small_portraits_dwordC8 заполняется в конструкторе диалога
выбора карты. Слоты 0..162 загружаются из таблицы HPS-портретов героев,
а слот 163 - из hpsrand.pcx:

    .text:0057C7B1                 xor     edi, edi
    .text:0057C7B3                 mov     esi, 0C8h
    .text:0057C7B8                 mov     edx, A0_HeroInfo_off_67DCE8
    .text:0057C7BE                 mov     ecx, [edx+edi+30h] ; this
    .text:0057C7C2                 call    A0_Load_Pcx8_sub_55AA10
    .text:0057C7C7                 mov     [esi+ebx], eax

    .text:0057C7CD                 add     edi, 5Ch ; '\\'
    .text:0057C7D0                 cmp     esi, 354h
    .text:0057C7D6                 jl      short loc_57C7B8
    .text:0057C7D8                 mov     ecx, offset aHpsrand_pcx ; "hpsrand.pcx"
    .text:0057C7DD                 call    A0_Load_Pcx8_sub_55AA10
    .text:0057C7E7                 mov     [ebx+354h], eax

Загрузчик A0_Load_Pcx8_sub_55AA10 возвращает либо ресурс из дерева, либо
новый Pcx8, загруженный из файла/LOD. Если ресурс не найден в исходном
имени, код пытается пройти через запасной ресурс; при полном провале
может вернуть 0:

    .text:0055ABFA                 push    ebx             ; _GameResource_ *
    .text:0055AC05                 call    A0_ResourceError_NotDef_sub_559750
    .text:0055AC7F                 push    offset stru_64109C ; _GameResource_ *
    .text:0055AC8E                 call    A0_ResourceError_NotDef_sub_559750
    .text:0055AC93                 xor     eax, eax
    .text:0055ACA5                 retn

Есть и более точное совпадение с текущим вылетом: конструктор Pcx8 из
файла сначала зануляет поля, включая буфер [+0x30], затем вызывает
чтение файла. Если ранний разбор PCX возвращает ошибку, функция выходит
с кодом 1, но объект остаётся созданным с нулевым буфером:

    .text:0044F5D9                 mov     [ebx+1Ch], eax
    .text:0044F5DC                 mov     [ebx+20h], eax
    .text:0044F5DF                 mov     [ebx+24h], eax
    .text:0044F5E2                 mov     [ebx+28h], eax
    .text:0044F5E5                 mov     [ebx+2Ch], eax
    .text:0044F5E8                 mov     [ebx+30h], eax
    .text:0044F67E                 call    A0_Pcx8_ReadFromFile_sub_44F710
    .text:0044F684                 mov     eax, ebx

Внутри чтения PCX ранний путь ошибки действительно не выделяет буфер:

    .text:0044F721                 push    esi             ; a2
    .text:0044F722                 call    sub_604860
    .text:0044F727                 test    eax, eax
    .text:0044F729                 jz      short loc_44F738
    .text:0044F72B                 pop     esi
    .text:0044F72C                 mov     eax, 1
    .text:0044F735                 retn    1Ch

А нормальный путь выделяет буфер и записывает его в [+0x30]:

    .text:0044F738                 mov     eax, [ebp+var_18]
    .text:0044F73B                 mov     ecx, [ebp+var_1C]
    .text:0044F73E                 mov     [ebx+28h], eax
    .text:0044F745                 mov     [ebx+24h], ecx
    .text:0044F748                 mov     [ebx+2Ch], ecx
    .text:0044F74B                 mov     [ebx+20h], eax
    .text:0044F74E                 mov     [ebx+1Ch], eax
    .text:0044F751                 call    ??2@YAPAXI@Z    ; operator new(uint)
    .text:0044F759                 mov     [ebx+30h], eax

Итого поставщик данных теперь сужен до двух проверяемых веток:

1) индекс портрета из заголовка карты/выбранного героя выходит за
   допустимый диапазон 0..163, потому что проверяется только -1;
2) индекс допустим, но соответствующий HPS*.PCX или hpsrand.pcx загружен
   как Pcx8-объект с нулевым буфером после ошибки чтения/ресурса.

Текстовый лог не содержит значение port_id_v68 и содержимое Pcx8-объекта,
поэтому разделить эти две ветки без минидампа нельзя. Но в обоих случаях
первичный дефект находится не в HD_TC2, а в поставщике Pcx8 для диалога
выбора карты: перед вызовом рисовалки отсутствует проверка диапазона
портрета и валидности буфера изображения.

7. ПЕРВОПРИЧИНА С ДОКАЗАТЕЛЬСТВОМ
---------------------------------
Функция HD_TC2 sub_20066E0 зарегистрирована как замена исходной функции
SoD h3hota.exe:0044F940:

    .text:02009C2E                 push    offset sub_20066E0
    .text:02009C33                 push    1
    .text:02009C35                 push    0
    .text:02009C37                 mov     ecx, [eax]
    .text:02009C39                 push    1
    .text:02009C3B                 push    44F940h
    .text:02009C40                 push    eax
    .text:02009C41                 call    dword ptr [ecx+3Ch]

Оригинальная функция SoD A0_Pcx8_Draw_sub_44F940 также читает источник без
проверки нулевой базы. В ветке со специальными цветами:

    .text:0044F9B8                 mov     esi, [edi+30h]
    .text:0044F9BB                 add     edx, esi
    .text:0044F9C0                 add     edx, esi
    .text:0044F9C2                 mov     esi, edx
    .text:0044F9EF                 mov     dl, [ecx]
    .text:0044F9F1                 inc     ecx
    .text:0044F9F7                 jz      short loc_44FA0A

В ветке без специальной прозрачности:

    .text:0044FA3C                 test    ecx, ecx
    .text:0044FA3E                 mov     edx, esi
    .text:0044FA47                 mov     bl, [edx]
    .text:0044FA49                 inc     edx
    .text:0044FA4B                 mov     bx, [edi+ebx*2+50h]
    .text:0044FA50                 mov     [eax-2], bx

У оригинала нет проверки, что [Pcx8+0x30] не равен 0. Порт HD_TC2 делает
то же самое, только пишет 32-битный цвет вместо 16-битного. Это означает,
что отсутствие проверки источника в самой рисовалке - не самостоятельный
дефект HD_TC2, а унаследованный контракт функции.

Сравнение с соседней функцией HD_TC2 sub_20065F0 дает тот же результат:
она тоже читает источник по рассчитанному указателю без проверки базы:

    .text:02006690                 movzx   ecx, byte ptr [eax+1]
    .text:0200669D                 movzx   ecx, byte ptr [eax-3]
    .text:020066A4                 movzx   ecx, byte ptr [eax-4]

Итоговый механизм:

    1) Вызывающий код передает Pcx8-структуру в порт рисовалки.
    2) В этой структуре или в аргументах рассчитанный источник строки
       становится равен 0.
    3) sub_20066E0 доверяет источнику, как и исходная функция SoD.
    4) Первое чтение byte ptr [EDX] падает, потому что EDX = 0.

По статическому логу не видно, кто именно занулил или недонаполнил Pcx8.
Для этого нужен точный листинг вызывающей версии HD_HOTA.dll и, лучше,
полный дамп памяти с полями Pcx8 в момент вызова.

8. УСЛОВИЯ, СПРОВОЦИРОВАВШИЕ СБОЙ
---------------------------------
Доказанные условия по логам и стеку:

    HotA version = 1.8.0 во всех семи вылетах.
    Место сбоя = HD_TC2.dll+0x6805 во всех семи вылетах.
    Активен 32-битный подмодуль отрисовки HD_TC2.dll.
    OpenGL.Upscaling = 2 во всех семи вылетах.
    Стек проходит через перерисовку диалога выбора карты.

Состояние, которое меняется и не выглядит обязательным:

    Network Game = 1 только в двух из семи вылетов.
    Graphics.Threads принимает 2, 6, 8, 12 и 16.
    Graphics.Mode принимает 4 и 5.
    Map file различается между логами.

Показаний игрока нет. Поэтому сценарий из логов восстанавливается по
цепочке вызовов, а не по пользовательским комментариям.

Поля Some ingame values цитируются как состояние лога. Их нельзя
использовать как доказательство, что вылет связан с конкретной картой
или активным днем партии: стек указывает на интерфейс выбора карты.

9. ДОСТОВЕРНОСТЬ И ОГРАНИЧЕНИЯ
------------------------------
Достоверность сигнатуры высокая: семь уникальных логов разных версий
HD-мода имеют один адрес сбоя, одну операцию чтения и одинаковые ключевые
регистры EDX = 0 и EDI = 0.

Достоверность разбора инструкции высокая: адресная проверка сходится
не через поле структуры, а напрямую через регистр источника:

    EDX == 0x00000000 == адрес чтения из лога.

Достоверность утверждения о контракте рисовалки высокая: функция HD_TC2
зарегистрирована как замена h3hota.exe:0044F940, а исходная функция SoD
и соседний порт также читают источник без проверки базы.

Полезность лога для исправления средняя. Лог хорошо показывает симптом:
нулевой Pcx8-источник пришел в рисовалку. Но точечный корень выше по
цепочке вызовов не локализован: отсутствуют точные листинги HD_HOTA.dll
для части версий серии, а текстовый лог не содержит памяти объекта Pcx8.
Поэтому исправление в самой инструкции HD_TC2 было бы только скрытием
симптома; настоящий поставщик нулевой поверхности остается выше по стеку.

Ограничения по листингам:

    HD_TC2.dll.txt есть и используется для точки сбоя.
    HotA.dll-180.txt соответствует HotA 1.8.0.
    h3hota.exe.txt используется как исходная функция SoD.
    HD_HOTA.dll точной версии нужен для трассировки поставщика.

Для полной трассировки нужны листинги:

    Модули/HD_HOTA.dll-5.7 RC6.txt, size 0x0119B000
    Модули/HD_HOTA.dll-5.7 RC10.txt, size 0x0119B000
    Модули/HD_HOTA.dll-5.7 RC12.txt, size 0x0119B000

Для версий 5.7 RC11, 5.7 R10 и 5.7 R17 в проекте есть близкие или точные
листинги HD_HOTA.dll, но отчет не переносит по ним вывод на отсутствующие
RC6, RC10 и RC12.

10. ПРИЗНАКИ ПОИСКА ВЫЛЕТА В СТАРЫХ ВЕРСИЯХ HotA / В SoD
---------------------------------------------------------
Стабильные признаки по контексту:

    Module: HD_TC2.dll
    Address: HD_TC2.dll+0x6805 или соседнее смещение той же функции
    Information: read of address 0x00000000
    EDX = 0x00000000
    EDI = 0x00000000
    EAX и ESI содержат ширину строки, часто 0x30

Стабильная сигнатура регистрации в подмодуле HD_TC2:

    push offset sub_20066E0
    push 44F940h
    call dword ptr [ecx+3Ch]

Стабильная сигнатура падающего кода:

    расчет source_row из [this+2Ch], [this+30h] и аргументов;
    mov edx, edi;
    movzx eax, byte ptr [edx];
    mov eax, [palette+eax*4];
    запись 32-битного цвета в приемник.

Сигнатура исходной функции SoD:

    h3hota.exe A0_Pcx8_Draw_sub_44F940;
    чтение [edx] или [ecx] из исходной Pcx8-поверхности;
    отсутствие проверки [Pcx8+0x30] перед чтением.

Для поиска родственных отчетов полезны также нижние кадры стека:

    A0_MapAddOptions_Player_Draw_sub_58D7E0
    A0_MapSelectDlg_MapList_Redraw_sub_584820
    A0_MapSelectDlg_Run_sub_584EC0

11. ВОСПРОИЗВЕДЕНИЕ ВЫЛЕТА
--------------------------
Гарантированных шагов вида "нажать кнопку X - получить вылет" по этим
логам нет. Комментариев игроков нет, а источник нулевой Pcx8-поверхности
находится выше точки сбоя. Поэтому сценарий воспроизведения вероятностный
и нацелен на повтор перерисовки того же интерфейсного пути.

Роль 1 - тестер за игрой:

    1) Запустить HotA 1.8.0 с HD-модом 5.7 из диапазона логов.
    2) Использовать режим отрисовки, при котором загружен HD_TC2.dll.
    3) Оставить OpenGL.Upscaling = 2.
    4) Открывать диалог выбора карты и список сценариев.
    5) Переключать карты, параметры игроков и элементы списка, вызывая
       частую перерисовку блока параметров игрока.
    6) Повторять прогон на разных картах из логов и на пустом новом
       профиле, фиксируя только сам факт нового вылета и его адрес.

Роль 2 - подтверждение причины с отладчиком или полным дампом:

    1) Поставить остановку на runtime-адрес HD_TC2.dll base + 0x6805.
    2) На остановке проверить регистры EDX, EDI, EAX, ESI.
    3) Для той же сигнатуры ожидается EDX = 0 и чтение адреса 0.
    4) Найти объект Pcx8, который был this для sub_20066E0, и проверить
       поля [this+0x30] и [this+0x2C].
    5) Подняться на один-два уровня выше и определить, какой код создал
       или передал Pcx8 с нулевой базой изображения.

Диагностический дифференциальный прогон:

    A) Повторить тот же интерфейсный путь в режиме, где не используется
       HD_TC2.dll. Если входные данные те же, адрес сбоя должен либо
       исчезнуть, либо сместиться в исходную функцию h3hota.exe:0044F940.
    B) Повторить путь с полным дампом памяти и сверить, что новый лог
       сохраняет инварианты EDX = 0, Information = read 0x00000000,
       стек через MapSelectDlg.

Цель этих прогонов - не обход вылета, а подтверждение, что один и тот же
нулевой Pcx8-источник доходит до контрактной рисовалки в разных режимах
отрисовки.

ВЫВОД
-----
HD_TC2.dll+0x6805 падает не из-за самостоятельной ошибки попиксельного
цикла, а потому что в порт Pcx8_Draw приходит нулевой источник. HD_TC2
повторяет контракт исходной функции SoD h3hota.exe:0044F940: источник
должен быть уже валиден на входе. Корень нужно искать выше по цепочке
перерисовки диалога выбора карты, в коде, который создает или передает
Pcx8-поверхность с нулевой базой изображения.
