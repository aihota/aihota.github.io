Краткое резюме¶

Игра падает в меню выбора сценария, когда открывается окно настроек карты («дополнительные условия»). HD-мод в этот момент обходит все ячейки карты и для каждой читает поле по смещению 0x1E. Указатель на ячейку считается как buffer + 38*(x + width*y + width*width*z); если индекс выходит за реально выделенный буфер ячеек, чтение попадает на неотображённую страницу и игра падает.

7 уникальных вылетов на чтении по адресу HD_HOTA.dll+0x97CAD, HotA 1.8.0, HD-мод 5.7 R11/R12. Во всех дампах faulting_addr = EDI + 0x1E, EAX == EDI, EBX == ESI (малый индекс цикла 0x27..0xD2). Адрес сбоя во всех дампах попадает на самый конец страницы (EDI оканчивается на ...FF0..FFE или ...000), а +0x1E переходит на следующую неотображённую — это линейный выход за конец буфера ячеек, а не случайный мусор.

Стек всех дампов поднимается к A0_WndMgr_Run_MapSelectDlg / A0_MapSelectDlg_Run (он же A0_MapAddOptions_Dialog_Run) / A0_DialogFunction — это меню выбора сценария ПЕРЕД началом игры. Поля Day и Map file в логе — остаточные значения из настроек и к стадии партии отношения не имеют (игра ещё не запущена).

Сопоставление адресов¶

Листинг: HD_HOTA.dll-5.7 R12.txt, Imagebase = 0x1000000 Формула: IDA_addr = offset + imagebase = 0x97CAD + 0x1000000 = 0x1097CAD Адрес: .text:01097CAD cmp dword ptr [edi+1Eh], 22h Проверка: EDI + 0x1E = faulting_addr (напр. 0x1BB37FF0 + 0x1E = 0x1BB3800E). Совпадает во всех 7 дампах.

Минидамп hd_crash.dmp (родственный лог +0x9701D, HD 5.7 R19) разобран и подтвердил, что CONTEXT в txt-логах не искажён: регистры и EIP в дампе совпадают с текстовым логом байт-в-байт.

Точка сбоя — разбор инструкций¶

.text:01097C97  push    edx              ; координата (предел по Z)
.text:01097C98  push    ecx              ; координата
.text:01097C99  push    eax              ; координата
.text:01097C9A  mov     eax, 6992B8h     ; глобал объекта карты (AdvMgr)
.text:01097C9F  mov     ecx, [eax]
.text:01097CA1  mov     eax, 4086D0h     ; A0_GameMap_GetCell
.text:01097CA6  mov     ecx, [ecx+5Ch]   ; this = (*0x6992B8)->field_5C
.text:01097CA9  call    eax              ; EAX = GetCell(this, x, y, z)
.text:01097CAB  mov     edi, eax         ; EDI = указатель на ячейку
.text:01097CAD  cmp     dword ptr [edi+1Eh], 22h  ; КРАХ: чтение [EDI+0x1E]

GetCell (h3hota.exe:0x4086D0) — чистая арифметика без проверки границ: return buffer + 38 * (x + width*y + width*width*z) где buffer = this→field_0xD0, width = this→field_0xD4, 38 = размер структуры ячейки (0x38 байт). Функция всегда возвращает указатель (не NULL); если индекс за пределами буфера — указатель невалиден.

После вызова EDI не сверяется с границей буфера; сразу читается поле [edi+1Eh] (поле в пределах структуры ячейки 0x00..0x37) и сравнивается с 0x22. Если EDI указывает на последнюю ячейку буфера у границы страницы — чтение +0x1E переходит на неотображённую страницу.

Регистровые инварианты¶

  • EAX == EDI == указатель на ячейку (возврат GetCell через EAX).
  • faulting_addr = EDI + 0x1E (7/7). EDI во всех дампах оканчивается у границы страницы (...FF0/FF6/FF2/FFE или ...000); +0x1E выходит на следующую неотображённую страницу. Это сигнатура линейного перебега за конец выделенного буфера ячеек, а не случайного указателя.
  • EBX == ESI: значение малое (0x27..0xD2) и одинаковое в двух регистрах — это координата/индекс внутреннего цикла обхода, не ось, переставленная местами. Значение различается между картами (зависит от размеров).
  • EBP — ESP = -60: фиксированный фрейм одной и той же функции.

Восстановленная цепочка вызовов¶

Падающая функция — sub_1097390. Она перебирает ячейки карты вложенными циклами и для каждой через GetCell проверяет/переписывает поле типа ячейки (значение 0x22 меняется далее по коду на 0x62). Границы циклов берутся из глобалов размеров карты: A0_Map_X_Size_dword_6783C8 (ширина), A0_Map_Y_Size_dword_6783CC (высота), предел по Z — из поля [GameMgr+1FC48h].

Первопричина¶

Потребитель (цикл обхода в sub_1097390) перебирает ячейки по ГЛОБАЛЬНЫМ размерам карты A0_Map_X_Size / A0_Map_Y_Size и пределу по Z. Поставщик (GetCell) вычисляет адрес ячейки по СОБСТВЕННЫМ полям объекта карты: buffer = AdvMgr→field_5C→field_0xD0, width = ...→field_0xD4.

Если эти две границы расходятся — глобальные размеры выбранной для предпросмотра карты больше, чем реально выделенный буфер ячеек объекта карты (в меню полноценная игровая карта ещё не загружена, объект может держать буфер прежней/дефолтной/меньшей геометрии) — то индекс x + width*y + width*width*z выходит за конец буфера, и GetCell отдаёт указатель за выделенной областью. Чтение поля ячейки +0x1E у последней страницы буфера падает.

Сигнатура подтверждает именно перебег, а не порчу: EDI у всех дампов на конце страницы, индекс цикла малый и закономерный, faulting линейно следует за концом буфера. Несколько разных карт дают один адрес сбоя в одной функции — значит дело не в конкретном файле карты, а в установке буфера/размеров в контексте меню.

Что НЕ доказано по статике: какая именно из границ устаревшая (глобальные размеры опережают буфер объекта, или буфер не переаллоцирован под выбранную карту). Минидамп hd_crash.dmp — без кучи, поля buffer/width объекта карты из него недоступны. Для окончательного ответа нужен полный дамп (с кучей), где видно AdvMgr→field_5C→field_0xD0/field_0xD4 и глобалы A0_Map_X_Size/Y_Size в момент сбоя.

Условия, спровоцировавшие сбой¶

  • Контекст — меню выбора сценария, окно «дополнительные условия» карты (по стеку: A0_WndMgr_Run_MapSelectDlg, A0_MapSelectDlg_Run / A0_MapAddOptions_Dialog_Run). Игра ещё не начата (Network Game = 0).
  • Day = 1 M 11 и Map file во всех дампах — остаточные значения настроек, а не стадия партии: их нельзя трактовать как «281-й день игры».
  • 5 разных карт при одном адресе сбоя — баг не привязан к одному файлу карты; срабатывает на разных картах при открытии их настроек.
  • Блок Comment в дампах серии содержит только служебный автотекст (показаний игрока нет). Родственный лог +0x9701D (HD 5.7 R19) содержит показание игрока, прямо указывающее на этот же контекст (см. раздел 9).

Достоверность и ограничения¶

Подтверждено / высокая достоверность
  • Достоверность сигнатуры: ВЫСОКАЯ (7 дампов, единый адрес, единые регистровые инварианты, единый верхний путь стека).
  • Достоверность контекста (меню выбора сценария): ВЫСОКАЯ — в стеке 4 из 7 дампов явно присутствуют кадры A0_WndMgr_Run_MapSelectDlg и A0_MapSelectDlg_Run; остальные 3 имеют тот же верхний путь HD_HOTA (усечён снизу). Дополнительно подтверждается родственным логом +0x9701D, где игрок дословно написал: «В меню дополнительные условия (при выборе сценария) игра вылетает уже не первый раз» — это показание, а не доказанный кодом факт, но оно совпадает со стеком.
  • Достоверность механизма: СРЕДНЯЯ. Перебег за конец буфера доказан сигнатурой; конкретный источник рассогласования границ (глобальные размеры карты vs буфер объекта) по статике не разрешается — минидамп без кучи, нужен полный дамп.
  • Минидамп hd_crash.dmp разобран: подтвердил подлинность CONTEXT (регистры/EIP совпали с txt байт-в-байт), но память кучи в нём отсутствует (объект карты, buffer, width недоступны).
  • Родственный адрес +0x9701D (HD 5.7 R19, функция sub_1096780) — тот же дефект в соседней функции обхода ячеек того же меню; это другой адрес, по нему отдельный отчёт.

Признаки поиска в старых версиях HotA / в SoD¶

Контекст-инварианты:

  • EAX == EDI == указатель на ячейку после GetCell
  • faulting_addr = EDI + 0x1E (чтение [edi+1Eh])
  • EDI на конце страницы (линейный перебег за конец буфера ячеек)
  • EBX == ESI == малый индекс цикла (0x27..0xD2)
  • EBP — ESP = -60

Текстовые сигнатуры дизассемблера:

  • cmp dword ptr [edi+1Eh], 22h (инструкция сбоя)
  • mov eax, 4086D0h; mov ecx, [ecx+5Ch]; call eax; mov edi, eax (вызов GetCell с адресацией через AdvMgr→field_5C)
  • границы цикла из A0_Map_X_Size_dword_6783C8 / A0_Map_Y_Size_dword_6783CC

Стабильные адреса h3hota.exe:

  • A0_GameMap_GetCell_sub_4086D0 (0x4086D0)
  • A0_Map_X_Size_dword_6783C8, A0_Map_Y_Size_dword_6783CC (размеры карты)
  • A0_AdvMgr_dword_6992B8 (объект приключенческой карты)
  • A0_WndMgr_Run_MapSelectDlg_sub_602D60, A0_MapSelectDlg_Run_sub_584EC0 (он же A0_MapAddOptions_Dialog_Run_sub_584EC0) — меню сценария

Воспроизведение вылета¶

Контекст воспроизведения локализован: меню выбора сценария, открытие окна настроек/«дополнительных условий» карты. Показание игрока (из родственного лога +0x9701D) подтверждает направление: «вылетает в меню дополнительные условия при выборе сценария, не первый раз».

Сценарий: 1) HotA 1.8.0 + HD-мод 5.7 (R11/R12; родственный путь — и на R19). 2) Главное меню → Одиночная/Сетевая игра → выбор сценария. 3) Открыть окно настроек карты («дополнительные условия») для карт с подземельем; перебирать карты в списке, открывая их настройки. 4) Ожидать вылет на чтении ячейки при обходе карты для предпросмотра.

Поскольку вылет наблюдался на 5 разных картах, целиться надо не в один файл, а в набор карт: прогон по списку сценариев с открытием окна настроек у каждого, разные размеры карт и наличие подземелья.

Дифференциальная проверка причины (для подтверждения механизма, не как совет): снять в полном дампе значения A0_Map_X_Size/A0_Map_Y_Size и AdvMgr→field_5C→field_0xD4 (ширина буфера) в момент сбоя — если размеры карты превышают ширину/ёмкость буфера, рассогласование границ подтверждено.

Что ловить в полном дампе: buffer (field_0xD0) и width (field_0xD4) объекта карты, глобалы размеров A0_Map_X_Size/Y_Size; регистровый инвариант EAX==EDI, EBX==ESI, EBP-ESP=-60, EDI на конце страницы.

Вывод¶

Виновник — HD-мод (HD_HOTA.dll): обход ячеек карты sub_1097390, запускаемый при открытии окна настроек карты в меню выбора сценария (A0_MapSelectDlg_Run / A0_MapAddOptions_Dialog_Run). Цикл перебирает ячейки по глобальным размерам карты, а A0_GameMap_GetCell адресует по собственным буферу и ширине объекта приключенческой карты; при несовпадении этих границ вычисленный указатель ячейки уходит за конец буфера на неотображённую страницу, и чтение поля ячейки +0x1E приводит к нарушению доступа. Падает на разных картах, то есть причина в установке буфера/размеров в контексте меню, а не в конкретном файле карты. Какая из границ устаревшая — по статике не разрешается, нужен полный дамп с кучей.
