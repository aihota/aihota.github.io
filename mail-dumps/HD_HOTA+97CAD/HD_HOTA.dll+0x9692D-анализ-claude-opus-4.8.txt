АНАЛИЗ ВЫЛЕТА HD_HOTA.dll+0x9692D (предпросмотр карты в меню выбора
сценария)
========================================================================

Модель: Claude Opus 4.8
Дата: 16.06.2026
Заключение: Нарушение доступа на чтении [edi+1Eh] при обходе ячеек карты
в меню выбора сценария (раздел «дополнительные опции», в т.ч. при запуске
случайной карты). Цикл перебирает ячейки по глобальным размерам карты, а
A0_GameMap_GetCell адресует по собственным буферу/ширине объекта карты;
при их несовпадении вычисленный указатель ячейки уходит за конец буфера
на неотображённую страницу. Тот же дефект, что HD_HOTA.dll+0x97CAD и
+0x9701D, в другой версии HD-мода (адрес функции смещён версией).

КРАТКОЕ РЕЗЮМЕ
--------------
Игра падает в меню выбора сценария при открытии раздела «дополнительные
опции» (по показанию игрока — при попытке начать случайную карту). HD-мод
в этот момент обходит все ячейки карты и для каждой читает поле по
смещению 0x1E. Указатель на ячейку считается как
buffer + 38*(x + width*y + width*width*z); если индекс выходит за реально
выделенный буфер ячеек, чтение попадает на неотображённую страницу и игра
падает.

3 уникальных вылета на чтении по адресу HD_HOTA.dll+0x9692D, HotA 1.8.0,
HD-мод 5.7 R9/R10. Во всех faulting_addr = EDI + 0x1E, EAX == EDI,
EBX == ESI (малый индекс цикла 0x3D..0x71). Адрес сбоя у границы страницы
(EDI оканчивается на ...FF2/FF4/002), +0x1E переходит на следующую
неотображённую — линейный выход за конец буфера ячеек.

Стек поднимается к A0_WndMgr_Run_MapSelectDlg / A0_MapSelectDlg_Run (он же
A0_MapAddOptions_Dialog_Run) / A0_DialogFunction — меню выбора сценария
ПЕРЕД началом игры. Поля Day и Map file в логе — остаточные значения
настроек, к стадии партии отношения не имеют (игра ещё не запущена).

3. Сопоставление адресов
-------------------------
Листинг: HD_HOTA.dll-5.7 R9.txt, Imagebase = 0x1000000
Формула: IDA_addr = offset + imagebase = 0x9692D + 0x1000000 = 0x109692D
Адрес: .text:0109692D  cmp dword ptr [edi+1Eh], 22h
Проверка: EDI + 0x1E = faulting_addr (напр. 0x0231EFF2 + 0x1E =
0x0231F010). Совпадает во всех дампах.
Версия R10 (дамп 3) — листинг HD_HOTA.dll-5.7 R10.txt тоже доступен;
адрес функции в R10 совпадает с R9 по тому же offset.

4. Точка сбоя
-------------
  .text:01096917  push    edx              ; координата (предел по Z)
  .text:01096918  push    ecx              ; координата
  .text:01096919  push    eax              ; координата
  .text:0109691A  mov     eax, 6992B8h     ; глобал объекта карты (AdvMgr)
  .text:0109691F  mov     ecx, [eax]
  .text:01096921  mov     eax, 4086D0h     ; A0_GameMap_GetCell
  .text:01096926  mov     ecx, [ecx+5Ch]   ; this = (*0x6992B8)->field_5C
  .text:01096929  call    eax              ; EAX = GetCell(this, x, y, z)
  .text:0109692B  mov     edi, eax         ; EDI = указатель на ячейку
  .text:0109692D  cmp     dword ptr [edi+1Eh], 22h  ; КРАХ: чтение [EDI+0x1E]
  .text:01096937  imul    esi, [edi], 492h ; далее — индексация по ID ячейки

GetCell (h3hota.exe:0x4086D0) — чистая арифметика без проверки границ:
  return buffer + 38 * (x + width*y + width*width*z)
где buffer = this->field_0xD0, width = this->field_0xD4, 38 = размер
структуры ячейки (0x38 байт). Функция всегда возвращает указатель;
если индекс за пределами буфера — указатель невалиден.

После вызова EDI не сверяется с границей буфера; сразу читается поле
[edi+1Eh] (поле в пределах структуры ячейки 0x00..0x37) и сравнивается
с 0x22. Если EDI указывает на последнюю ячейку буфера у границы страницы
— чтение +0x1E переходит на неотображённую страницу.

5. Регистровые инварианты
-------------------------
- EAX == EDI == указатель на ячейку (возврат GetCell через EAX).
- faulting_addr = EDI + 0x1E (3/3). EDI во всех дампах оканчивается у
  границы страницы; +0x1E выходит на следующую неотображённую страницу.
  Это сигнатура линейного перебега за конец буфера ячеек.
- EBX == ESI: значение малое (0x3D..0x71), одинаковое в двух регистрах —
  координата/индекс внутреннего цикла обхода; различается между картами.
- EBP - ESP = 0x3C (фиксированный фрейм одной и той же функции).

6. Цепочка вызовов и архитектура
---------------------------------
Падающая функция — sub_1096010. Она перебирает ячейки карты вложенными
циклами и для каждой через GetCell проверяет/переписывает поле типа
ячейки (значение 0x22). Границы циклов берутся из глобалов размеров карты:
  A0_Map_X_Size_dword_6783C8 (ширина), A0_Map_Y_Size_dword_6783CC (высота),
предел по Z — из поля [GameMgr+1FC48h].

Восстановленный путь (по стеку дампа 1):

  A0_WndMgr_Run_MapSelectDlg (h3hota 0x602D60)
    -> A0_MapSelectDlg_Run / A0_MapAddOptions_Dialog_Run (0x584EC0)
       -> A0_DialogFunction (0x5FFAC0)
          -> [мост HD-мода и трамплины patcher_x86]
             -> HD_HOTA.dll+0x158F.. / +0x150.. / +0x8E50 / +0x90D8
                -> sub_1096010 (HD_HOTA.dll+0x96010) — обход ячеек
                   -> GetCell -> чтение [edi+1Eh]  (КРАХ)

Это родственная функция тому же дефекту в +0x97CAD (sub_1097390, HD R12)
и +0x9701D (sub_1096780, HD R19): идентичный фрагмент
call GetCell; mov edi/ebx,eax; cmp [reg+1Eh],22h; imul esi,[reg],492h.
Адрес функции смещается между версиями HD-мода, дефект один.

7. Первопричина с доказательством
---------------------------------
Потребитель (цикл обхода в sub_1096010) перебирает ячейки по ГЛОБАЛЬНЫМ
размерам карты A0_Map_X_Size / A0_Map_Y_Size и пределу по Z. Поставщик
(GetCell) вычисляет адрес ячейки по СОБСТВЕННЫМ полям объекта карты:
buffer = AdvMgr->field_5C->field_0xD0, width = ...->field_0xD4.

Если эти границы расходятся — глобальные размеры выбранной для
предпросмотра карты больше, чем реально выделенный буфер ячеек объекта
карты (в меню полноценная игровая карта ещё не загружена, объект может
держать буфер прежней/дефолтной/меньшей геометрии) — то индекс выходит за
конец буфера, и GetCell отдаёт указатель за выделенной областью. Чтение
поля ячейки +0x1E у последней страницы буфера падает.

Сигнатура подтверждает перебег, а не порчу: EDI у всех дампов на конце
страницы, индекс цикла малый и закономерный, faulting линейно следует за
концом буфера. Несколько разных карт и две версии HD-мода дают один адрес
сбоя в одной функции — дело не в файле карты, а в установке
буфера/размеров в контексте меню.

Что НЕ доказано по статике: какая именно из границ устаревшая. Для
окончательного ответа нужен полный дамп (с кучей), где видно
AdvMgr->field_5C->field_0xD0/field_0xD4 и глобалы A0_Map_X_Size/Y_Size
в момент сбоя. Имеющиеся логи — текстовые, без дампа памяти.

  A0_WndMgr_Run_MapSelectDlg, A0_MapSelectDlg_Run /

  привязан к одному файлу карты.

9. Достоверность и ограничения
-------------------------------
- Достоверность сигнатуры: ВЫСОКАЯ (3 уникальных дампа, единый адрес,
  единые регистровые инварианты, единый верхний путь стека; плюс
  родственные адреса +0x97CAD и +0x9701D с тем же кодом).
- Достоверность контекста (меню выбора сценария): ВЫСОКАЯ — в стеке явно
  присутствуют A0_WndMgr_Run_MapSelectDlg и A0_MapSelectDlg_Run, плюс
  дословное показание игрока про «дополнительные опции / случайная карта».
- Достоверность механизма: СРЕДНЯЯ. Перебег за конец буфера доказан
  сигнатурой; конкретный источник рассогласования границ (глобальные
  размеры карты vs буфер объекта) по статике не разрешается — нужен
  полный дамп с кучей.
- Дамп памяти по этому адресу не предоставлен (только текстовые логи);
  подлинность CONTEXT косвенно подтверждается родственным минидампом
  +0x9701D, где регистры/EIP совпали с txt байт-в-байт.
- Смешанные версии HD-мода в серии (R9, R10): листинги обеих доступны,
  адрес функции по offset совпадает; диагноз не зависит от конкретной
  версии HD-мода.

10. Признаки поиска вылета в старых версиях HotA / в SoD
---------------------------------------------------------
Контекст-инварианты:
  - EAX == EDI == указатель на ячейку после GetCell
  - faulting_addr = EDI + 0x1E (чтение [edi+1Eh])
  - EDI на конце страницы (линейный перебег за конец буфера ячеек)
  - EBX == ESI == малый индекс цикла (0x3D..0x71)

Текстовые сигнатуры дизассемблера:
  - cmp dword ptr [edi+1Eh], 22h; imul esi, [edi], 492h
  - mov eax, 4086D0h; mov ecx, [ecx+5Ch]; call eax; mov edi, eax
  - границы цикла из A0_Map_X_Size_dword_6783C8 / A0_Map_Y_Size_dword_6783CC

Стабильные адреса h3hota.exe:
  - A0_GameMap_GetCell_sub_4086D0 (0x4086D0)
  - A0_Map_X_Size_dword_6783C8, A0_Map_Y_Size_dword_6783CC
  - A0_AdvMgr_dword_6992B8 (объект приключенческой карты)
  - A0_WndMgr_Run_MapSelectDlg_sub_602D60, A0_MapSelectDlg_Run_sub_584EC0
    (он же A0_MapAddOptions_Dialog_Run_sub_584EC0)

Родственные адреса того же дефекта: HD_HOTA.dll+0x97CAD (sub_1097390,
HD R11/R12), HD_HOTA.dll+0x9701D (sub_1096780, HD R19).

11. Воспроизведение вылета
---------------------------
Контекст локализован: меню выбора сценария, раздел «дополнительные опции»,
в т.ч. запуск случайной карты. Показание игрока подтверждает направление.

Сценарий:
  1) HotA 1.8.0 + HD-мод 5.7 (R9/R10; родственные пути — R11/R12/R19).
  2) Главное меню -> выбор сценария / случайная карта.
  3) Открыть раздел «дополнительные опции»; для случайной карты — задать
     параметры и нажать старт; перебирать карты/шаблоны в списке.
  4) Ожидать вылет на чтении ячейки при обходе карты для предпросмотра.

Вылет наблюдался на нескольких картах и двух версиях HD-мода, целиться
надо не в один файл, а в набор: прогон по списку сценариев/шаблонов
случайной карты с открытием «дополнительных опций», разные размеры карт.

Дифференциальная проверка причины (для подтверждения механизма, не как
совет): снять в полном дампе A0_Map_X_Size/A0_Map_Y_Size и
AdvMgr->field_5C->field_0xD4 (ширина буфера) в момент сбоя — если размеры
карты превышают ёмкость буфера, рассогласование границ подтверждено.

Что ловить в полном дампе: buffer (field_0xD0) и width (field_0xD4)
объекта карты, глобалы размеров; регистровый инвариант EAX==EDI, EBX==ESI,
EDI на конце страницы.

ВЫВОД
-----
Виновник — HD-мод (HD_HOTA.dll): обход ячеек карты sub_1096010,
запускаемый при открытии раздела «дополнительные опции» в меню выбора
сценария (A0_MapSelectDlg_Run / A0_MapAddOptions_Dialog_Run), в том числе
при запуске случайной карты. Цикл перебирает ячейки по глобальным размерам
карты, а A0_GameMap_GetCell адресует по собственным буферу и ширине
объекта карты; при несовпадении границ вычисленный указатель ячейки уходит
за конец буфера на неотображённую страницу, и чтение поля ячейки +0x1E
приводит к нарушению доступа. Это тот же дефект, что HD_HOTA.dll+0x97CAD и
+0x9701D — родственные функции обхода ячеек того же меню в других версиях
HD-мода. Какая из границ устаревшая — по статике не разрешается, нужен
полный дамп с кучей.
