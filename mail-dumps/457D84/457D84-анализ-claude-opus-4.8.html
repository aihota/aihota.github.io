Визуализация вылета ¶

Внутри менеджера игры по смещению +0x1F879 лежит массив команд игроков -- 8 байт, по одному на слот (валидный индекс 0..7). Код читает этот массив дважды: по номеру слота (EDI, всегда в пределах) и по "выбранному игроку" (ECX). Перед вторым чтением проверяется только знак (id >= 0), но НЕ верхняя граница. "Выбранный игрок" содержит мусор, поэтому адрес base + ECX + 0x1F879 уходит далеко вправо за пределы массива и памяти.

BEAX = база
team[0..7]+0x1F879, 8 байт
Tteam[EDI] -- норма
...за массивом
Fteam[ECX] -- СБОЙ

чтение по "выбранному игроку" уходит далеко правее массива из 8 элементов

B база менеджера игры (EAX), выровнена по 64K -- валидна.
T штатное чтение team[EDI] по номеру слота (0..7) на 0x00457D7D -- проходит во всех вылетах.
F адрес сбоя base + ECX + 0x1F879, где ECX ("выбранный игрок") вне диапазона 0..7 -- проверена только нижняя граница.

Краткое резюме¶

При открытии/обновлении диалога выбора сценария кампании игра падает, пытаясь прочитать байт команды (альянса) "выбранного игрока" из таблицы заголовка карты. Номер выбранного игрока возвращает виртуальный метод объекта сценария кампании, после чего он используется как индекс в массив из 8 элементов. Перед использованием код проверяет только то, что номер не отрицательный (знак), но НЕ проверяет верхнюю границу. В аварийных логах этот номер содержит значение далеко за пределами 0..7, поэтому адрес чтения уходит за пределы структуры менеджера игры.

Тип исключения: EXCEPTION_ACCESS_VIOLATION, чтение (read). Где: h3hota.exe, адрес 0x00457D84, инструкция mov bl,[eax+ecx+1F879h]. Корневой регистр: ECX = "выбранный игрок" (вне диапазона) при валидной базе EAX.

Подробный разбор типов и аргумента -- в разделе 7.

Регистровые инварианты¶

Постоянные во всех 19 вылетах: EDI = 0x00000001 -- цикл всегда падает на слоте игрока №1. На слоте №0 флаг наличия (0x00457D64) отрицателен, слот пропускается; на слоте №1 флаг неотрицателен, путь доходит до сравнения команд и падает. EDX = 0xFFFFFF00 -- младший байт DL = 0: это team[слот №1], успешно прочитанный на 0x00457D7D перед сбоем (значит соседнее чтение по валидному индексу проходит). EBX = 0x00000072 -- рабочий байт (id элемента диалога; в функции lea ebx,[edi+71h] = 0x71 + слот).

Переменные: EAX -- база менеджера игры, выровнена по 64K (0x..0000); своя аллокация на каждой машине -- штатно. ECX -- "выбранный игрок": в 14 вылетах большое произвольное значение (0x0EEB9003 ... 0x7B5F1D82), в 5 вылетах структурное 0x00080004. Оба профиля -- значения, далеко выходящие за диапазон 0..7. Адрес чтения = EAX + ECX + 0x1F879 -- следствие выхода ECX за границу.

Контракт нарушен на величину, невозможную для исправного объекта: метод Get_PlayerID по контракту возвращает игрока в диапазоне байта (0..255, см. раздел 7), а в логах ECX -- сотни мегабайт либо структурный dword 0x00080004. Значит вернулось не валидное поле игрока, а целый dword из чужого/неверно интерпретированного объекта. Произвольность ECX между вылетами на одной и той же карте (test.h3m) подтверждает, что значение не является стабильной величиной из файла сценария.

Первопричина¶

Непосредственный дефект (точка сбоя): функция расстановки союзников и врагов берёт "выбранного игрока" виртуальным методом объекта сценария кампании и использует его как индекс в массив команд игроков (8 элементов) без проверки верхней границы. Из трёх проверок перед чтением (флаг наличия слота, знак номера слота, знак "выбранного игрока") ни одна не ограничивает "выбранного игрока" сверху значением 8.

Поставщик значения разрешён по листингу. Объект scenario→start_options (поле +0xA4) -- полиморфный _CampScenarioStartOptions_, который создаётся при чтении файла кампании (A0_CampScenario_Read_sub_487980) одним из трёх конкретных типов по тег-байту из файла: 1 → Bonus, 2 → Player, 3 → Hero, иначе start_options остаётся NULL. Слот таблицы виртуальных функций +0x20 во всех трёх типах -- это метод Get_PlayerID. Его реализации НЕ эквивалентны по диапазону результата:

Player (sub_4852C0)movsx eax, byte ptr [list+arg*2] -- знаковый байт, диапазон -128..127 (со знаковым клампом снизу);
Bonus (sub_484CB0)mov eax, [ecx+4] -- dword, записанный как (x & 0xFF), диапазон 0..255;
Hero (sub_485680)mov eax, [[ecx+8] + arg*8] -- ЦЕЛЫЙ dword по индексу arg, БЕЗ клампа.

Аргумент метода (arg) -- это camp_info.selected_bonus_ix (поле менеджера игры +0x1F468), считываемое на 0x00457D06 и тоже не проверяемое сверху.

Отсюда два уровня отсутствующих проверок верхней границы, оба на индексах, восстановленных из сохранённого состояния кампании: 1) scenario_ix (this+0x6C) -- выбор объекта scenarios_list[scenario_ix]; 2) selected_bonus_ix (camp_info +0x1F468) -- аргумент Get_PlayerID, используемый Hero-вариантом как [base + arg*8] и Player-вариантом как [list + arg*2]. Структурное значение ECX=0x00080004/0x00080006 (старшее слово 0x0008) согласуется именно с Hero-вариантом: возвращён целый dword, а не зажатый байт игрока. Этот dword функция sub_457CF0 принимает за номер игрока 0..7 и без верхней проверки кладёт в индекс таблицы команд.

Почему объект-поставщик оказывается негодным. Список сценариев и их объекты start_options НЕ сериализуются в сейв: при продолжении кампании из сохранения путь главного меню A0_MainMenu_sub_4EEA70 заново строит scenarios_list из файла .h3c вызовом A0_CampInfo_Load_sub_4883C0 (число сценариев = таблица A0_CampMapInfos[campaing_id].region_count). Из сейва восстанавливаются только индексы (scenario_ix, selected_bonus_ix). Если сохранённые индексы не соответствуют заново загруженному списку (другая версия или мод кампании, иной campaing_id, иное число сценариев), scenario[scenario_ix] и/или Get_PlayerID(selected_bonus_ix) читают за границей → возвращается мусор → чтение team[id] падает. Тот же паттерн "индекс из сейва → Get_PlayerID → индекс игрока без проверки" повторно встречается в A0_MainMenu_sub_4EEA70 (0x4EFB81, 0x4EFFCD), что доказывает системность контракта, а не единичную опечатку.

Сравнение с соседними путями (доказательство пропущенной проверки): в главном EXE около 120 чтений того же поля команды team[+0x1F879], и почти все парные сравнения берут оба индекса из циклов 0..7. Путь 0x00457D7D/0x00457D84 -- единственный, где один индекс приходит из виртуального метода, а не из ограниченного цикла; именно у него нет проверки верхней границы. Глобал SelectedPlayerID (0x00694E1C) пишется и читается только внутри sub_457CF0 -- извне его подменить нельзя, значение целиком задаёт виртуальный вызов 0x00457D1B.

Вывод по первопричине: место сбоя -- симптом отсутствующей проверки верхней границы индекса. Поставщик значения разрешён (Get_PlayerID полиморфного объекта start_options), и механизм понятен: два индекса из сохранённого состояния кампании используются без верхней границы поверх списка сценариев, заново загружаемого из .h3c. КОНКРЕТНЫЙ корраптор (какой именно фактор разводит сохранённые индексы с перезагруженным файлом -- версия/мод кампании, повреждение сейва) по статике не фиксируется: для него нужен сейв игрока или минидамп.

Условия, спровоцировавшие сбой¶

  • Экран выбора сценария кампании (одиночная игра): Game Type = 0, Network Game = 0, Day = 1, до начала партии.
  • Путь сбоя исполняется при открытии/смене сценария в диалоге кампании (вызов расстановки союзников/врагов из обработчика выбора сценария).
  • Падение детерминировано доходит до слота игрока №1 (EDI=1 во всех вылетах): слот №0 пропускается по флагу наличия.
  • Настройки графики/боя/памяти (Quick Combat, Memory.Mod, Graphics.*) на путь сбоя не влияют -- они различаются между вылетами при одном адресе.
  • Карты и пути установки разные; общий знаменатель -- сам экран выбора сценария кампании, а не конкретная карта или сборка HD-мода.
  • По показаниям игроков из SoD-серии того же адреса (раздел 10) сбой привязан к продолжению СОХРАНЁННОЙ кампании и клику на сценарий, что согласуется с механизмом рассинхрона индексов сейва и файла кампании.

Достоверность и ограничения¶

Верификация листинга: модуль сбоя -- h3hota.exe. Листинг Модули/h3hota.exe.txt: File Name h3hota.exe, базовый адрес образа 0x00400000 -- совпадает с логом; инструкция по 0x00457D84 (mov bl,[eax+ecx+1F879h]) совпадает байт-в-байт с местом сбоя. Маппинг прямой (адрес загрузки = базовый адрес образа). Достоверность точки сбоя и арифметики -- высокая: формула faulting = EAX + ECX + 0x1F879 сошлась во всех 19 вылетах.

Достоверность сигнатуры -- очень высокая: 19 вылетов, один адрес, один вызывающий код, устойчивые инварианты (EDI=1, EDX=0xFFFFFF00, EBX=0x72). Достоверность МЕХАНИЗМА -- высокая: поставщик значения разрешён по листингу (виртуальный Get_PlayerID полиморфного объекта start_options), доказаны два непроверенных индекса (scenario_ix, selected_bonus_ix) из сохранённого состояния кампании поверх списка, заново загружаемого из .h3c. Достоверность КОНКРЕТНОГО КОРРАПТОРА -- низкая: какой фактор разводит сохранённые индексы с перезагруженным файлом (версия/мод кампании, повреждение сейва), по статике не определить -- нужен сейв игрока или минидамп.

Полезность лога -- СРЕДНЯЯ: место сбоя и контракт поставщика локализованы точно, но первопричина (рассинхрон индексов сейва с перезагруженным списком сценариев) находится выше по стеку и по статике не сводится к одному фиксу в точке сбоя. Проверка верхней границы индекса в точке сбоя (и/или в Get_PlayerID) убрала бы падение, но это защита от симптома; полноценный фикс -- согласование сохранённых индексов с заново загружаемой кампанией. Само по себе отсутствие проверки границы полезность не повышает -- её повышает доказанное использование невалидного индекса, восстановленного из сохранённого состояния.

Кадры HD_HOTA.dll и трамплины patcher_x86 в стеке не маппятся в листинг (это драйвер интерфейса HD-мода и runtime-стабы хуков); на диагноз виновного модуля это не влияет. Рассинхрон версий HD-мода (5.7 R9/R12/R13) -- только в периферийном модуле; падающий модуль h3hota.exe постоянен, поэтому рассинхрон диагноз не блокирует.

Признаки поиска в старых версиях HotA / в SoD¶

Код места сбоя находится в главном модуле h3hota.exe (диалог выбора сценария кампании) и унаследован от движка Shadow of Death. Тот же вылет по адресу 0x00457D84 подтверждён в чистом SoD-режиме (модуль Heroes3.exe, отдельная серия из 41 лога, папка 457D84-SoD), поэтому баг не специфичен для HotA и существовал до версии 1.8.0. Там же -- показания игроков о сценарии: клик на сценарий кампании "Spoils of War" при загрузке сохранённой кампании, "happend 3 times and keeps happening".

Стабильные сигнатуры для опознания вылета независимо от версии:

  • Context-инварианты: EDI=0x00000001, EDX=0xFFFFFF00 (DL=0), EBX=0x00000072; адрес сбоя -- чтение по EAX + ECX + 0x1F879 при валидном EAX и выходящем за диапазон ECX.
  • Текстовая сигнатура по дизассемблеру (поиск в листинге любой версии): mov ecx, <SelectedPlayerID-глобал> test ecx, ecx jl short <skip> mov dl, [eax+edi+1F879h] mov bl, [eax+ecx+1F879h] <- инструкция сбоя cmp dl, bl Характерно отсутствие сравнения индекса с верхней границей (нет cmp с 8) между проверкой знака и чтением.
  • Стабильные адреса h3hota.exe: функция расстановки союзников/врагов 0x00457CF0, вызов из обработчика выбора сценария 0x004578E4, поставщик значения -- виртуальный метод Get_PlayerID объекта start_options (слот таблицы виртуальных функций +0x20). Эти адреса общие для Heroes3.exe (SoD) и h3hota.exe (HotA), так как оба имеют базовый адрес 0x400000.
  • Сиблинг: соседнее чтение 0x00457D7D (того же поля команды по индексу слота EDI с проверкой знака слота) -- проверки верхней границы там также нет, потому что слот ограничен циклом 0..7; для "выбранного игрока" такого ограничения нет.

Воспроизведение вылета¶

Уровень воспроизводимости: частичный. Путь сбоя детерминирован (открытие/ обновление сценария в диалоге кампании всегда доходит до слота игрока №1), но само падение зависит от того, что объект сценария вернул значение вне диапазона 0..7, а это определяется рассинхроном сохранённого состояния кампании с заново загружаемым файлом .h3c. Гарантированной последовательности "нажми X → вылет" в общем случае нет, но по показаниям игроков SoD-серии сбой устойчиво повторяется на конкретном сейве кампании.

Условие активации пути (из кода): функция расстановки союзников/врагов вызывается при выборе/смене сценария в диалоге кампании; до чтения по индексу "выбранного игрока" слот №0 должен быть пропущен по флагу наличия, а слот №1 -- иметь неотрицательный флаг (во всех логах так и есть).

Роль 1 -- тестер за игрой (без отладчика):

  • Запустить ту же сборку (HotA 1.8.0 поверх HD-мода 5.7 R9/R12/R13).
  • Войти в одиночную кампанию; приоритетно -- ПРОДОЛЖИТЬ сохранённую кампанию, затем открыть диалог выбора сценария и кликать по сценариям.
  • Повышает шанс: использовать сейв кампании, сделанный в другой версии или сборке кампании, чем установленная сейчас (рассинхрон индексов); перебирать сценарии, в том числе пользовательские и тестовые файлы (в логах встречается test.h3m).
  • Что ловить: сам факт вылета с адресом 0x00457D84. Повторять загрузку сохранённой кампании и вход в диалог многократно.
  • Варьировать: разные кампании (стоковые и пользовательские .h3c), разные сценарии внутри кампании, повторные открытия диалога, сейвы кампании из разных версий.

Роль 2 -- подтверждение причины (с отладчиком / минидампом):

  • Точка останова на 0x00457D84. Подтвердить, что ECX ("выбранный игрок") вне диапазона 0..7 и что faulting = EAX + ECX + 0x1F879.
  • Подняться к виртуальному вызову 0x00457D1B: снять тип объекта start_options ([ecx] -- какая из таблиц Player/Bonus/Hero), значение аргумента selected_bonus_ix (camp_info +0x1F468) и возврат метода.
  • Сверить scenario_ix (this+0x6C) и реальный размер scenarios_list с тем, что даёт перезагрузка .h3c (число = A0_CampMapInfos[campaing_id]. region_count): расхождение подтверждает рассинхрон индексов сейва.
  • Дифференциальный тест: условный останов/патч, пропускающий путь при "выбранном игроке" >= 8, должен убрать именно этот вылет -- это подтверждает, что за границы выводит индекс, а не база.
  • Минимальные данные для репорта разработчикам: краш-лог + минидамп, тип объекта start_options, значения scenario_ix и selected_bonus_ix, имя кампании/сценария и сейв кампании.

Вывод¶

Виновник -- код главного модуля h3hota.exe (диалог выбора сценария кампании, функция расстановки союзников и врагов sub_457CF0), унаследован от движка SoD. "Выбранный игрок" возвращает виртуальный метод Get_PlayerID объекта сценария кампании и используется как индекс в массив команд игроков из 8 элементов с проверкой только знака и без проверки верхней границы. Механизм разрешён по статике: два индекса из сохранённого состояния кампании (выбор сценария и аргумент Get_PlayerID) применяются без верхней границы поверх списка сценариев, заново загружаемого из файла .h3c; при их рассинхроне метод возвращает значение вне диапазона 0..7, и чтение байта команды уходит за пределы памяти. Точка сбоя и контракт поставщика подтверждены арифметикой и листингом во всех 19 вылетах; конкретный фактор рассинхрона (версия/мод кампании, повреждение сейва) по статике не локализуется -- для него нужен сейв игрока или минидамп.
