Визуализация вылета #

Объект на клетке карты хранит номер героя. Вызывающий код превращает номер в указатель на запись героя по формуле this = GameMgr + номер*1170 + 0x21620, проверяя номер только на значение -1. Метод снятия объекта сразу читает поле this+6. Если номер вне диапазона массива героев или равен -1 — указатель указывает мимо массива либо равен нулю, и чтение роняет игру.

Fномер = -1 -> this = 0
B = базамассив героев
записи 0..N-1шаг 1170 байт
Fномер >= N -> за границей

проверка только на -1; нет нижней и верхней границы номера

B База массива записей героев: GameMgr + 0x21620, шаг записи 1170 байт.
F Адрес сбоя this+6: при номере -1 указатель нулевой (this = 0, адрес сбоя 0x6); при номере вне диапазона указатель уходит за массив. Во всех вылетах адрес сбоя = ECX + 6.

Краткое резюме¶

Игра берёт "номер героя", записанный в объекте на клетке карты, и по этому номеру лезет в массив героев, не проверив, что номер вообще осмысленный. Если номер битый (или это особое значение "нет героя" = -1), вычисленный адрес героя оказывается мусором либо нулём, и первая же попытка прочитать его поле роняет игру. Чаще всего (33 из 56 вылетов) это случается при загрузке сохранённой игры (по комментариям игроков — рестарт сетевой партии), реже — при формировании всплывающей подсказки клетки.

Тип исключения: EXCEPTION_ACCESS_VIOLATION (чтение). Где: h3hota.exe, адрес 0x004D7956, инструкция mov al, [ecx+6] — первая инструкция приватного метода A0_MoveableObject_RemoveFromMap_sub_4D7950 (снятие движимого объекта с клетки карты). Что произошло: ECX (указатель this на запись героя) невалиден; читается поле this→is_on_map_cell по смещению +6. Адрес сбоя всегда равен ECX+6. Корневой регистр: ECX.

Корреляция дампов (сравнительная таблица)¶

Все 56 уникальных вылетов: один адрес сбоя 0x004D7956, операция read, во всех faulting = ECX + 6. Точных дублей нет. Серия СМЕШАННАЯ по версиям (6 версий HotA, десятки ревизий HD-мода) — это кросс-версионная выборка одного дефекта в неизменном движке главного EXE.

Выборка (значение ECX = указатель this; EAX справочно):

Соответствует ветке вызывающего кода, где номер героя равен -1 и this намеренно обнуляется. B) Большой невалидный this (ECX порядка десятков-сотен МБ, у каждого вылета свой): указатель ведёт за пределы массива героев либо в освобождённую/чужую память. C) this со взведённым старшим битом (EAX вида 0x8200xxxx, реже 0x4100xxxx): источник номера героя содержит явно битое значение. Этот профиль НЕ случайный мусор: младшая часть — правдоподобный номер героя, а старшая — один взведённый старший бит (25 либо 24) в поле номера (подробно в разделе 5). Этот профиль почти целиком идёт через путь подсказки клетки (0x0040AFFE).

Комментарии игроков про рестарт относятся ИМЕННО к tooltip-пути (багу движка): 4 из 5 дампов с комментарием идут через подсказку клетки на 4 РАЗНЫХ картах. Разные карты, разные дни, разные игроки и разные версии при одном адресе сбоя и одном инварианте faulting = ECX + 6 указывают на системный шаблон (доверяющая функция без проверки), а не на один локальный баг.

Сопоставление адресов¶

Главный EXE загружен по 0x00400000, базовый адрес образа листинга также 0x00400000, поэтому адрес загрузки совпадает с базовым адресом образа и адреса транслируются один в один:

IDA_addr = runtime_addr — load_base + imagebase IDA_addr = 0x004D7956 — 0x00400000 + 0x00400000 = 0x004D7956

Адрес сбоя ложится на .text:004D7956 в листинге Модули/h3hota.exe.txt.

Проверка совпадения адреса сбоя с регистром (во всех вылетах): faulting_addr = ECX + 6

дамп 1.8.00x2DEADFD2 + 6 = 0x2DEADFD8 (совпадает)
дамп 1.7.30x00000000 + 6 = 0x00000006 (совпадает)
дамп 1.6.10x0E1194D8 + 6 = 0x0E1194DE (совпадает)

Поле +6 разыменовывается по указателю this (ECX) — адресная проверка сошлась.

Точка сбоя — разбор инструкций¶

Функция A0_MoveableObject_RemoveFromMap_sub_4D7950 (thiscall, this в ECX), пролог и инструкция сбоя:

.text:004D7950  push    ebp
.text:004D7951  mov     ebp, esp
.text:004D7953  sub     esp, 8
.text:004D7956  mov     al, [ecx+6]        <-- СБОЙ (чтение this+6)
.text:004D7959  test    al, al
.text:004D795B  jz      loc_4D7A1C
.text:004D7961  mov     edx, [ecx+7]

Поле +6 — это флаг is_on_map_cell записи объекта. Имя и смещение поля подтверждаются соседней функцией A0_MoveableObject_PlaceToMap_sub_4D7840, которая по тому же смещению пишет этот флаг:

.text:004D7846  mov     al, [ecx+6]        ; if ( !this->is_on_map_cell )
.text:004D78DB  mov     byte ptr [ecx+6], 1 ; this->is_on_map_cell = 1

Метод сразу, без какой-либо проверки указателя, читает поле по this. При невалидном this (нуль, мусор или висячий указатель) первое же чтение [ecx+6] вызывает access violation.

Регистровые инварианты¶

Единый инвариант всех 56 вылетов: faulting = ECX + 6. Это сигнатура доступа к полю +6 по битому this, а не описание механизма.

ECX (this) НЕ постоянен и НЕ образует фиксированного индекса: значения варьируются от нуля до больших невалидных адресов. Широкий разброс при стабильном смещении +6 означает, что разыменовывается невалидный указатель РАЗНОЙ природы (нулевой; за пределами массива героев; со взведённым старшим битом), а не один и тот же вычисленный индекс.

В профиле "нулевой this" EAX = 0x22 (= 34) — это тип объекта на клетке, оставшийся в EAX от предшествующего сравнения в вызывающем коде (cmp eax, 22h). Он отмечает именно ветку, где номер героя равен -1.

Восстановление номера героя для профиля C (путь подсказки клетки, 21 вылет). В этом пути вызывающий вычисляет EAX = номер * 65 (shl eax,6; add eax,ecx), поэтому номер восстанавливается из EAX напрямую (умножением на обратный элемент 65 по модулю 2^32), без опоры на адрес GameMgr. Результат для всех 21 вылетов:

номер = (старший бит) | k, k из {1, 2, 21, 26, 36, 37, 45, 71, 73, 113, 115, 139, 148, 161, 162}

Проверка: 0x02000000 * 65 = 0x82000000, поэтому k=1 даёт ровно EAX = 0x82000041 (совпадает с дампом). То есть младшая часть k — ВАЛИДНЫЙ индекс героя (1..162, в диапазоне реальных героев игры), а повреждение — это ОДИН взведённый СТАРШИЙ бит в поле номера, а НЕ случайный мусор. Из 21 вылета в 20 это бит 25 (0x02000000), в 1 — бит 24 (0x01000000, EAX = 0x41000082). Это сужает поиск повреждающего кода до того, что выставляет старший бит (24/25) сериализованного id (вероятно — служебный флаг, не очищенный перед использованием поля как индекса).

Восстановление номера для профиля B (savegame-путь, 26 вылетов). Здесь вызывающий вычисляет EAX = номер * 585, поэтому номер тоже восстанавливается из EAX (умножением на обратный элемент 585 по модулю 2^32). Во всех 26 дампах EAX ровно кратен 585, номер извлекается чисто:

номер из {740, 3057, 3059, 3065, 3067, 3068, 3069, 3072, 3073, 49161}

В отличие от профиля C, здесь НЕТ взведённого старшего бита — это правдоподобно большие, но всё равно OUT-OF-BOUNDS номера (реальных героев ~156, а здесь 740, 3069, 49161). Кластер вокруг 3069 (3057..3073) повторяется у разных дампов — признак устойчивого значения в данных сейва, а не случайного разброса. То есть два пути дают РАЗНЫЙ характер порчи поля setup: tooltip — валидный индекс с одним лишним старшим битом; savegame — большое целое вне диапазона. Регистр EDX в момент сбоя НЕ равен адресу GameMgr (равенство ECX = EDX + номер*1170 + 0x21620 не сходится), поэтому номер восстанавливается из EAX, а не из ECX.

Восстановленная цепочка вызовов¶

Основной путь (33 из 56) — загрузка сохранённой игры:

Подсистема: приключенческая карта — размещение и снятие движимых объектов (записей героев) на клетках карты. Основной вызывающий при загрузке сейва перебирает все клетки карты и для каждой верфи (объект типа 34) снимает с клетки стоящего на ней героя; второй путь формирует данные для всплывающей подсказки клетки.

Первопричина¶

Указатель this для RemoveFromMap вычисляется в вызывающем коде как адрес записи героя по её номеру, прочитанному из объекта на клетке карты. Номер проверяется ТОЛЬКО на -1, без проверки верхней границы.

Вызывающий A0_TipSavedInfo_PrepareCell_ForTip_sub_40AF10:

.text:0040AFCC  mov     ecx, [edi]          ; ecx = fore_item->setup (номер)
.text:0040AFCE  cmp     ecx, 0FFFFFFFFh
.text:0040AFD1  jnz     short loc_40AFE5
.text:0040AFD3  xor     ecx, ecx            ; this = 0
.text:0040AFD7  call    RemoveFromMap        ; RemoveFromMap(NULL)
...
.text:0040AFE5  mov     edx, A0_GameMgr_dword_699538
.text:0040AFEB  mov     eax, ecx
.text:0040AFED  shl     eax, 6
.text:0040AFF0  add     eax, ecx             ; eax = номер * 65
.text:0040AFF2  lea     ecx, [eax+eax*8]
.text:0040AFF5  lea     ecx, [edx+ecx*2+21620h] ; this = GameMgr +
                                              ; номер*1170 + 0x21620
.text:0040AFFE  call    RemoveFromMap        ; RemoveFromMap(this)

То есть this = GameMgr→hero_array[номер] (размер записи героя 1170 байт, массив по смещению +0x21620). Проверка только на -1; для любого другого невалидного номера (большого или повреждённого) this уходит за пределы массива героев.

Основной вызывающий (загрузка сейва) A0_GameMgr_Setup_PlayerDockyardCoordLists_sub_4BC7F0 перебирает все клетки карты (вложенные циклы по x/y) и использует тот же непроверенный номер:

.text:004BC904  mov     eax, [ecx+eax*2+1Eh] ; object_type клетки
.text:004BC908  cmp     eax, 22h             ; тип 34 = верфь?
.text:004BC90B  jnz     short loc_4BC936
.text:004BC90D  mov     ecx, [esi]           ; ecx = setup (номер)
.text:004BC90F  cmp     ecx, 0FFFFFFFFh
.text:004BC912  jnz     short loc_4BC918
.text:004BC914  xor     ebx, ebx             ; this = 0 (номер = -1)
.text:004BC916  jmp     short loc_4BC929
.text:004BC918  mov     eax, ecx             ; иначе: номер*1170
.text:004BC91A  shl     eax, 6
.text:004BC91D  add     eax, ecx
.text:004BC91F  lea     eax, [eax+eax*8]
.text:004BC922  lea     ebx, [edx+eax*2+21620h] ; this = hero_array[номер]
.text:004BC929  mov     ecx, ebx
.text:004BC92B  call    RemoveFromMap

Та же схема (проверка только на -1, без верхней границы), но номер берётся ПРЯМО из только что прочитанных данных карты. Это объясняет преобладание savegame-пути: при загрузке частично или неконсистентно прочитанного сейва поле setup верфи содержит битый номер героя, и this уходит за массив.

Это объясняет оба класса невалидного this:

  • профиль "нулевой this": номер = -1, вызывающий НАМЕРЕННО передаёт NULL, а RemoveFromMap не имеет проверки на NULL и падает;
  • профили "большой/битый this": номер повреждён или вне диапазона, this указывает за массив героев / в чужую память.

Сравнение с родственными функциями (контракт "доверяй входу"):

  • соседний метод той же структуры A0_MoveableObject_PlaceToMap_sub_ 4D7840 ТАКЖЕ первой инструкцией читает [ecx+6] без проверки this;
  • то есть оба метода доверяют корректности this по контракту — это не регрессия конкретной функции, а проектное соглашение приватных методов структуры движимого объекта.

Следовательно падающий метод по своему контракту исправен; первопричина

  • у поставщика данных: объект клетки карты содержит номер героя, не проходящий проверку верхней границы, а в вырожденной ветке вызывающий сознательно передаёт NULL в метод, не рассчитанный на NULL. Точечная проверка на NULL в RemoveFromMap убрала бы только профиль "нулевой this"; вылеты с большим/битым this (повреждённый номер) она бы не устранила.

Условия, спровоцировавшие сбой¶

Состояние и настройки из логов:

  • Game Type = 2 во всех просмотренных дампах; у части вылетов Network Game = 1 (сетевые партии);
  • карты, игровой день и игрок РАЗНЫЕ между вылетами;
  • графические режимы и версии HD-мода РАЗНЫЕ. Ни один из этих параметров не является общим триггером. Общее одно: обрабатывается клетка карты с объектом-героем (тип 34), номер героя в котором не проходит проверку диапазона либо равен -1.

Комментарии игроков (блок Comment, дословно). В шести логах серии есть пояснения отправителей; три из них независимо указывают на рестарт сетевой партии во время или сразу после боя:

  • "завершил бой в погосте и похоже что оп нажал рестарт." (сетевая игра, ход игрока);
  • "Ошибка появилась вперые, играл с оппонентом, он нажал рестарт и появилась ошибка." (сетевая игра, ход игрока);
  • "вышел из боя одновременно с рестартом. вылет" (сетевая игра, ход игрока);
  • "закупался в нетральном ларее" (сетевая игра);
  • "wystapil blad taki prosze naprawcie to" (польск.: "произошла такая ошибка, пожалуйста, исправьте"). Во всех трёх "рестарт"-комментариях Network Game = 1 и Active Player = Me (рестарт инициировал оппонент, пока локальный игрок обрабатывал бой или объект на карте). Это согласуется с профилем "большой/битый this": сброс/перезагрузка сессии по сети разрушает состояние партии (запись героя освобождается/переписывается), а локальный код продолжает снимать объект-героя с клетки по уже невалидному указателю. Это показания игроков, а не подтверждённый кодом сценарий. Само значение битого номера восстановлено для обоих путей (раздел 5), но КОД, выставляющий этот номер в поле клетки, по статическому листингу не локализуется (не в стеке сбоя).

Достоверность и ограничения¶

Верификация листинга: модуль сбоя — главный EXE (h3hota.exe), базовый адрес образа 0x00400000 совпадает с адресом загрузки; адреса сходятся один в один. Главный EXE неизменен между версиями HotA, поэтому адрес 0x004D7956 стабилен во всех 6 версиях серии — листинг Модули/h3hota.exe.txt применим ко всем вылетам.

Достоверность СИГНАТУРЫ — очень высокая: 56 уникальных вылетов на одном адресе при 6 версиях HotA и десятках ревизий HD-мода, единый инвариант faulting = ECX + 6, разобранный код вызывающих и падающей функции.

Достоверность МЕХАНИЗМА — средняя: профиль "нулевой this" детерминирован и доказан кодом (ветка номер = -1 → RemoveFromMap(NULL) без проверки); профили "большой/битый this" доказывают отсутствие проверки верхней границы номера. Сами битые номера восстановлены для обоих путей (раздел 5: tooltip — валидный индекс плюс старший бит 24/25; savegame — большое целое вне диапазона, кластер ~3069), но код, выставляющий этот номер в поле клетки, по статическому листингу не локализуется (не в стеке сбоя, для этого нужен минидамп или листинг сериализатора сейва). Комментарии игроков (раздел 8) поддерживают этот профиль: три независимых отчёта связывают вылет с рестартом сетевой партии во время боя. Это показания пользователей, а не подтверждённый кодом механизм; они задают направление воспроизведения, но не меняют арифметически доказанный диагноз точки сбоя.

Ограничение по периферийному модулю: часть вылетов приходит через путь hota.dll (вызывающий hota.dll+0x167B19). Падающий модуль (h3hota.exe) во всех вылетах один и тот же и совпадает с листингом, поэтому диагноз виновного кода в силе; листинги всех версий hota.dll под этот периферийный путь не сверялись (для трассировки этого пути потребовался бы листинг hota.dll соответствующей версии).

Полезность лога — СРЕДНЯЯ: падающая функция исправна по своему контракту, первопричина — у поставщика данных (номер героя из объекта клетки без проверки границы; намеренная передача NULL в вырожденной ветке). Лог чётко указывает подсистему и поставщика, но точечный фикс не в самой точке сбоя. Отсутствие проверки this в RemoveFromMap само по себе не засчитывается как дефект логики, поскольку это контракт, разделяемый соседним методом той же структуры.

Признаки поиска в старых версиях HotA / в SoD¶

Адрес 0x004D7956 лежит в главном EXE и стабилен между версиями HotA (EXE неизменен под HotA), поэтому опознаётся напрямую по адресу. Дополнительные стабильные сигнатуры:

  • Инвариант Context: faulting_addr = ECX + 6 (операция read); вырожденная ветка — ECX = 0 и EAX = 0x22 (тип объекта 34).
  • Сигнатура повреждённого номера (путь подсказки клетки): EAX вида 0x8200xxxx (реже 0x4100xxxx), то есть номер = старший бит (25 либо 24) плюс валидный малый индекс k; связь EAX = номер * 65.
  • Текстовая сигнатура по листингу (точка сбоя — начало функции): sub esp, 8 mov al, [ecx+6] test al, al в самом начале proc по адресу 0x004D7950.
  • Стабильные адреса главного EXE для поиска пути вызова: call 0x004D7950 из 0x0040AFFE (подсказка клетки) и из 0x004BC92B (списки верфей игрока); вычисление this как GameMgr + номер*1170 + 0x21620.
  • Родственный метод без проверки this: A0_MoveableObject_PlaceToMap_ sub_4D7840 (0x004D7840) — тот же контракт "доверяй this".

Код находится в главном EXE (наследие движка SoD), поэтому сигнатура ожидается и в более старых версиях HotA, и в исходном движке SoD по тому же адресу.

Воспроизведение вылета¶

У вылета ДВА НЕЗАВИСИМЫХ корня (раздел 2), воспроизводятся по-разному:

(А) savegame-ветка — ДЕТЕРМИНИРОВАННЫЙ репро по конкретной карте. 24 из 26 savegame-вылетов — на карте krolestwo-w-ogniu (и её копиях) у разных игроков и в 4 версиях HotA, номер устойчиво ~3069. Это дефект ДАННЫХ этой карты — сама карта служит готовым репро: загрузить сейв/партию на krolestwo-w-ogniu и выполнить повторное чтение сейва (рестарт партии). Репро по содержимому артефакта надёжнее рукотворного.

(Б) tooltip-ветка — НЕДЕТЕРМИНИРОВАННый баг движка, не зависящий от карты (21 вылет на 21 разной карте). Здесь номер — валидный индекс с одним взведённым старшим битом (24/25); гарантированной последовательности "нажми X" нет. Именно к этой ветке относятся 4 из 5 комментариев про рестарт (на 4 разных картах).

Роль 1 — тестер за игрой (без отладчика):

  • Сценарий А (детерминированный, savegame): взять карту krolestwo-w-ogniu, загрузить партию/сейв на ней и выполнить повторное чтение сейва (рестарт партии). Наиболее надёжный путь повторить вылет — дефект в данных этой карты воспроизводится у разных игроков.
  • Сценарий Б (недетерминированный, по комментариям игроков, tooltip): сетевая партия на двоих (Network Game = 1). Один игрок ведёт бой (либо закупается/взаимодействует с объектом на карте), а второй в этот момент жмёт рестарт партии. Повторять рестарт ВО ВРЕМЯ боя / выхода из боя у второго игрока (комментарии: "завершил бой... оп нажал рестарт", "вышел из боя одновременно с рестартом").
  • Дополнительно воспроизводить ситуации, затрагивающие объекты-герои на карте приключений: наведение курсора на клетку с героем/объектом героя (всплывающая подсказка клетки) и пересчёт списков верфей игрока.
  • Прогонять разные карты (в том числе сетевые партии, Game Type 2, Network Game = 1, как в части логов), на разных днях и игроках.
  • Что ловить: сам факт вылета с адресом 0x004D7956 (тот же адрес в краш-логе). Регистры тестеру снимать не нужно.
  • Поскольку условие — повреждённое/граничное состояние данных, полезен массовый повтор партий и продолжение длинных партий, где состояние объектов на карте успевает измениться.

Роль 2 — подтверждение причины (с отладчиком или по минидампу):

  • Точка останова на инструкции сбоя 0x004D7956.
  • Снять ECX и проверить инвариант: faulting_addr = ECX + 6.
  • Восстановить номер героя: для пути 0x0040AFFE / 0x004BC92B this = GameMgr + номер*1170 + 0x21620; вычислить номер = (ECX — GameMgr — 0x21620) / 1170 и убедиться, что он вне диапазона существующих героев (либо ECX = 0 при номере -1).
  • Дифференциальная проверка причины: на той же точке останова пропускать вызов при невалидном номере (номер = -1 или номер >= числа героев); если именно эти вылеты исчезают — первопричина (непроверенный номер у поставщика) подтверждена.
  • Для профилей "большой/битый this" дополнительно снять минидамп и по карте памяти установить, откуда в объект клетки попал повреждённый номер героя.

Вывод¶

Виновник — h3hota.exe (главный исполняемый файл, код наследия движка SoD). Приватный метод снятия движимого объекта с клетки карты (A0_MoveableObject_RemoveFromMap_sub_4D7950) по контракту доверяет указателю this и первой же инструкцией читает его поле без проверки. Вызывающий код вычисляет this из номера героя, записанного в объекте клетки, проверяя номер только на -1 и не проверяя верхнюю границу, а в вырожденной ветке намеренно передаёт в метод нулевой указатель. В результате при повреждённом или граничном номере героя (а также при номере -1) метод разыменовывает невалидный либо нулевой указатель и роняет игру.
