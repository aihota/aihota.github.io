Загрузка карты, расстановка городов

Заключение
Вылет при загрузке карты. Функция расстановки городов доверяет флагу карты "у этого города есть стартовая настройка" и индексу настройки, но список стартовых настроек городов оказался ПУСТ. Доступ к пустому списку идёт без проверки → чтение по почти нулевому адресу. Падающая функция — унаследованный из SoD код, доверяющий карте; первопричина — почему список пуст при наличии городов на карте (поставщик данных при загрузке), и требует самой карты для точной локализации.

Визуализация вылета #

Расстановка городов берёт из клетки карты номер стартовой настройки (idx) и вычисляет адрес элемента как база + 136·idx, затем читает поле +0x54. Но база списка настроек ([edi+98h]) равна NULL — список пуст. Адрес вырождается в 136·idx + 0x54 и попадает в невыделенную область памяти у нулевого адреса (младшие 64 КБ зарезервированы ОС под отлов NULL-указателей).

N idx = setup_dword0 — номер стартовой настройки города, взятый из клетки карты. Корректен (EAX = 136·idx без остатка), мусором не является.

A штатно база указывала бы на выделенный массив элементов по 136 байт; здесь массива нет (контракт «город есть → настройка есть» нарушен данными карты).

Краткое резюме¶

Игра падает сразу при загрузке карты (Day = 1 во всех логах), на этапе расстановки городов. Движок проходит по клеткам карты; встретив клетку с объектом "город" или "случайный город", помеченную флагом "есть стартовая настройка", он лезет в список стартовых настроек городов за элементом по номеру настройки и читает первый же байт этого элемента (признак "задано ли имя"). Но список стартовых настроек ПУСТОЙ — его базовый указатель равен нулю. Поэтому чтение уходит по адресу "ноль плюс маленькое смещение" (136·номер + 0x54) и падает.

Проверки тут нет: код безусловно доверяет тому, что если у городской клетки выставлен флаг настройки, то в списке есть соответствующий элемент. Это поведение унаследовано из оригинальной игры (контракт "доверяй данным карты"). Сам по себе код расстановки исправен — он падает потому, что ему дали несогласованные данные: на карте есть города, а список их стартовых настроек пуст.

Тип исключения: EXCEPTION_ACCESS_VIOLATION (0xC0000005), чтение. Что произошло: чтение поля элемента списка стартовых настроек городов по базовому указателю, равному NULL. Где: h3hota.exe @ 0x004CA963, функция A0_GameMgr_LoadMap_PlaceTowns_ sub_4CA730, инструкция "mov cl, [eax+54h]". Корневой инвариант: EAX кратен 136 (= размер элемента настройки), фолт = EAX + 0x54; база списка (GameMgr+0x98) = NULL во всех дампах.

Корреляция дампов (сравнительная таблица)¶

Все 62 уникальных вылета: один адрес 0x004CA963, операция read, непосредственный вызывающий 0x004BFCD3 (LoadMap). Точных дублей нет.

Главный инвариант (доказан по всем 62 дампам): адрес сбоя == EAX + 0x54 И EAX кратен 136 (0x88). То есть EAX = 136·idx, где idx — номер стартовой настройки (setup_dword0) из городской клетки карты. Базовый указатель списка при этом = NULL, поэтому EAX = база(0) + 136·idx = 136·idx.

Выборка (адрес сбоя = EAX + 0x54, idx = EAX/136):

Детерминированность по карте (доказано): для карт, встретившихся в нескольких логах, EAX (= упавший город) ИДЕНТИЧЕН во всех повторах, при разных версиях HD-мода:

Один и тот же упавший город при разных пользователях и версиях HD-мода → краш ЖЁСТКО детерминирован содержимым конкретной карты, а не гонкой или случайной порчей памяти.

Разброс среды при общем адресе: 62 разных пользователя, разные карты, 22 версии HD-мода (5.6 R7 ... 5.7 R6), Windows и WINE. Один адрес, одна инструкция, один инвариант (база=NULL, idx=EAX/136) при таком разбросе → это системный паттерн состояния данных, а не баг одной карты/машины.

Кросс-версионное подтверждение (два отдельных дампа вне серии 1.8.0):

  • HotA 1.7.3 (h3hota.exe): EAX=0x88 (idx 1), адрес сбоя 0xDC=0x88+0x54.
  • SoD (Heroes3.exe, imagebase 0x400000): EAX=0xAA0 (idx 20), адрес сбоя 0xAF4=0xAA0+0x54. Оба — тот же VA 0x004CA963, та же цепочка 0x4CA730<-0x4BFCD3, тот же инвариант (адрес сбоя=EAX+0x54, EAX кратен 136), Day=1, Seed=0, карта "{Default}/По умолчанию". Вылет НЕ новый в 1.8.0 и НЕ специфичен для HotA: он живёт в унаследованном из SoD коде расстановки городов (разделы 9, 10).

Версии HD-мода в серии СМЕШАННЫЕ — это периферийный модуль (внешний игровой цикл, см. раздел 6), не виновный код; на диагноз падающего модуля h3hota.exe рассинхрон версий HD-мода не влияет.

Сопоставление адресов¶

Листинг: Модули/h3hota.exe.txt File Name h3hota.exe, Imagebase 0x400000, Compiler Visual C++, Input SHA256 1766ADBF6ED99B58A923338E42A78C0C517934A3550BA499D55DC8 6036696657, MD5 FBA90A0411674DEC2D2E2BA1F7FFBD45, CRC32 D3259A0D.

Главный EXE имеет фиксированный imagebase 0x400000 и грузится по тому же адресу (Modules: 0x00400000 h3hota HD.exe во всех логах), поэтому runtime-адрес и адрес листинга совпадают напрямую.

Формула: IDA_addr = runtime_addr — load_base + imagebase = 0x004CA963 — 0x00400000 + 0x00400000 = 0x004CA963

Проверка: search_for_text по 0x004CA963 в листинге даёт ".text:004CA963 mov cl, [eax+54h]" внутри A0_GameMgr_LoadMap_PlaceTowns_sub_4CA730. Адрес заголовка лога (h3hota HD.exe @ 0x004CA963) совпадает. Подтверждено.

Точка сбоя — разбор инструкций¶

Функция A0_GameMgr_LoadMap_PlaceTowns_sub_4CA730 ставит города на карту при загрузке. Двойной цикл по клеткам карты (по Y, по X); для каждой клетки-города выполняется:

.text:004CA8EB  mov     eax, [esi+1Eh]     ; тип объекта клетки
.text:004CA8EE  cmp     eax, 62h           ; OBJ_TOWN ('b')
.text:004CA8F1  jz      short loc_4CA8FC
.text:004CA8F3  cmp     eax, 4Dh           ; OBJ_RANDOM_TOWN ('M')
.text:004CA8F6  jnz     loc_4CAA83         ; не город -> пропустить

.text:004CA90E  mov     ecx, eax

...
.text:004CA941  cmp     eax, 4Dh           ; ещё раз OBJ_RANDOM_TOWN?
.text:004CA944  jnz     short loc_4CA960   ; обычный город -> к +0x54
.text:004CA953  mov     ax, [edx+68h]      ; random ЧИТАЕТ ПОЛЕ +0x68

Механика отказа: town_start_info = town_start_info_list.data(база) + 136·setup_dword0. Поле +0x54 (has_name_setup_byte54) определяет, задано ли городу имя на карте. Когда база = NULL (список пуст), указатель вырождается в 136·idx, и чтение [eax+54h] падает по адресу 136·idx+0x54.

Падают ТОЛЬКО обычные города (доказано адресом сбоя): для случайного города (OBJ_RANDOM_TOWN) при пустом списке фолт случился бы РАНЬШЕ — на 0x004CA953 (чтение поля +0x68, адрес сбоя = 136·idx + 0x68). Но ВСЕ 62 дампа падают на 0x004CA963 с адресом сбоя = 136·idx + 0x54 → random-ветка не исполнялась ни разу; все вылеты — на ПРЕДРАЗМЕЩЁННЫХ (фиксированных) городах.

Адресная проверка R4: адрес сбоя == EAX + 0x54 во всех дампах, EAX = 136·idx. Пример (дамп 42): EAX=0x2A8=136·5, адрес сбоя=0x2A8+0x54=0x2FC. Сходится.

Пример (дамп 5)EAX=0, адрес сбоя=0+0x54=0x54. Сходится.

Регистровые инварианты¶

EAX (все дампы)кратен 136 (0x88), = 136·setup_dword0; это указатель town_start_info при НУЛЕВОЙ базе списка. idx = 0..38.
адрес сбоя== EAX + 0x54 (смещение поля has_name_setup_byte54).
EDI (все дампы)база объекта GameMgr (this), большой адрес с выравни-ванием по странице (0x0FBE0000, 0x07300000, ...). Это крупная куча-аллокация (поля до +0x4E000), а не мусор; [edi+98h] = town_start_info_list.data = NULL.
EBX/ESIEBX = элемент towns_list (валиден), ESI = указатель на клетку карты (валиден). Эти ветки сбоя не вызывают.
ECX0x100/0x200/0x300/0x600/0 — масштаб строки карты из вычисления адреса ячейки (0x4CA8C3), НЕ размер буфера.

База списка = РОВНО NULL (не мусорный указатель), а idx даёт EAX без остатка по 136 → список town_start_info ПУСТ (data = NULL), а не повреждён и не недоразмерен. Это отличает диагноз от "битый индекс": индекс корректен, отсутствует сам массив.

Восстановленная цепочка вызовов¶

Сверху вниз (V1 + V2 стек лога):

V1: "[0x004CA730] called from [0x004BFCD3]" — это прямой вызов PlaceTowns из LoadMap (строка 139 псевдокода 0x4BFB30). V2 подтверждает: 004CA963 → 004BFCD8 (возврат в LoadMap) → hota MainProc. Кадры 0x02A8xxxx / 0x07xxxxxx — трамплины patcher_x86 и кадры HD/HW-мода (внешний запуск партии), листингов по ним нет по определению (трамплины), перешагиваем.

Падение целиком объясняется внутри h3hota.exe: PlaceTowns читает пустой town_start_info_list. Периферийные кадры лишь привели в LoadMap и на диагноз не влияют.

ЦЕПОЧКА ПОСТАВЩИКА (где список ДОЛЖЕН наполняться) — восстановлена по листингу. Единственная запись в town_start_info_list.data (кроме обнуления в конструкторе 0x4CDC0E) идёт через InsertRange при чтении каждого города:

EXEA0_GameMap_Load_sub_4FDCE0 (чтение .h3m) | сначала ОЧИСТКА: DeleteRange town_start_info_list (0x4FDE28)
EXE0x4FDF2C call LoadTemplatesAndObjects_sub_504AC0 (объектная секция)
EXE0x5035C6 call LoadObject_sub_503450 (jumptable cases 0x4D/0x62)
0x50274C call InsertRange(&town_start_info_list,...,1,&v52) ; +1 элемент НА КАЖДЫЙ город

То есть список наполняется ПОЭЛЕМЕНТНО проходом по городам объектной секции; оба типа города (OBJ_TOWN 0x62 и OBJ_RANDOM_TOWN 0x4D) идут в LoadTown (диспетчер 0x5035BE, cases 77/98). Путь сейва (A0_Savegame_ReadMap_sub_4FE210) LoadTemplatesAndObjects НЕ вызывает — restore town-setup на нём идёт иным механизмом.

ПРОГЛАТЫВАНИЕ ОШИБКИ ЗАГРУЗКИ (механизм, доказан кодом). Нижний уровень корректно пробрасывает сбой чтения объектной секции:

.text:004FDF31  test eax, eax
.text:004FDF33  jge  short loc_4FDF3F
.text:004FDF35  or   eax, 0FFFFFFFFh    ; 4FDCE0 -> return -1

А 4C2110 (Load_GameMap, _BOOL1) возвращает 0 при неуспехе (0x4C2150 xor al,al). НО вызывающий LoadMap_4BFB30 результат ИГНОРИРУЕТ:

.text:004BFC79  call A0_GameMgr_Load_GameMap_sub_4C2110
.text:004BFC7E  lea  edi,[ebx+20B0Eh]   ; AL не проверяется, ветки нет
...
.text:004BFCD3  call PlaceTowns          ; исполняется БЕЗУСЛОВНО

Поэтому частично/неуспешно загруженная карта всё равно доходит до расстановки городов с пустым (или недонаполненным) списком настроек.

Проверка соседних адресов (НЕ родственники этого вылета): адресов EXE принадлежат другим функциям/подсистемам и общей первопричины не разделяют — 0x004C9E51 (генерация героев в замках), 0x004CA0A0 (боевой ИИ, оценка армии), 0x004CDB12 (анализ подземных врат). Близость адресов обманчива; группировать с PlaceTowns по подсистеме нельзя. "Семьи" по общей первопричине у этого вылета нет: в PlaceTowns единственная точка разыменования пустого списка (+0x54).

Первопричина¶

Список town_start_info_list (GameMgr+0x98, поля data/end/cap по +0x98/ +0x9C/+0xA0) — вектороподобный контейнер. Конструктор инициализирует его пустым:

.text:004CDC0E  this->town_start_info_list.data_dword4 = 0;
.text:004CDC14  this->town_start_info_list.end_dword8  = 0;
.text:004CDC1A  this->town_start_info_list.end_mem_dwordC = 0;

При загрузке карты он должен заполняться разделом стартовых настроек городов. Читатели подтверждают семантику и размер элемента 136:

.text:005020A4  v4 = GameMgr->town_start_info_list.data_dword4;
.text:005020AA  if ( v4 ) ... count = (end - data)/136   (LoadTown 0x502040)

Поведение ПОСТАВЩИКА vs ПОТРЕБИТЕЛЯ (граница списка):

  • Везде, где список ЧИТАЕТСЯ "правильно" (0x4CA7AC в прологе самой PlaceTowns, 0x5020A4 в LoadTown), стоит проверка "if (data)" и вычис-ление count = (end-data)/136. То есть штатный паттерн доступа — с проверкой непустоты.
  • НО в цикле расстановки (0x4CA908..0x4CA963) индексация идёт БЕЗ проверки: код берёт setup_dword0 прямо из клетки карты и обращается к data[136·idx], полагаясь на инвариант "если у клетки выставлен флаг [esi+0Dh]&0x10, то элемент в списке существует".

Пролог функции (0x4CA79F) даже ВЫЧИСЛЯЕТ count с проверкой "if(v7)":

.text:004CA79F  v7 = town_start_info_list.data_dword4;
.text:004CA7AC  if ( v7 ) v8 = (end - v7)/136; else v8 = 0;

но это значение (v8) используется лишь для ресайза СПИСКА ГОРОДОВ (towns_list), а не как граница для доступа по setup_dword0. Сам доступ +0x54 ничем не защищён.

Критерий "симптом или причина": Это унаследованный из SoD код расстановки городов (подтверждено SoD-дампом Heroes3.exe @ 0x004CA963 с тем же инвариантом и цепочкой, раздел 9). Контракт "флаг клетки ↔ элемент в списке настроек" — часть формата карты, которому движок доверяет, как и оригинальная игра. Следовательно, падающая функция сама по себе ИСПРАВНА в рамках своего контракта, а вылет — СИМПТОМ несогласованных входных данных: на карте есть города (клетки OBJ_TOWN/OBJ_RANDOM_TOWN с флагом настройки), а список их стартовых настроек пуст (data=NULL).

ПОСТАВЩИК флага и индекса на клетке (где флаг 0x10 УСТАНАВЛИВАЕТСЯ): есть два пути записи флага, и оба рядом пишут setup_dword0 в ту же клетку, НО это РАЗНЫЕ фазы:

.text:00505E07  or  byte ptr [edi+0Dh], 10h   ; загрузка карты (Setup_Cell)
.text:00505E21  mov [edi], eax                ; setup_dword0 — ТОЛЬКО при IsMapStart
.text:004D78FE  or  byte ptr [eax+0Dh], 10h   ; РАНТАЙМ-размещение (PlaceToMap)
.text:004D7908  mov [eax], edx                ; setup_dword0 = ob_setup

Второй путь (0x4D78FE, A0_MoveableObject_PlaceToMap_sub_4D7840) — размещение объекта ВО ВРЕМЯ ИГРЫ (XREF включают _WinMain/игровые вызовы), а НЕ путь загрузки карты; к вылету на Day 1 отношения не имеет. На пути загрузки (0x505E07) флаг 0x10 ставится БЕЗУСЛОВНО, а setup_dword0 — ТОЛЬКО когда IsMapStart (0x505E1B jz, и параллельная ветка 0x506049): флаг-без-индекса — реальное состояние кода.

ДВЕ ОТКРЫТЫЕ МОДЕЛИ пустого списка (по статике НЕ разрешаются): (A) РАССИНХРОН СЕКЦИЙ: setup-индекс клетки приходит из объектной секции файла, а секция town-setup пуста/не прочитана. (B) ПРОГЛОЧЕННАЯ ОШИБКА: загрузка объектной секции прервалась, а ошибка -1 проглочена в LoadMap (раздел 6), список остался пуст. Обе модели согласуются со всеми уликами (детерминизм, idx 0..38, "(Legacy)"); выбрать по статике нельзя — см. раздел 9.

О возможном различителе (НЕ доведён до вывода — требует .h3m). В LoadTown есть ранняя запись в ОБЪЕКТ (obj, _Object_*, 3-й арг): до файловых чтений туда пишется ТЕКУЩИЙ размер списка:

.text:005020AE  town_start_info_count_v6 = 0  (если data=NULL)
.text:005020D4  mov [esi], edx   ; obj->setup_dword0 = count (ДО append)

а InsertRange (+1) — в КОНЦЕ (0x50274C), между ними ранние return -1 (0x50213A, 0x502793, 0x5027C5). ЕСЛИ бы setup, читаемый PlaceTowns из КЛЕТКИ карты, равнялся этому count, то idx=38 требовал бы 38 предыдущих append'ов (против модели B). НО ЭТО НЕ ДОКАЗАНО: PlaceTowns читает [esi] из сетки клеток (MapCell), а туда setup копирует Setup_Cell (0x505E21) из objects_list10[ix] — равенство этого поля LoadTown'овскому count (а не сырому файловому порядковому номеру) по статике НЕ установлено. Поэтому обе модели остаются равноправными; прежнее "сужение развилки" отозвано как опиравшееся на непроверенную предпосылку.

То есть флаг и индекс на клетке, а setup_dword0 и сам список town_start_info_list — ОТДЕЛЬНЫЕ сущности (отдельная секция карты). Краш возникает именно из рассинхрона: клетка-город несёт флаг+индекс, а секция настроек пуста.

Первопричина (требует .h3m для точной локализации): ПОЧЕМУ список пуст при наличии городов. Данные логов уточняют происхождение карт:

  • Last RMG Seed = 0 во всех 62 логах (runtime-генерации нет, карта уже лежит файлом) + Day = 1 (первая загрузка).
  • Имена карт идут единым шаблоном "<игрок> <дата> <суффикс>.h3m", где суффиксы совпадают с именами RMG-шаблонов (Jebus Cross, 2sm4d(3), "test rmg 06", SVO, Zar, {Default}). ГИПОТЕЗА (по именам, строго не доказано): это сохранённые RMG-карты (генератор случайных карт расставляет фиксированные стартовые замки — согласуется с тем, что падают ПРЕДРАЗМЕЩЁННЫЕ города, раздел 4).
  • 7 из карт несут в имени "(Legacy)" — признак конвертации из старого формата. Вероятный механизм рассинхрона — путь конвертации/сохранения, переносящий city-клетки с флагом, но не наполняющий секцию town-setup. Штатный редактор карт такого рассинхрона не даёт (он пишет setup-запись и клеточный флаг согласованно); источник — либо RMG/ конвертация, либо сбой десериализации. Точный шаг по статике (без самой .h3m) не выбрать.

Замечание о детерминированности: инвариант idx=EAX/136 стабилен, и краш ДЕТЕРМИНИРОВАН по карте (раздел 2: один EAX во всех повторах карты). Это НЕ off-by-one в этом коде — граница потребителя (доступ по idx) и поставщика (заполнение списка) согласованы там, где проверка есть; падает именно непроверенный путь при ПУСТОМ списке.

Условия, спровоцировавшие сбой¶

Падение происходит на этапе загрузки карты (Day = 1 во всех логах), до начала игрового хода. Значения Me/Active Player (0/0, 1/1, 3/3, 5/5, 6/6, 7/7 по разным логам) на путь сбоя не влияют — расстановка городов идёт один раз при инициализации партии.

ini-настройки (Graphics.Mode, Memory.Mod, Quick Combat, Threads, TournamentSaver/BattleSaver) в логах варьируются и с путём сбоя не связаны: PlaceTowns — не графика, не бой и не сейв. Network Game = 0, Game Type = 0 во всех (одиночная загрузка/старт).

Значимый фактор — сама карта: на ней присутствуют ОБЫЧНЫЕ (предразме-щённые) города с выставленным флагом стартовой настройки, тогда как список настроек на момент расстановки пуст. Краш детерминирован по карте (раздел 2), а 43 уникальных карты у разных пользователей указывают на КЛАСС карт (вероятно сохранённые RMG-карты, см. раздел 7), а не на один файл.

Достоверность и ограничения¶

Подтверждено / высокая достоверность
  • МЕХАНИЗМ и ПОСТАВЩИК теперь ДОКАЗАНЫ КОДОМ (разделы 6-7): список наполняет только LoadTown/InsertRange (поэлементно, объектная секция); ошибка загрузки пробрасывается как -1, но ИГНОРИРУЕТСЯ в LoadMap (AL после 4C2110 не проверяется), и PlaceTowns исполняется безусловно.
  • ПЕРВОПРИЧИНА НЕ РАЗРЕШЕНА по статике: остаются ДВЕ равноправные модели (раздел 7): (A) рассинхрон секций файла и (B) проглоченная ошибка загрузки объектной секции. Обе согласуются со всеми уликами (детерминизм, idx 0..38, "(Legacy)"). Попытка развести их через idx=38 ОТОЗВАНА: она опиралась на непроверенное равенство setup-поля клетки счётчику LoadTown.
  • УНАСЛЕДОВАННОСТЬ ИЗ SoD ПОДТВЕРЖДЕНА дампом SoD-режима (Heroes3.exe @ 0x004CA963): тот же VA, тот же инвариант (адрес сбоя = EAX+0x54, EAX=136·20), идентичная цепочка 0x4CA730<-0x4BFCD3. Оба бинаря (h3hota.exe и Heroes3.exe) имеют общий imagebase 0x400000, поэтому унаследованный код лежит по тому же адресу (§13). Бинарный дифф чистого SoD не требуется: связь установлена по адресу + цепочке + регистровому инварианту. (SoD-лог — под HD-модом, модули HD_SOD/HW_SOD; это подтверждает наличие в SoD-ДВИЖКЕ; различение "чистый SoD без HD" не требуется — виновник в главном EXE, не в HD-моде).
  • Версии периферийного модуля HD-мода в серии разные. Это внешний запускающий слой (загрузка партии), не виновный код; рассинхрон версий периферии диагноз падающего модуля h3hota.exe не блокирует. Обоснование полезности (СРЕДНЯЯ): точка сбоя и класс локализованы точно, лог полезен. Но падающая функция в рамках своего контракта исправна — точечный фикс не в месте сбоя (добавить проверку idx<count здесь лишь скрыл бы симптом). Чинить нужно ПОСТАВЩИКА: гарантировать, что список стартовых настроек заполнен прежде расстановки, либо что клетки городов не несут setup-флаг при пустом списке. Само по себе отсутствие проверки непустоты в унаследованном коде дефектом не засчитывается. Код места сбоя (h3hota.exe) — закрытый сторонний бинарь; точечный фикс не в месте сбоя, а у поставщика данных при загрузке.
Ограничения и неизвестное
  • РАЗРЕШАЕТ развилку только .h3m из лога и/или mini-dump: снять town_start_info_list.data/end (GameMgr+0x98/+0x9C) и сверить раздел стартовых настроек с числом city-клеток. Карт в логах нет (приходит только текст лога).

Признаки поиска в старых версиях HotA / в SoD¶

Адрес главного EXE стабилен между сборками HotA (фикс. imagebase 0x400000), поэтому 0x004CA963 — устойчивая сигнатура внутри одной линейки. Дополнительные признаки для опознания этого вылета:

  • Context-инвариант: read, адрес сбоя == EAX + 0x54, EAX кратен 136 (0x88); значение EAX/136 = небольшое целое 0..несколько десятков (номер города). База GameMgr (EDI) — крупный page-aligned адрес.

- Текстовая сигнатура по дизассемблеру (расстановка городов):

mov   eax, [esi+1Eh]      ; тип объекта
cmp   eax, 62h / cmp eax, 4Dh   ; OBJ_TOWN / OBJ_RANDOM_TOWN
test  byte ptr [esi+0Dh], 10h   ; флаг "есть стартовая настройка"
mov   ebx, [edi+98h]      ; town_start_info_list.data (база)
...
mov   cl, [eax+54h]       ; чтение has_name_setup_byte54
  • Стабильное смещение поля town_start_info_list = GameMgr+0x98, размер элемента стартовой настройки = 136 (0x88) байт; смещение проверяемого поля = +0x54.
  • Признак связи с картой: имя карты вида "<игрок> <дата> <шаблон>.h3m" (суффиксы RMG-шаблонов: Jebus Cross, 2sm4d(3), {Default}, ...), часть с "(Legacy)"; Last RMG Seed = 0, Day = 1. Одинаковый EAX во всех повторах одной карты (детерминизм).
  • Признак "Были до 1.8.0": ПОДТВЕРЖДЕНО дампом HotA 1.7.3 (h3hota.exe @ 0x004CA963, EAX=0x88=136·1, адрес сбоя 0xDC=136+0x54, та же цепочка 0x4CA730<-0x4BFCD3). Инвариант совпадает с серией 1.8.0.
  • Признак "Были в SoD": ПОДТВЕРЖДЕНО дампом SoD-режима (Heroes3.exe @ 0x004CA963, imagebase 0x400000, EAX=0xAA0=136·20, адрес сбоя 0xAF4=0xAA0+0x54, идентичная цепочка 0x4CA730<-0x4BFCD3, модули HD_SOD/HW_SOD). По §13: общий imagebase 0x400000, связь по адресу + цепочке + инварианту → код расстановки городов унаследован из SoD и падает там же (см. раздел 9).

Воспроизведение вылета¶

Уровень воспроизводимости: ДЕТЕРМИНИРОВАН по карте (доказано в разделе 2: для карт из нескольких логов упавший город один и тот же независимо от версии HD-мода). Это означает: ЛЮБАЯ из 43 карт из логов падает стабильно при загрузке — готовый воспроизводящий артефакт, лучше рукотворного.

О ручном создании карты: штатным редактором рассинхрон не воспроиз-вести (он пишет town-setup запись и клеточный флаг согласованно). Теоретически возможна hex-правка .h3m (обнулить/усечь секцию стартовых настроек, сохранив city-клетки с флагом), но байтовая раскладка этой секции по статике не выводится (нужна спецификация формата .h3m либо реальный падающий файл). Этого и не требуется — реальные карты из логов уже дают детерминированный repro.

Условие активации пути: путь PlaceTowns исполняется ВСЕГДА при загрузке карты/старте партии. Сбой возникает, если на карте есть обычный (предразмещённый) город с флагом стартовой настройки ([esi+0Dh]&0x10), а список town_start_info пуст к моменту расстановки.

Роль 1 — тестер за игрой (без отладчика):

  • Взять карту из лога (имя из "Map file", напр. "...2sm4d(3).h3m", "...Jebus Cross.h3m"), особенно с пометкой "(Legacy)", и просто загрузить её / начать по ней партию в HotA 1.8.0. Ловить: тот же вылет h3hota.exe @ 0x004CA963 при загрузке (Day 1).
  • Ожидание: для карты из лога вылет СТАБИЛЕН при каждой загрузке (детерминизм по карте доказан), а не "через раз".
  • Если карты нет под рукой — создать RMG-карту по одному из шаблонов из имён (Jebus Cross и др.) либо взять старую карту и прогнать через конвертацию (Legacy).

Роль 2 — подтверждение причины (с отладчиком / mini-dump):

  • Точка останова на 0x004CA963. Снять: EAX (должен быть кратен 136), адрес сбоя == EAX+0x54, и [EDI+0x98] (town_start_info_list.data) — ожидается 0. Проверить [EDI+0x9C] (end) — тоже 0 при пустом списке.
  • Подняться к LoadMap (0x4BFB30) и месту, где должен заполняться раздел стартовых настроек: убедиться, что он пуст/не загрузился, тогда как клетки городов уже несут setup-флаг и индекс.
  • Дифф-тест нацелен на ПОСТАВЩИКА: взять конкретный .h3m и сверить наличие/целостность раздела стартовых настроек городов с числом городских клеток, несущих флаг настройки. Совпадение "города есть, список пуст" подтверждает первопричину у данных/загрузчика, а не в точке сбоя.

Минимальные данные для довершения диагноза: crash-log + сама карта (.h3m) хотя бы из одного лога; при наличии — mini-dump для снятия town_start_info_list.data/end и значения setup_dword0 упавшей клетки.

Вывод¶

Виновник места сбоя — h3hota.exe @ 0x004CA963, функция расстановки городов при загрузке карты (A0_GameMgr_LoadMap_PlaceTowns_sub_4CA730). Она безусловно доверяет флагу карты "у города есть стартовая настройка" и индексу настройки, обращаясь к списку town_start_info_list[136·idx+0x54] без проверки. Во всех 62 дампах база этого списка = NULL (список пуст), поэтому чтение падает по адресу 136·idx+0x54 (малый ненулевой адрес).

Это СИМПТОМ несогласованных входных данных: на карте присутствуют города, а список их стартовых настроек к моменту расстановки пуст. Падающий код — унаследованный из SoD контракт "доверяй карте" и в рамках контракта исправен; точечная проверка здесь лишь замаскировала бы симптом. Первопричина — почему список пуст при наличии городов (сбой/неполнота загрузки раздела стартовых настроек либо несогласованный формат карты) — лежит у поставщика данных при загрузке и требует самой карты (.h3m) для точной локализации.
