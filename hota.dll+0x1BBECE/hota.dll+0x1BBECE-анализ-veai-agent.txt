================================================================================
АНАЛИЗ ВЫЛЕТА  hota.dll+0x1BBECE  (генератор случайных карт)
================================================================================
    Модель: Veai Agent
    Дата:   06.06.2026
    Заключение: Виновник — hota.dll: обработчик конечной точки сегмента
    без проверки превращает внутреннюю SoD-точку границы в указатель RMGCell*.


КРАТКОЕ РЕЗЮМЕ
--------------
    Вылет происходит в HotA-обработчике генератора случайных карт после вызова
    AtCoords_sub_537DD0. Эта вспомогательная функция не проверяет границы: она
    только считает адрес клетки по формуле cells_base + index*48. После неё
    HotA сразу читает поле клетки по смещению +0x22:

        .text:101BBECC  call    eax                      ; AtCoords(...)
        .text:101BBECE  cmp     byte ptr [eax+22h], 0    ; <-- СБОЙ

    По дампам доказано: faulting address всегда равен EAX+0x22, а при
    ненулевой базе cells смещение EAX−EDX кратно 48. Значит EAX — не случайный
    мусорный указатель, а типизированный адрес «как будто RMGCell*», вычисленный
    AtCoords. Ошибка в том, что этот адрес не принадлежит массиву cells[].

    Первое неверное действие — не инструкция cmp, а предшествующий вызов
    AtCoords на непроверенном входе. HotA берёт X/Y/Z и идентификатор зоны из
    сохранённого кадра SoD-функции RandomizeEdge:

        frame = [ctx+14h]
        X     = [frame+10h]
        Y     = [frame+14h]
        zone  = [frame+18h]
        Z     = [frame+1Ch]

    Эти X/Y — рабочая конечная точка сегмента/ломаной границы, а не публичный
    клеточный API. Для SoD такая точка может быть корректной как геометрия
    линии, но ещё не обязана быть валидной координатой клетки. Перед штатным
    доступом к клетке SoD нормализует координаты; HotA-добавка выполняет
    доступ к конечной точке отдельно и не восстанавливает полный контракт AtCoords.

    Поэтому корректная причинная формула: HotA-обработчик конечной точки выполняет
    непроверенное преобразование «конечная точка SoD до проверки -> RMGCell*». Это
    даёт типизированный выход за границы массива клеток: чтение cell+0x22
    падает, а если адрес читаем, тот же путь может молча записать zone_id в
    cell+0x22 и флаг в cell+0x28 уже вне массива.

    Важное отличие от прежней версии диагноза: падающий loc_101BBEB0 не надо
    описывать как прямую замену штатного SoD-доступа к клетке через resume
    53C6E1h. Связка с 53C6E1h относится к соседней логике/фазе и сама по себе
    не доказывает причину этого вылета. Доказанный дефект находится локально:
    перед call 537DD0h в loc_101BBEB0 нет слоя «конечная точка до проверки -> допустимая
    клетка или отказ».

        .text:101BBECC  call    eax                      ; AtCoords(map,X,Y,Z)
        .text:101BBECE  cmp     byte ptr [eax+22h], 0    ; <-- СБОЙ

        .text:101BBED4  mov     ecx, [esi+14h]

        .text:101BBEDA  mov     [eax+22h], cl            ; возможная запись
        .text:101BBEDD  mov     ecx, [esi+14h]
        .text:101BBEE0  cmp     byte ptr [ecx-0Dh], 0

    реакция обработчика — ничего не писать и вернуть 1.

3. СОПОСТАВЛЕНИЕ АДРЕСОВ
------------------------
    адрес в листинге = адрес в памяти − адрес загрузки + базовый адрес образа

    Пример для HotA 1.8.0: адрес загрузки hota.dll 0x5FDE0000, базовый адрес
    образа 0x10000000. Проверка:

        0x5FF9BECE − 0x5FDE0000 + 0x10000000 = 0x101BBECE

    Полученный адрес совпадает со строкой листинга HotA.dll-180.txt:

        .text:101BBECE  cmp     byte ptr [eax+22h], 0

    Адрес AtCoords 0x537DD0 находится в h3hota.exe и вызывается литералом:

        .text:101BBEC7  mov     eax, 537DD0h
        .text:101BBECC  call    eax

    Кадры patcher_x86 — служебные переходники перехватов. Они объясняют, как
    HotA-обработчик получает сохранённый машинный контекст, но сами не являются
    виновным кодом.


4. ТОЧКА СБОЯ — РАЗБОР
----------------------
    AtCoords_sub_537DD0 — низкоуровневая адресная функция. Она не проверяет
    вход, а только вычисляет указатель:

        .text:00537DD3  mov     eax, [ecx+10h]      ; y_size
        .text:00537DD6  mov     edx, [ebp+Y]
        .text:00537DD9  imul    eax, [ebp+Z]
        .text:00537DDD  add     eax, edx
        .text:00537DDF  mov     edx, [ebp+X]
        .text:00537DE2  imul    eax, [ecx+0Ch]
        .text:00537DE6  add     eax, edx
        .text:00537DE8  mov     edx, [ecx+8]        ; cells_base
        .text:00537DEB  lea     eax, [eax+eax*2]
        .text:00537DEE  shl     eax, 4              ; * 48
        .text:00537DF1  add     eax, edx
        .text:00537DF4  retn    0Ch

    Контракт этой функции: map/cells валидны, x/y/z уже описывают клетку карты.
    HotA нарушает именно этот контракт. После возврата код сразу трактует EAX
    как RMGCell* и читает поле зоны:

        .text:101BBECE  cmp     byte ptr [eax+22h], 0

    Эта проверка — не проверка безопасности указателя. Это бизнес-проверка:
    знаковый байт cell+22h < 0 означает «зона ещё не назначена». Если адрес
    за пределами массива клеток читаем и случайный байт выглядит отрицательным, код пойдёт дальше и
    запишет:

        .text:101BBEDA  mov     [eax+22h], cl
        .text:101BBEE6  or      dword ptr [eax+28h], 10000000h

    Поэтому сбой является не только AV на чтении, а потенциальным типизированным
    чтением/записью за пределами массива клеток по полям RMGCell.


5. РЕГИСТРОВЫЕ ИНВАРИАНТЫ
-------------------------
    Главные инварианты:
      - faulting_addr = EAX + 0x22;
      - при EDX != 0: (EAX − EDX) % 48 = 0;
      - EDX — база cells, загруженная AtCoords из [ecx+8];
      - EAX — результат AtCoords, то есть cells_base + index*48.

    Примеры:
        дамп №1:   +0x31ADF0 / 48 = 67 839 клеток от базы;
        дамп №4:   -0x6AE0 / 48 = -570 клеток до базы;
        дамп №23:  EDX = 0, адрес считается от нулевой базы.

    Кратность 48 показывает, что это не произвольная порча EAX после вызова.
    Ошибка возникла раньше: HotA передала AtCoords вход, из которого получается
    индекс за пределами cells[] либо невалидная база cells.


6. ЦЕПОЧКА ВЫЗОВОВ
------------------
    Снизу вверх:

      A0_ARandMapGen_PrepareSettings_AndGenerateMap_sub_5863B0
        -> A0_ARandMapGen_RmgSettings_CreateRandomMap_sub_54C450
          -> A0_ARandMapGen_Main_sub_549E20
            -> A0_ARandMapGen_Level_Create_Extra_WaterAndRock_Zones_SetZonesCells_SetZonesDists_sub_53E540
              -> A0_ARandMapGen_RmgGraphEdge_Zone_Setup_Edge_sub_53C880
                -> A0_ARandMapGen_CellsLine_RandomizeEdge_And_SetAsBelongToZone_sub_53C4E0
                  -> patcher_x86 dispatch
                    -> hota.dll loc_101BBEB0
                      -> h3hota.exe AtCoords_sub_537DD0
                        -> hota.dll cmp [eax+22h]  <-- СБОЙ

    Важная поправка к прежней интерпретации. loc_101BBEB0 не нужно описывать
    как прямую замену штатного SoD-доступа к клетке. По коду он возвращает
    eax=1 и даёт штатному выполнению продолжиться. Это добавочный обработчик,
    который заранее смотрит на конечную точку: берёт значения из сохранённого кадра
    SoD и пытается дозаполнить клетку зоны.

    Связка с адресом 53C6E1h относится к соседней фазе/обработчику и не является
    достаточным доказательством причины этого вылета. Надёжная причинная цепочка
    строится не от предполагаемого resume-адреса, а от локального факта:
    loc_101BBEB0 сам вызывает AtCoords и сам использует результат как RMGCell*.


7. ПЕРВОПРИЧИНА: НЕПРОВЕРЕННОЕ ПРЕОБРАЗОВАНИЕ КОНЕЧНОЙ ТОЧКИ В RMGCell*
------------------------------------------------------------------------
    SoD-функция RandomizeEdge работает с геометрией границы зоны. Она хранит
    end_x/end_y как рабочую точку ломаной. Такая точка корректна для
    алгоритма линии, но не обязана уже быть валидной клеткой карты.

    Поставщик данных — SoD-код. Он извлекает очередную точку из локального
    списка и пишет её в end_x/end_y:

        .text:0053C55A  mov     ecx, [eax-8]
        .text:0053C560  mov     [ebp+end_x], ecx
        .text:0053C564  mov     edx, [eax+4]
        .text:0053C56A  mov     [ebp+end_y], edx

    Этот записывающий код корректен для своего слоя: он выдаёт геометрическую точку
    границы. Ошибка не в нём, а у HotA-потребителя, который требует от
    этих же значений более сильный контракт — «валидные координаты клетки».

    HotA-обработчик получает сохранённый кадр через контекст patcher:

        .text:101BBEB5  mov     eax, [esi+14h]      ; frame
        .text:101BBEBE  push    dword ptr [eax+14h] ; end_y до проверки
        .text:101BBEC4  push    dword ptr [eax+10h] ; end_x до проверки
        .text:101BBECC  call    AtCoords

    Между «точка геометрии до проверки» и «указатель RMGCell*» отсутствует
    слой проверки. Должен быть слой:

        конечная точка до проверки -> допустимая клетка или отказ от HotA-записи

    Сейчас этот слой отсутствует. Поэтому HotA выполняет непроверенное
    преобразование типа:

        конечная точка SoD до проверки -> RMGCell*

    Если x/y до проверки находятся вне карты, если z недопустим или если cells_base
    равен NULL, AtCoords всё равно механически вернёт адрес. Дальше бизнес-
    проверка cell+22h использует этот адрес как настоящую клетку.

    Почему это не обязано воспроизводиться как AV на каждом зерне. Сид может
    повторить опасную геометрию, но вылет зависит от памяти: адрес за пределами
    массива клеток может попасть в нечитабельную страницу, в читаемую чужую структуру или в область,
    где бизнес-проверка не приведёт к записи. Поэтому правильный критерий
    воспроизведения — не только «упало», а «до AtCoords нарушен клеточный
    контракт».

    Выбор между нормализацией и пропуском не доказан одним текстовым логом. Если HotA-добавка
    должна сопровождать будущую SoD-обработку конечной точки, SoD-подобная
    нормализация x/y может быть верной. Если конечная точка вне карты не должна
    порождать дополнительную клетку, безопаснее пропуск. В обоих вариантах
    доказанное обязательное условие одно: loc_101BBEB0 не должен вызывать
    AtCoords, пока context/cells/z/zone_id/x/y не образуют допустимый вход.


8. УСЛОВИЯ, СПРОВОЦИРОВАВШИЕ СБОЙ
---------------------------------
    - Исполняется генерация случайной карты, стадия расстановки зон и рисования
      границ.
    - Путь проходит через Setup_Edge и RandomizeEdge генератора границ зон.
    - HotA-обработчик конечной точки получает сохранённый кадр SoD и пытается выполнить
      дополнительную работу с конечной клеткой.
    - Срабатывание AV зависит от того, куда попал вычисленный адрес за пределами массива клеток:
      нечитабельная память даёт вылет; читаемая память может дать пропуск действия или
      скрытую запись за пределами массива.
    - Зерно ГСЧ полезно для повторения геометрии, но не гарантирует AV, потому
      что AV зависит также от раскладки памяти и предыдущих побочных эффектов.


9. ДОСТОВЕРНОСТЬ И ОГРАНИЧЕНИЯ
------------------------------
    - Версия листинга падающего модуля HotA.dll-180.txt подтверждена точным
      совпадением адреса 0x101BBECE и окружающего кода loc_101BBEB0.
    - Локализация точки сбоя — высокая: инструкция одна, faulting_addr =
      EAX+0x22, вызов AtCoords непосредственно перед падением.
    - Локализация нарушенного контракта — высокая: AtCoords не проверяет
      вход; HotA вызывает его на значениях из сохранённого кадра SoD без
      полной проверки map/cells/z/zone_id/x/y.
    - Наиболее вероятный конкретный источник основного профиля — end_x/end_y до проверки
      до нормализации SoD. Однако текстовый краш-лог не содержит памяти по
      frame = [ctx+14h], поэтому точные x/y/z/cells на момент входа
      нужно подтвердить минидампом или runtime-логом перед call 537DD0h.
    - Подслучай cells_base = NULL показывает, что один только clamp x/y не
      является полным фиксом. Нужна проверка всего контракта AtCoords.
    - Полезность лога для исправления — ВЫСОКАЯ: место защиты локализовано
      перед call 537DD0h в loc_101BBEB0. Ограничение касается не fix-site, а
      политики обработки конечной точки вне карты: нормализация или пропуск.
    - Связку с 53C6E1h нельзя использовать как центральное доказательство
      причины этого вылета. Она полезна как контекст соседних фаз, но падающий
      loc_101BBEB0 сам вызывает AtCoords и сам создаёт указатель RMGCell* за пределами массива.
    - В SoD этот конкретный дефект не подтверждён: SoD содержит геометрический
      поставщик точек до проверки, но не содержит HotA-обработчика конечной точки,
      который непроверенно превращает такую точку в RMGCell*.


10. ПРИЗНАКИ ПОИСКА ВЫЛЕТА В СТАРЫХ ВЕРСИЯХ HotA / В SoD
--------------------------------------------------------
    Адрес hota.dll+0x1BBECE не стабилен между версиями. Устойчивая сигнатура
    дефекта — не конкретный offset, а поведение:

        push dword ptr [eax+1Ch]
        push dword ptr [eax+14h]
        push dword ptr [eax+10h]
        mov  eax, 537DD0h
        call eax
        cmp  byte ptr [eax+22h], 0
        jge  ...
        mov  [eax+22h], cl
        or   dword ptr [eax+28h], 10000000h

    Регистровая сигнатура:
      - faulting_addr − EAX = 0x22;
      - при EDX != 0: (EAX − EDX) кратно 48;
      - EAX выходит за cells[] либо EDX = 0;
      - стек идёт через RandomizeEdge / Setup_Edge генератора границ зон.

    Эта сигнатура подтверждает один и тот же HotA-дефект в фактических дампах
    1.6.1, 1.7.0, 1.7.1, 1.7.2, 1.7.3 и 1.8.0. Для 1.5.3, 1.5.4 и 1.6.0
    похожий блок виден по коду листингов, но без фактических дампов этих
    версий корректнее отмечать это как кодовое предположение, а не как
    подтверждённый вылет.

    Для чистого SoD прямой флаг «тот же дефект» не выставляется. Общая функция
    RandomizeEdge унаследована, но дефектный потребитель — HotA-обработчик конечной точки —
    находится в hota.dll.


11. ВОСПРОИЗВЕДЕНИЕ ВЫЛЕТА
--------------------------
    Условие активации пути: генерация случайной карты проходит через этап
    построения границ зон, где исполняется RandomizeEdge и HotA-обработчик конечной точки
    loc_101BBEB0.

    Базовый сценарий:
      - Генерировать случайные карты в HotA на крупных размерах и с большим
        числом зон, чтобы увеличить число граничных сегментов.
      - Использовать зерно ГСЧ из дампа как попытку повторить ту же геометрию.
      - Не считать отсутствие AV опровержением: адрес за пределами массива клеток может быть читаемым.

    Дифференциальный тест причины:
      - Поставить точку останова или логирование непосредственно перед
        .text:101BBECC call AtCoords в loc_101BBEB0.
      - Снять значения:

            ctx   = ESI
            frame = [ctx+14h]
            this  = [frame-1Ch]
            x     = [frame+10h]
            y     = [frame+14h]
            zone  = [frame+18h]
            z     = [frame+1Ch]
            cells = [this+0Ch+8]
            width = [this+18h]
            height= [this+1Ch]

      - Причина подтверждена, если до AtCoords выполняется хотя бы одно:
            cells == 0;
            width/height невалидны;
            z недопустим;
            x/y вне карты и не нормализованы;
            zone_id недопустим.
      - Критерий исправления: loc_101BBEB0 больше не вызывает AtCoords, пока
        полный клеточный контракт не выполнен. При любом отказе обработчик должен
        вернуть 1 без HotA-записи и без изменения SoD-frame.


ВЫВОД
-----
    Виновник — hota.dll, конкретно обработчик конечной точки loc_101BBEB0 генератора
    случайных карт. Он берёт end_x/end_y/z до проверки из сохранённого кадра SoD и
    без проверки полного контракта вызывает AtCoords. Результат AtCoords затем
    используется как RMGCell*, хотя может не принадлежать массиву cells[].

    Это не ошибка writer-а координат и не прямой перехват штатной SoD-клетки.
    SoD-код пишет рабочую геометрическую конечную точку, что корректно для
    алгоритма линии. Ошибка у HotA-потребителя: он требует от этой точки более
    сильный контракт — валидную клетку карты — и не добавляет адаптер между
    двумя уровнями.

    Класс дефекта — типизированный выход за границы массива RMG-клеток.
    Видимый AV на cell+0x22 — только одно проявление; если адрес за пределами массива
    клеток читаем, возможна скрытая запись вне массива через cell+0x22 и cell+0x28. Исправление
    должен стоять перед call 537DD0h: проверить context/cells/z/zone_id,
    x/y нормализовать или отвергнуть, а при сомнении вернуть 1 без записи.
